## Verified behaviors

1. **Commit ordering / data-loss seam (move=True).** `workspace/transfer.go` commits
   the DST graft first (`CommitWorldLoaded(dstWorldID)`), and only on `dstRev.ID != 0`
   commits the staged source deletes (`CommitWorldLoaded(srcWorldID)`). The two commits
   are sequential and self-contained (neither holds `w.mu` across the other). A
   source-delete-commit failure after a successful dst commit returns an error that
   names the half-applied state explicitly ("graft committed to destination ... rev N,
   but source ... delete commit failed — the entity REMAINS in the source; re-run the
   source delete manually"). Failure leaves a recoverable DUPLICATE, never loss.
   - The source delete is staged per-entry in `TransferEntries` ONLY after that entry's
     dst graft succeeds, so a graft failure never leaves an orphan source delete staged
     ahead of an absent graft. Partial stages die in-memory on both sessions.
   - `TestAutomation_TransferMoveCommitsDstBeforeSource` genuinely pins the ordering:
     mutation-tested by reordering the two commits to source-first in a scratch copy →
     test FAILS ("dst revision id 4 is not < source revision id 3"). Not an incidental
     pass — it rests on monotonic `RecordRevisionAdvancingHead` id allocation.

2. **body.id allocator distinctness (remap_ids=True).** `freshDstBodyID`/`dstBodyIDUsage`
   in `transfer_identity.go` mirror the species allocator exactly, including the
   load-bearing staged-fold over `t.dst.StagedOperations()` (`OperationAppendEntry`
   bibite payloads). The observed-max-only difference (no nextBodyID counter) is
   documented so a future reader/M8 will not "fix" it.
   - `TestTransferMultiGraftRemapsDistinctBodyIDs` would fail an observed-max-only
     allocator: mutation-tested by deleting the staged-fold in a scratch copy → test
     FAILS ("both grafts remapped to the SAME body.id 45"). The invariant is real.
   - The `freshDstBodyID` `!has → fresh=0` fallback is dead-but-defensive on the
     collision path: a collision implies a live dst bibite carries the id via the same
     `bibiteBodyID` reader `dstBodyIDUsage` scans, so `has` is necessarily true there.
   - **Dangling-child guard stays loud with remap on.** `reconcileGraftIdentity` gates
     only the body.id-collision error on `!remapIDs`; the dangling-child check runs
     unconditionally. `TestTransferAppendEntryRemapKeepsDanglingChildLoud` asserts a
     loud failure + 0 staged ops with `RemapIDs=true`. Verified.
   - 0-staged-ops-on-rejection invariant preserved: the body.id rewrite happens on the
     clone BEFORE any `StageAppend`, alongside the existing pre-stage checks.

3. **Contract additivity.** The return dict ADDS `moved`/`source_committed`/
   `source_revision_id` and keeps `transferred`/`committed`/`revision_id`/`sha256`
   (verified in `automation.go`: all four originals present, no renames). `app.js`
   drops the regex scrape entirely and reads typed fields from a deterministic
   per-line emit (`print(r["transferred"])` / `r["moved"]`), so a format change can
   no longer show `transferred ?`. New `index.html` move / remap-ids checkboxes match
   the existing modal markup.

4. **Scope.** The settings-copy / array-feed bridge (`CollectSettingsValue`/
   `AppendArray` etc.) is untouched in `transfer_bridge.go`, and the species-ancestry
   parentID-reset note at `transfer.go:321-326` is intact and untouched. No silent
   widening.

## Perf

population and selection size, run once per transfer — same shape as the proven
species allocator, not a hot path. No N+1, no new quadratic surface. Plan carries no
perf flag; none warranted.
