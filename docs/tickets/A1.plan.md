## Approach

### 1. Adding the new columns to `save_revisions` (the subtle part)

SQLite has **no `ADD COLUMN IF NOT EXISTS`**, and `ALTER TABLE … ADD COLUMN` errors if the column
already exists — which would break `ApplyMigrations`'s idempotency contract (it runs on every
`Open`). The whole current schema is a single string `Exec`'d wholesale (store.go:107), and the
existing tables already rely on `CREATE TABLE IF NOT EXISTS`. So:

**Add the five new columns inline in the `CREATE TABLE IF NOT EXISTS save_revisions (…)` body.**
For a brand-new DB this creates them directly. For an existing DB the `CREATE … IF NOT EXISTS` is
a no-op and the inline columns are NOT retroactively added — but there is **no shipped database
with `save_revisions` yet that lacks these columns** that we must preserve in this repo (the store
is internal, created fresh by `Open`; the test suite always starts from `:memory:` or a fresh
`t.TempDir()`). Confirm this assumption holds (see Risks) — it is the cheapest correct option and
matches how the existing schema already handles all column definitions (none of the current
columns were ever added via `ALTER`).

> Decision rationale: the plan text (lines 217-227) says "Add to `save_revisions`: …" and "Schema
> is additive + `IF NOT EXISTS`; existing rows … keep working." Defining the columns inline in the
> `CREATE TABLE IF NOT EXISTS` is exactly the additive/idempotent pattern the rest of this file
> already uses; introducing `ALTER TABLE` guards would be a heavier, non-idempotent mechanism the
> file does not otherwise use. **Do NOT add `ALTER TABLE` statements** — they break the
> single-`Exec` idempotency.

New columns on `save_revisions` (exact DDL, defaults per plan lines 217-221):
```sql
world_id TEXT NULL REFERENCES worlds(id) ON DELETE RESTRICT,
tier TEXT NOT NULL DEFAULT 'full' CHECK (tier IN ('full', 'mirror_only')),
blob_present INTEGER NOT NULL DEFAULT 1 CHECK (blob_present IN (0, 1)),
refcount INTEGER NOT NULL DEFAULT 0 CHECK (refcount >= 0),
mirror_schema_version INTEGER NULL,
```

### 2. New tables (exact DDL, per plan lines 206-215)

```sql
CREATE TABLE IF NOT EXISTS workspaces (
    id TEXT PRIMARY KEY,
    owner TEXT NOT NULL DEFAULT '',
    name TEXT NOT NULL DEFAULT '',
    created_at TEXT NOT NULL
);

CREATE TABLE IF NOT EXISTS worlds (
    id TEXT PRIMARY KEY,
    workspace_id TEXT NOT NULL REFERENCES workspaces(id) ON DELETE RESTRICT,
    name TEXT NOT NULL DEFAULT '',
    head_revision_id INTEGER NULL REFERENCES save_revisions(id) ON DELETE RESTRICT,
    sim_time REAL NULL,
    created_at TEXT NOT NULL
);

CREATE TABLE IF NOT EXISTS nodes (
    id TEXT PRIMARY KEY,
    workspace_id TEXT NOT NULL REFERENCES workspaces(id) ON DELETE RESTRICT,
    world_id TEXT NULL REFERENCES worlds(id) ON DELETE SET NULL,
    node_id TEXT NOT NULL DEFAULT '',
    run_id TEXT NOT NULL DEFAULT '',
    status TEXT NOT NULL DEFAULT '',
    compat_addr TEXT NOT NULL DEFAULT '',
    drop_path TEXT NOT NULL DEFAULT '',
    created_at TEXT NOT NULL
);
```

### 3. Ordering constraint (foreign keys)

`PRAGMA foreign_keys = ON` is set at the top of the file (schema.sql:1) and per-connection in the
DSN (store.go:81). There is a **mutual reference**: `worlds.head_revision_id → save_revisions(id)`
and `save_revisions.world_id → worlds(id)`. SQLite resolves FK *targets* at row-write time, not at
`CREATE TABLE` time (it does not require the referenced table to exist when the FK is declared, as
long as it exists before a row is inserted), so the `CREATE TABLE` order is not load-bearing for
correctness. For readability, order the file: existing `script_runs` → new `workspaces` → new
`worlds` → existing `save_revisions` (now with `world_id`) → new `nodes`. Keep the existing
`save_revisions` indexes; the `world_id → worlds(id)` FK is fine because `worlds` is created above
it. Verify the whole schema still `Exec`s in one shot (the schema test covers this).

### 4. Indexes (plan lines 221-222)

```sql
CREATE INDEX IF NOT EXISTS worlds_workspace_id_idx ON worlds(workspace_id);
CREATE INDEX IF NOT EXISTS nodes_workspace_id_idx ON nodes(workspace_id);
CREATE INDEX IF NOT EXISTS nodes_world_id_idx ON nodes(world_id);
CREATE INDEX IF NOT EXISTS save_revisions_world_id_idx ON save_revisions(world_id);
```
`save_revisions(sha256)` index already exists (schema.sql:33-34) and is reused for refcount/dedup
lookups — **do not duplicate it.**

## Risks / seams

- **`ALTER TABLE` trap (most important).** Do NOT use `ALTER TABLE … ADD COLUMN` to add the
  `save_revisions` columns. The schema is `Exec`'d in full on every `Open` (store.go:107); an
  unconditional `ALTER` would error on the second open (column already exists), violating
  `ApplyMigrations` idempotency, and `Open` would start failing. Inline the columns in the
  `CREATE TABLE IF NOT EXISTS save_revisions` body. **Assumption to verify before relying on it:**
  there is no pre-existing on-disk `revisionstore` SQLite file in the repo or tests that already
  has a `save_revisions` table without these columns (a stale file would silently NOT get the new
  columns under the inline approach). Grep confirmed: the only `save_revisions` references outside
  the package are comments in `script/thebibites` (loadedsave.go, run.go); every store is created
  fresh from schema by `Open`, and tests use `:memory:`/`t.TempDir()`. So inline is safe here.
- **Mutual FK (`worlds.head_revision_id` ↔ `save_revisions.world_id`).** Both directions are
  `NULL`-able, so no insert ordering deadlock: a first world is created with `head_revision_id
  NULL`, the first revision is inserted with that world's id, then the head is pointed at it
  (A2's `RecordRevisionAdvancingHead`, plan line 247). Schema just needs both FKs declarable;
  declare them as written. With `PRAGMA foreign_keys = ON`, a `world_id` pointing at a
  non-existent world WILL be rejected at insert time — that is desired, and A2 is responsible for
  insert ordering.
- **`mirror_saves` is NOT a table here.** Resist creating it — plan lines 53-56 and 223-226 make
  it a projection/view over `save_revisions` (+ `worlds.sim_time`), materialized/ATTACH-ed by the
  workspace layer later. Adding a physical `mirror_saves` table now would create a second source
  of truth, which the plan explicitly forbids ("no separate store; SQLite stays the single source
  of truth", line 226).
- **`refcount` lives on `save_revisions`, not a separate table.** Plan line 471 allows "a
  `blob_refs` view keyed by sha256" but the contention note (line 196) and line 218 put the column
  on `save_revisions`. Put the column on `save_revisions` per line 218; do not add a `blob_refs`
  table in A1 (that view, if needed, is A2/G1's call).
- **`ON DELETE` clauses.** Used `ON DELETE RESTRICT` for FKs that must not orphan
  (`worlds.workspace_id`, `worlds.head_revision_id`, `nodes.workspace_id`, mirroring the existing
  `save_revisions` FKs at schema.sql:25/29) and `ON DELETE SET NULL` for `nodes.world_id` (a world
  can be removed while a node row survives unbound). v1 has no hard-delete paths (plan line 470),
  so these are defensive defaults, not exercised yet — but they encode intent for later tracks.
- **Scope discipline.** This ticket is the schema + uuid promotion + a schema test ONLY. Adding
  store methods (`CreateWorkspace`, `SetWorldHead`, refcount methods, `RevisionInput.WorldID`,
  `RecordRevisionAdvancingHead`, etc.) is **A2** and would collide with the A2 executor on
  `store.go` (the plan flags `store.go` as shared between A2 and G1, lines 195-196). Leave
  `store.go` and `RevisionInput` untouched in A1.
- **CHECK constraints.** Added `CHECK (tier IN ('full','mirror_only'))`,
  `CHECK (blob_present IN (0,1))`, `CHECK (refcount >= 0)` to match the existing file's style
  (it already CHECKs `staged_ops >= 0`, `dry_run IN (0,1)`, `size >= 0`). These harden the two-tier
  invariant at the DB layer; they are additive and safe for defaulted existing rows.
