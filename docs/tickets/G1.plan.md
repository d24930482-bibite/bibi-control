What A2 deliberately deferred (A2.plan.md:24, "wire up call sites" for G1): the recording
path never establishes a revision's own reference. `RecordRevision` (store.go:250) and
`RecordRevisionAdvancingHead` (store.go:276) insert via `insertRevisionTx` (store.go:318),
which intentionally omits `refcount` so the schema default (`0`) applies. So **every
recorded revision lands at refcount 0** ‚Äî the safety net is never armed. The plan's
explicit contract is: "On `RecordRevision`/`AddWorld`/`IngestAutosave` call
`IncBlobRef(sha)`" (workspace_plan.md:469). `AddWorld`/`IngestAutosave` do not exist yet
(future C/D-track tickets); the only recording entry points today are the two `Record*`
methods. G1's remaining work is to wire the auto-reference into those two methods and fix
the now-shifted refcount expectations in the existing tests.

## Goal

"Done" = recording a revision automatically establishes that revision's own blob
reference **atomically with the insert**, so the refcount invariant `refcount(sha) ==
count of revisions referencing sha` holds the instant a revision exists. Concretely: after
`RecordRevision`/`RecordRevisionAdvancingHead` returns, the returned `Revision.Refcount`
is 1 (for a brand-new hash) and the on-disk row reflects +1 for every row sharing that
hash; the increment is committed in the same SQLite transaction as the insert (a crash
can never leave a refcount-0 row that `UnreferencedBlobs`/GC would wrongly reap). The
existing eviction/GC semantics are unchanged ‚Äî they already account this self-ref
(`EvictRevisionBlob` refuses when `refcount > 1`, store.go:878). All `revisionstore` tests
stay green after updating the few refcount expectations that the auto-ref shifts.

## Approach

1. Extract the inc SQL into `incBlobRefTx(ctx, q sqlExec, sha256 string) error` (reuse the
   `sqlExec` interface defined at store.go:37). Have the existing public `IncBlobRef`
   (store.go:783) delegate to it after its nil/`usableContext`/`validateSHA256` checks and
   `RowsAffected` accounting ‚Äî behavior unchanged.
2. In `insertRevisionTx` (store.go:318), after obtaining the new row `id`, call
   `incBlobRefTx(ctx, q, revision.SHA256)` and return its error if any. Because callers
   pass the tx (`*sql.Tx`), the inc commits with the insert. The per-hash UPDATE touches
   the just-inserted row and any pre-existing dedup-sharing rows, so the shared count
   stays equal to the number of referencing revisions (the per-hash model A2.plan.md:151
   chose ‚Äî document this in the comment).
3. Convert `RecordRevision` (store.go:250) to a tx exactly like
   `RecordRevisionAdvancingHead` minus the head UPDATE: `BeginTx` ‚Üí `insertRevisionTx(ctx,
   tx, ...)` ‚Üí `tx.Commit()` ‚Üí `RevisionByID`. This is required so the insert+inc are
   atomic (the current direct-`s.db` path would split them into two autocommitted
   statements).
4. Leave `EvictRevisionBlob`/`UnreferencedBlobs`/`PromoteRevision`/`DecBlobRef` untouched ‚Äî
   their `refcount > 1` / `refcount = 0` logic already assumes each revision carries its
   own +1 (the "this revision's own ref is counted" convention documented at
   store.go:840-842). Wiring the self-ref is precisely what makes that documented
   convention true in practice.

## Risks / seams the executor must not break

- **Atomicity is the whole point.** The self-ref inc must be in the *same tx* as the
  insert. Do not increment from `RecordRevision` after `RevisionByID` (that would be a
  second autocommit and reintroduce the crash window). Wrap `RecordRevision` in a tx; reuse
  the tx already present in `RecordRevisionAdvancingHead`.
- **Per-hash, all-rows model is intentional** (A2.plan.md:151). The inc UPDATE touches
  every row sharing the sha256, so a dedup-shared hash carries the same count on all its
  rows. Do not "fix" this into a per-row counter ‚Äî `EvictRevisionBlob`/`UnreferencedBlobs`
  read it as a per-hash count.
- **Schema default stays 0.** The +1 lives in the Go recorder, not in `schema.sql`. Do not
  add a column default of 1 ‚Äî `TestSchemaAppliesAndIsIdempotent` (and the raw-INSERT path
  any future migration test uses) expects the raw default to be 0, and a column default
  would also wrongly +1 future hand-written inserts.
- **`CHECK (refcount >= 0)` must never be violated** (schema.sql:53). The inc only ever
  raises the count, so it is safe; the existing `DecBlobRef` `WHERE refcount > 0` guard
  (store.go:822) remains the floor. The updated lifecycle test must still be able to drive
  the count to exactly 0 before asserting the `UnreferencedBlobs`/evict candidacy.
- **Public `IncBlobRef`/`DecBlobRef` stay as A2 shipped them** ‚Äî they are the manual levers
  G2 (`EvictWorldHistory`) and G3 (`GCUnreferencedBlobs`) and the workspace transfer path
  will call for additional/removed references beyond the recorder's self-ref. Refactoring
  the shared SQL into `incBlobRefTx` must not change `IncBlobRef`'s signature, nil-guard,
  `validateSHA256`, or rows-affected return.
- **Do not touch blobstore bytes or DuckDB.** G1 is catalog-only refcount accounting; byte
  deletion is G2/G3, DuckDB is the workspace's (workspace_plan.md:471, 477-484).
