# M8 — Save-format drift capture & write-path hardening (parser + mutator)

Source: `docs/missing.md` §8. Philosophy (project memory): **loud but localized on
drift — harden, don't rewrite the parser.**

## Goal

Close the two **correctness** edges in §8 that are silent data-corruption / write-guard
gaps, and add the missing **version gate** so format drift is observable. "Done" means:

1. **Gene-SET value-type guard.** The mutator's `resolveGeneColumn` enforces that the
   ref's `value_type` matches the target column (`number_value`→`number`, etc.) — exactly
   like the settings resolver (`requireSQLRefValueType`) — so a gene write can no longer
   route a wrong-typed value into a typed column. The one live caller (the Starlark gene
   binding) is updated to populate `ValueType` so it keeps working.
2. **Scene `Version` gate.** `SceneState.Version` is parsed but nothing branches on it
   (`parse_scene.go:13`). Add a loud-but-localized diagnostic when the scene version is
   outside the known/supported set, so a save-format bump surfaces in the diagnostics
   table instead of silently mis-parsing. No parser rewrite — just a guard + diagnostic.

The remaining §8 bullets are explicitly scoped **out of v1** (see *Scope* below); they are
ergonomics / already-mitigated / documented-assumption, not silent-loss correctness.

Tier: **high — write-guard invariant (wrong-typed gene cell) on a shared resolver +
fan-out (mutator resolver feeds the whole SQL-ref write path) + format-drift gate**. The
gene guard is a property a naive passing test misses (a number written to `number_value`
still passes unless the test deliberately crosses types), and the resolver is shared
infrastructure (`ResolveSQLValueRef` is also reached by transfer), so a regression there is
silent and propagates. Executor = opus, reviewer = opus.

## Scope (what is v1 vs deferred)

**In v1 (correctness + observability):**
- Gene-SET value-type guard (§8 bullet 4) — write-guard gap, the highest-value item.
- Scene version gate (§8 bullet 2) — drift observability; small, localized.

**Deferred (record in plan, do NOT implement):**
- *Unknown/drifted JSON sections (§8 bullet 1).* **Already mitigated.** H4 + the existing
  `unknown_json_section` diagnostic (`parse_payload.go:115-118`, tested in
  `saveparser/thebibites/tests/unknown_json_diagnostic_test.go`) already make drift loud.
  The original "dropped, not captured" gap is closed; no new work. Do not re-add a capture
  table — that contradicts the H4 churn-resilience decision (diagnostic, not data trace).
- *Species/scene cascades no-op on missing paths + same-batch on-disk-JSON reads
  (§8 bullets 3 & 5).* These are **documented correctness assumptions**, not bugs:
  `adjustSceneCount` (`session.go:466`), `removeActiveSpecies` (`session.go:583`),
  `speciesHasOtherMembers` (`session.go:556`), `entriesReferencingChild` (`session.go:491`)
  all intentionally degrade quietly / read on-disk JSON, with comments saying so. Changing
  the same-batch read semantics is a behavior change to the commit/cascade ordering that
  belongs in its own ticket with its own atomicity tests, not bundled here. Leave as-is.
- *Append/insert can't create missing intermediate structure (§8 bullet 5).* `setJSONPath`
  already supports `CreateMissing` (`path.go:184`); `appendJSONArray`/`resolveContainer`
  require the container to exist (`path.go:288-291`, `:255-257`). Auto-creating intermediate
  containers on append is a meaningful semantic change (when is a missing container an error
  vs. a thing to synthesize?) and needs its own design; defer.

If review wants the version gate dropped too, the gene guard alone is a complete, shippable
v1 — it is the irreducible correctness fix.

## Approach

### 1. Gene-SET value-type guard (`sqlref_entities.go:47`)

Today:
```go
func resolveGeneColumn(ref SQLValueRef, kind tb.EntryKind, columns map[string]string) (Target, string, error) {
    if _, err := sqlRefColumnValue(ref, columns); err != nil {   // wantType discarded
        return Target{}, "", err
    }
    if err := requireSQLRefString(ref, ref.Path, "path"); err != nil { ... }
    ...
}
```
The settings resolver (`sqlref_settings.go:18-34`) does it right:
```go
wantType, err := sqlRefColumnValue(ref, columns)   // e.g. "number" for number_value
...
if err := requireSQLRefValueType(ref, wantType); err != nil { return ... }
```
Make `resolveGeneColumn` do the same: capture `wantType` from `sqlRefColumnValue`, and
after the existing `path` check, call `requireSQLRefValueType(ref, wantType)`
(`sqlref_require.go:26`). The gene column map `geneValueColumnTypes`
(`sqlref_generated.go:138-142`) already maps `number_value`→`number` etc., so no new
metadata is needed — this is pure derive-from-generated-metadata, no new allowlist.

Order the new guard after the `path` non-empty check and before the per-kind target
resolution, matching the settings resolver's guard ordering.

### 2. Populate `ValueType` on the live gene ref (`gene.go:152`)

`setGeneValue` builds the ref without `ValueType`. The `GeneRow` carries `Type`
(`tb.ScalarType`), used a few lines up for `validateValue(scalarTypeRule(row.Type), …)`
and `scalarValueColumn(row.Type)`. Add one field to the ref literal:
```go
ref := mutator.SQLValueRef{
    Table:     table,
    Column:    column,
    EntryName: row.EntryName,
    OwnerKind: row.OwnerKind,
    OwnerID:   row.OwnerID,
    Path:      row.Path,
    ValueType: string(row.Type),   // NEW — mirrors settings_value.go:198
}
```
`row.Type` is the canonical gene scalar type (number/bool/string) and `column` was derived
from the same `row.Type` via `scalarValueColumn`, so `ValueType` and the column-implied
`wantType` are guaranteed consistent — the live path keeps passing. This is the only live
constructor of a gene `SQLValueRef` (confirmed: grep for gene-ref construction across
`script/` and `savemutator/` finds only this site; `transfer.go` grafts whole entities and
never builds a gene cell ref).

### 3. Scene version gate (`parse_scene.go:13`)

`parseScene` already extracts `scene.Version` from `version` but never validates it. Add a
localized gate after the version is read: define a package-level set/list of known scene
versions (a single source-of-truth `var knownSceneVersions = map[string]struct{}{...}`
seeded with the version(s) the current fixtures carry — discover them with a quick grep of
the test fixtures / a parsed sample), and when `scene.Version` is non-empty AND not in the
set, emit `ctx.addDiagnostic(SeverityWarning, "scene_version_unsupported", entry.Name, …)`.
Keep parsing (loud-but-localized: a diagnostic, not an abort) — the rest of `parseScene`
runs unchanged. An empty/missing version keeps today's behavior (no diagnostic), so normal
saves stay quiet. Mirror the existing diagnostic codes' style (`scene_not_object` above it,
`unknown_json_section` in `parse_payload.go`).

To find the seed version(s): grep fixtures under `saveparser/thebibites/tests/` and any
`.bb8scene` test data for `"version"`; use whatever the suite's real saves carry so the
suite does not light up the new diagnostic spuriously.

## Tests

**Run:** `go test ./savemutator/thebibites/... ./script/thebibites/... ./saveparser/thebibites/...`
(from the worktree root). Full-suite sanity: `go test ./...` if time allows.

1. **Gene guard — positive (no regression).** The existing live gene write tests
   (`script/thebibites/gene_write_test.go`: `TestGeneWritePersists`,
   `TestGeneWriteMirrorEverything`, `TestGeneWriteMirrorKeyedByPath`,
   `TestGeneWriteViaScript`) must still pass unchanged — they exercise the binding, which
   now sets `ValueType`. This proves the live path survives the new guard.

2. **Gene guard — existing resolver allowlist cases must be updated.** `sqlref_test.go`
   `TestResolveSQLValueRefsAllowlist` builds gene cases via
   `addSQLRefValueColumnCases(&tests, "bibite_genes", SQLValueRef{... no ValueType}, …)`
   (lines 278-283) and the egg twin (lines 299-304). That helper (lines 725-737) loops all
   three value columns with **no `ValueType`** — these cases will now FAIL the guard.
   **Required edit:** make the gene cases set `ValueType` to match each column, exactly as
   the settings value cases do. Cleanest: give the gene cases their own loop (or extend the
   helper to take a per-column `ValueType`) that sets `ref.ValueType = "number"` for
   `number_value`, `"bool"` for `bool_value`, `"string"` for `string_value`. This is the
   load-bearing test change — do not skip it or the suite breaks.

3. **Gene guard — new negative case.** Add a resolver test asserting that a gene ref with a
   *mismatched* `value_type` (e.g. `Column: "number_value"`, `ValueType: "string"`) returns
   an error from `ResolveSQLValueRef`. Mirror the shape of any existing settings
   value-type-mismatch negative test if present; otherwise add a focused table case. This
   is the test that actually proves the fix (a same-type test passes with or without the
   guard, so the cross-type case is mandatory).

4. **Scene version gate.** Add a test under `saveparser/thebibites/tests/` (sibling to
   `unknown_json_diagnostic_test.go`, same `package tests`, reuse `hasDiagnostic`):
   - a scene with a known version emits NO `scene_version_unsupported` diagnostic;
   - a scene with a bogus version (e.g. `"version":"99.9-future"`) DOES emit it AND still
     parses `nBibites`/`simulatedTime` (gate is non-fatal);
   - a scene with no `version` key emits NO diagnostic (no spurious noise on legacy saves).
   Use `ParseEntryBytesAs`/`ParseBytesAs`/`createZip` as the existing test does.

## Risks / seams

- **Shared resolver blast radius (the core risk).** `resolveGeneColumn` is reached via the
  generated catalog (`sqlref_catalog.go:101-104,118-122`) and ultimately through
  `ResolveSQLValueRef`, which `transfer.go:85` also calls generically. Adding a required
  guard means *every* path that builds a gene ref must set `ValueType`. The plan covers the
  only live constructor (`gene.go`); the executor must NOT add the guard without the
  `gene.go` change in the same change, or the binding write breaks. Verify by running the
  `gene_write_test.go` suite.
- **Test fixtures gate the version diagnostic.** Seed `knownSceneVersions` from the actual
  versions in the suite's fixtures, or `go test ./saveparser/...` and any test that parses
  a real save will start emitting `scene_version_unsupported` and may fail count-zero
  assertions (cf. the "normal save emits zero" assertion in
  `unknown_json_diagnostic_test.go`). Grep fixtures first; choose the seed from real data.
- **Empty value_type semantics.** `requireSQLRefValueType` first requires `ValueType`
  non-empty (`sqlref_require.go:27`). Genes whose `Type` is somehow empty would now be
  rejected — acceptable (a typeless gene cell is exactly the corruption the guard catches),
  and the binding always sets a non-empty `row.Type` from the parser, so this only bites
  malformed direct callers, which is the intent.
- **M6 file-contention (herder sequencing note).** M8 touches `sqlref_entities.go`,
  `parse_scene.go` (and tests); the only `script/thebibites` touch is `gene.go`. The
  just-merged **M6 touched `transfer.go`, `transfer_identity.go`, `transfer_test.go`,
  `workspace.go`** in `savemutator/thebibites` — **zero overlap with M8's files.** M8 does
  NOT touch any `transfer*.go`. No merge sequencing constraint between M6 and M8 on these
  files. (Base already includes M6, confirmed via git log: commit `0565e13`.)
- **No perf signals.** All changes are O(1) per write / per scene parse (a map lookup and a
  string compare); no hot-path loop, no N+1, no unbounded scan. **No perf flag.**
