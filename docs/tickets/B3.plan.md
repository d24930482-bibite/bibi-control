## Goal

"Done" = the `save` predeclared Starlark global is gone. A save is no longer ambient: a Starlark
program obtains it by calling a new **`open()`** builtin that returns the existing `Save` value
(`save_value.go:13`, already `starlark.Value`/`HasAttrs`). The host globals dict produced for a
loaded save (today `Globals(ls)`, `bindings.go:10`) replaces its `"save": &Save{ls}` binding with
an `"open"` builtin that hands out `&Save{ls: ls}` bound to the already-loaded `LoadedSave`;
`autocommit` and the host aggregates (`sum`/`mean`/`median`) stay as standalone globals. Every
`script/thebibites` test whose embedded Starlark program referenced `save.<...>` is rewritten to
the object form (`s = open()` then `s.bibites` / `s.sql` / `s.settings` / `s.zones` / `s.commit`).
The staged-write pipeline, mirror, churn DoD, and the `Save` Go type itself are **unchanged** —
only *how a program obtains the save* changes. There is **no back-compat shim**: this is a hard
cutover. `go build ./...`, `go vet ./script/thebibites/`, and `go test ./script/thebibites/...`
all pass.

Scope is fully contained in `script/thebibites/`: `grep` confirms **no external consumer** of
`thebibites.Globals` / `thebibites.Save` outside the package.

   same file) returning `&Save{ls: ls}`; in `Globals`, swap `"save": &Save{ls: ls}` →
   `"open": starlark.NewBuiltin("open", openBuiltin(ls))`. `open()` takes no required args (accept
   and ignore an optional path so the surface reads naturally and forward-compat with Track E is
   easy: `starlark.UnpackArgs(b.Name(), args, kwargs, "path?", &path)` then ignore `path`). Keep
   `autocommit` + `hostAggregates`. Update the doc comment.
2. **(Optional) tighten `save_value.go` doc comments** only; no logic change.
3. **Rewrite the 11 test program strings.** Standard rewrite per program:
   - Prepend a bind line at module scope or as the first statement of the function that uses it:
     `s = open()`. For programs that reference the save inside a `def`, the cleanest hermetic form
     is `s = open()` at module top, then use `s` (Starlark closures capture module globals), OR
     bind inside the function. Match each program's existing structure (some define `def mutate()`
     and call it at module scope; bind `s = open()` at the same scope the save was used).
   - Mechanically replace `save.` → `s.` within that program string ONLY. Do **not** prefix

## Risks / seams

- **`open()` must bind the host-preloaded `ls`, not load a fresh one.** In the standalone path,
  `runLoaded` has already `Load`-ed `ls` and the program's staged mutations must land on *that*
  `LoadedSave` so `prepareCommit` (run.go:91) sees them. `openBuiltin` therefore captures `ls` by
  closure and returns `&Save{ls: ls}`; it must **not** re-`Load` or re-`ExtractTables`. Re-loading
  would silently break commit (staged ops on a throwaway `ls`, zero `stagedOps` on the committed
  one) — the churn tests would catch it, but the failure mode is subtle.
- **Don't prefix the standalone globals.** `autocommit`, `sum`, `mean`, `median` are module-level
  builtins, not Save attributes. Blindly `sed`-ing `save` → `s` across whole files (vs. within
  program strings, and only `save.`) would corrupt these and the `&Save{ls: ls}` Go constructions.
  Edit each program string deliberately.
- **Mirror/world-partitioning is out of scope.** B1 sets `ls.saveID = worldID`; B3 does not touch
  `ls.saveID`, `flushMirrorColumn`, or the mirror scoping. Leave the `save_id` SQL in test queries
  (e.g. analytics_test.go:75,127) as-is — those are Go-level queries against the in-memory DB, not
  the `save` global.
- **Forward-compat with Track E.** Accepting an optional `path?` arg in the standalone `open()`
  (ignored) keeps the surface shape stable so E1's `workspace.open(path)` / `world.open()` can land
  without re-churning these test programs. Do not over-build E here — B3 is global-removal + test
  rewrite only; the path-driven materialization is E1/E2's job (`B3 → E1` edge).
- **Barrier discipline.** Because `script/thebibites/` is a serialization point (plan lines
  189-193), keep this diff purely the global→object refactor + test rewrites; no opportunistic
  edits to mirror/sql/zones logic, or B1 and the C-track will collide on rebase.
- **`String()`/`Type()` cosmetics.** `Save.String()`/`Type()` return `"save"`. No test asserts on
  these (grep to confirm); leaving them avoids needless churn. If changed, run the full package
  tests to be sure.
