## The load-bearing deviation — adjudicated INDEPENDENTLY, executor is CORRECT

The plan (C3.plan.md Approach, "Why return Applied" note line 171-177) claims the
DuckDB import is a *pure transform of `wc.Applied` = `ls.session.Archive()`* — zero
extra reparse. The executor instead reparses the committed blob once
(`workspace.reparseCommitted`) at the workspace level. I verified the executor's
justification against the code, not its prose:

1. `Session.Apply` (savemutator/thebibites/session.go:128-178) clones each target
   entry's JSON (`entryUpdate.value = cloneJSON(entry.JSON)`, line 269; `cloneJSON` is a
   deep copy, json.go:26-43), mutates the clone, then REASSIGNS `update.entry.JSON =
   update.value` (line 165). It sets `s.state = StateApplied` and updates entry
   Raw/SHA256/CRC32. It does NOT touch the typed parser projections (`archive.Bibites`,
   `.Eggs`, `.Scene`, …).
2. At parse time `parseBibite` sets `bibite.Raw = asMap(entry.JSON)` (parse_entities.go:6,18)
   — `bibite.Raw` and `entry.JSON` are the SAME underlying map. But Apply reassigns
   `entry.JSON` to a fresh clone, leaving `bibite.Raw` pointing at the OLD pre-mutation map.
3. `ExtractTables` reads projections, e.g. `normalizeBibites` → `populateSQLRefFields(&row,
   bibite.Raw, "bibites")` (normalize.go:301), and `BibiteRow.Energy` is `sqlref:"body.energy"`
   (normalize_types.go:339). So `ExtractTables(key, wc.Applied)` reads `body.energy` from
   the STALE `bibite.Raw` → it would mirror the pre-mutation value.
4. session.go:21-23 documents exactly this: "Parser projections are invalid until Commit
   reparses the written save."

Conclusion: a literal `ExtractTables(wc.Applied)` would mirror stale rows. The plan's
"pure transform of wc.Applied" is architecturally false. The executor's reparse is
genuinely required for mirror correctness (or it would have needed to edit
savemutator/session.go to re-derive projections, which the plan forbids). The reparse is
NOT dead weight and does not mask a different bug.

Proof in a test (not a row count): `TestCommitWorldWorkingPartitionReflectsHead`
(workspace/commit_test.go:150-188) sets energy = 7777.0, then asserts
`bibiteEnergyInPartition(working/worldID) == 7777.0` AND
`bibiteEnergyInPartition(history/sha0) == oldValue`. It guards `oldValue != 7777` up
front. This pins the actual mutated VALUE in the working partition and the retained old
value in the prior-head history partition. (A literal-plan impl would FAIL this test by
mirroring the stale value into the working partition — which is precisely why the test is
the right pin.)

## Churn DoD — SATISFIED under the operative reading

Two DoD statements exist:
- C3 section, workspace_plan.md line 331 (plain English): "one WriteArchive, zero
  reparses, one history import + one working re-seed per commit".
- Verification section, workspace_plan.md lines 547-549 (the authoritative, testable
  definition): "churn DoD — `writeArchiveCount <= 1`, `reparseCount == 0`, one DuckDB
  import per AddWorld/commit".

My reading (based on lines 547-549, the operative definition): the churn DoD is expressed
in the named LoadedSave instrumentation counters `writeArchiveCount` / `reparseCount`
(loadedsave.go:97-104, purpose-built and asserted by tests), NOT as "zero parse calls
anywhere in the process." It constrains the **commit-serialization path**, which must not
write-then-reparse the save it just produced.

Under that reading the DoD HOLDS strictly:
- `reparseCommitted` calls the free function `tb.ParseFile` on a fresh archive; it does
  NOT increment `ls.reparseCount`. `TestRunAndCommitWorldChurn` asserts
  `ls.reparseCount == 0` (and `== 1` only under Verify) and `ls.writeArchiveCount == 1`.
- One WriteArchive per commit (prepareCommit, loadedsave.go:601-604).
- One projection feeds BOTH the additive history import and the working re-seed (the two
  `ReplaceExtractedSave` calls are the dual-key import, "one import" in the dual-key sense).
  `TestCommitWorldChurn` asserts exactly one new `save_archives` partition per commit.

HOWEVER — surfaced honestly: under the *plain-English* line 331 / the round-trip churn
philosophy, the executor's full-save reparse of the committed blob on EVERY commit is real
I/O + CPU and is a genuine deviation from the spec's evident intent of a reparse-free
commit. It is correct and necessary (see above), but the counter staying 0 is a
counter-scoping technicality, not a literal "no reparse happened." This must be FLAGGED TO
THE HUMAN AT MERGE as an accepted spec change: "commit re-projects the mirror via one
committed-blob reparse because post-Apply in-memory projections are stale." It is not a
silent relabel — the executor documents it loudly in both file headers
(commit_world.go:20-30, commit.go:99-113) and reparseCommitted's doc — but the workspace-plan
churn-DoD intent should be reconciled by the human.

## Post-commit handle drop — correct, leak-free, under lock, tested

`delete(w.worlds, worldID)` (commit.go:150) runs under `w.mu` (held since line 61). It is
required: after prepareCommit the cached session is StateApplied, so a second commit on the
same handle would fail `Stage` with "cannot stage after apply" (session.go:98-99). Dropping
the handle forces the next OpenWorld to lazy-reload a fresh stageable copy from the
now-current head. Leak-free: LoadedSave owns nothing closeable (`ls.db` is the shared
workspace DuckDB handle LoadInto never closes — confirmed working_set.go:144-145,
loadedsave.go:425). The drop uses the identical pattern as `Unload` (working_set.go:129).
`TestCommitWorldLazyLoads` exercises two SEQUENTIAL commits, asserts the handle is dropped
after #1, and that commit #2's `ParentID == rev1.ID` (proving the reload threads off the
new head, not a stale StateApplied copy). `TestCommitWorldHistoryRetained` also drives two
commits successfully — which is impossible without the drop.

## Residual concern (non-blocking, design-scoped)

OpenWorld is called before w.mu, so the working `ls` could in principle be loaded from head
H0 while a concurrent commit advances to H1 before this commit takes w.mu; this commit would
then record content-from-H0 with parent=H1 (a lost-update). This is a property of the
single-writer model the plan explicitly chose (workspace_plan.md line 521; the post-commit
handle drop forces reload between serial commits), and the plan lists "OpenWorld BEFORE
w.mu" as a required invariant — so it is in-scope-as-designed, not a regression. Serial usage
is the documented model. Noting it for the human; not a fix requirement for C3.
