# C1 — AddWorld: import → world + 1st revision + DuckDB import

Source: `docs/workspace_plan.md#c1` (the "Track C — C1 — `AddWorld(ctx, srcPath, name)`"
section, lines 299-309), read against the locked **dual-key DuckDB mirror** decision
(lines 43-52) and the per-commit "history import + working re-seed" rule C3 builds on
(lines 318-326). C1 ships the **first** revision of a world, so it establishes both
partition kinds for that world. Depends on A2 (`CreateWorld` /
`RecordRevisionAdvancingHead` / `IncBlobRef`) and B2 (the `workspace.Workspace` skeleton
this method hangs on).

Tier: medium — touches persistence (registry rows + dual-key DuckDB partitions), is the
first writer to compose blob + revision + mirror, and C2/C3/C4 fan out from it; but it is
purely additive (a new method on an existing struct, no schema/contract change) and every
underlying store op (CreateWorld, RecordRevisionAdvancingHead, ImportExtractedSave,
blobs.Put) already exists and is tested. exec=sonnet, review=opus.

## Goal

"Done" = `workspace/` grows an `AddWorld(ctx, srcPath, name) (revisionstore.World, error)`
method (plus an `AddWorldBytes(ctx, data []byte, name string)` variant) on the B2
`Workspace` struct that imports a save file as a brand-new world: it parses the save once,
creates the `worlds` registry row, stores the save bytes in the shared blobstore,
records a first revision that **advances the world head atomically** (parent=nil,
world_id=the new world), refcounts the blob, and projects the save's 46-table normalized
mirror into the per-workspace DuckDB under **both** the working-partition key (`save_id =
worldID`) and the immutable history-partition key (`save_id = revision sha256`). It runs
under the workspace mutex (single DuckDB writer) and returns the created `World` (with its
head set). No reparse beyond the single `ParseFile`. C1 implements **only** AddWorld — no
Load/Unload/Query/node ops (C2/C3/C4/D*).

## Approach

### Imports for `world.go`
`context`, `bytes`, `fmt`, `os`, `path/filepath`,
`tb "github.com/asemones/bibicontrol/saveparser/thebibites"` (the parse/extract package —
note `tb` is `saveparser/thebibites`, **not** `savemutator/thebibites`; confirmed at
`script/thebibites/loadedsave.go:22` and `duckdb/import.go:16`),
`github.com/asemones/bibicontrol/duckdb`, `github.com/asemones/bibicontrol/revisionstore`.

### `AddWorld(ctx, srcPath, name string) (revisionstore.World, error)`
1. Guard `ctx` (the store methods already call `usableContext`; passing `ctx` straight
   through is fine). Validate `srcPath != ""`.
2. `archive, err := tb.ParseFile(srcPath, nil)` (`saveparser/thebibites/parse_archive.go:13`)
   — the **single parse**. On error wrap and return. This is the only place bytes are
   parsed in C1.
3. Read the save bytes for the blob from disk: `data, err := os.ReadFile(srcPath)`. (The
   blob is the exact on-disk file the archive hash was computed over; `archive.SHA256` is
   the whole-file digest, `parse_archive.go:30/49`, so `blobstore.Put(data)` yields a ref
   whose `SHA256 == archive.SHA256` — assert this equality and fail loudly if it ever
   diverges, which would mean the file changed under us.)
4. Delegate to `importWorldFromArchive(ctx, archive, data, srcPath, name)`.

### `AddWorldBytes(ctx, data []byte, name string) (revisionstore.World, error)`
`ParseFile` is path-only (there is no in-memory archive parser — see the verify temp-file
note at `script/thebibites/loadedsave.go:638-642`). Mirror that pattern:
1. Validate `len(data) > 0`.
2. Write `data` to a temp file: `os.CreateTemp("", "bibiworld-*.zip")`, write, close,
   `defer os.Remove(tmpPath)` (exactly the `verifyRoundTrip` shape, loadedsave.go:643-655).
3. `archive, err := tb.ParseFile(tmpPath, nil)`.
4. Delegate to `importWorldFromArchive(ctx, archive, data, tmpPath, name)` (pass `data`
   directly as the blob bytes so we do not re-read the temp file).

### `importWorldFromArchive(ctx, archive *tb.Archive, data []byte, sourcePath, name string) (revisionstore.World, error)`
The shared core. **Hold the workspace mutex for the whole body** —
`w.mu.Lock(); defer w.mu.Unlock()` — because it is the single DuckDB writer per workspace
(workspace_plan.md lines 198-199/521; the `mu` field and its intended holders are
documented at workspace.go:41-45). Steps, in this order (registry/blob first, DuckDB last,
matching the "blob is a harmless orphan if no revision follows" discipline at
loadedsave.go:589-591):

1. **Create the world row.** `world, err := w.store().CreateWorld(ctx,
   revisionstore.WorldInput{WorkspaceID: w.ID(), Name: name})` (store.go:544). `world.ID`
   is the freshly allocated UUID — the stable working-partition key. Head/SimTime are NULL
   at this point (store.go:115-123).
2. **Store the blob.** `ref, err := w.blobs().Put(ctx, data)` (blobstore Store iface,
   blobstore.go:21; `FSStore.Put` at fsstore.go:98). `ref.SHA256` is the revision's content
   hash and the **history-partition key**. Sanity-check `ref.SHA256 == archive.SHA256`.
3. **Record the import script run.** `run, err := w.store().RecordScriptRun(ctx,
   revisionstore.ScriptRunInput{...})` (store.go:201) so the revision FK
   `save_revisions.script_run_id -> script_runs.id` is satisfied (the same ordering
   constraint documented at loadedsave.go:620-623). Set `ScriptSHA256` to `ref.SHA256`
   (the imported content; there is no script), `Status: "imported"`, `StartedAt` /
   `FinishedAt: now`, `StagedOps: 0`, `DryRun: false`. (Mirror the field set used by
   run.go:102-111; `ScriptSHA256` is required and validated — reuse `ref.SHA256` which is a
   valid 64-hex digest.)
4. **Derive sim_time.** `var simTime *float64`; if `archive.Scene != nil &&
   archive.Scene.HasTime { st := archive.Scene.SimulatedTime; simTime = &st }`
   (`archive.go:200-201`: `SceneState.SimulatedTime` + `HasTime`). Nil when the save has no
   scene time — A2 stores it nullable end-to-end (A2 plan lines 267-269).
5. **Record the first revision + advance head atomically.**
   `rev, err := w.store().RecordRevisionAdvancingHead(ctx, world.ID, simTime,
   revisionstore.RevisionInput{ ParentID: nil, WorldID: world.ID, SourcePath: sourcePath,
   BlobRef: ref, ScriptRunID: run.ID })` (store.go:276). This inserts the revision and sets
   `worlds.head_revision_id`+`sim_time` in one SQLite tx (store.go:294-309) — `ParentID` is
   nil for the very first revision (the world has no prior head), satisfying the
   head/parent atomicity invariant (workspace_plan.md line 519).
6. **Refcount the blob.** `if _, err := w.store().IncBlobRef(ctx, ref.SHA256); err != nil
   { return ... }` (store.go:783). This is the G1-folded refcount the eviction/GC tracks
   require on every AddWorld/commit/ingest (workspace_plan.md lines 467-471; G1 = "wire up
   call sites"). Per-content-hash semantics are A2's; C1 just increments once for this
   revision's reference to the bytes.
7. **Project the mirror — DUAL KEY.** Extract twice (a pure, cheap transform — see
   `tb.ExtractTables`, normalize.go:9) and import each under its key into the **shared
   per-workspace DuckDB** `w.duck()`:
   - **Working partition** (mutable, the open-world query/mutation target,
     `ls.saveID==worldID` invariant for C2): `working := tb.ExtractTables(world.ID,
     archive); duckdb.ImportExtractedSave(ctx, w.duck(), working)` (import.go:83 →
     `ReplaceExtractedSave` deletes-by-`save_id` then inserts, import.go:90-124; on a fresh
     world there is nothing to delete).
   - **History partition** (immutable, keyed by content sha256, retained forever so C4 can
     `history_query` across revisions): `hist := tb.ExtractTables(ref.SHA256, archive);
     duckdb.ReplaceExtractedSave(ctx, w.duck(), hist)`. Use `ReplaceExtractedSave` directly
     (not `ImportExtractedSave`) to avoid re-running `ApplyMigrations` a second time —
     migrations already ran on Open (workspace.go:226) and again inside the first
     `ImportExtractedSave`. C1 establishes the **first** history partition; C3 appends one
     per future commit without touching this one (the additive `save_id`-scoped delete is
     what keeps history immutable — workspace_plan.md lines 45-52/320-325). This is the
     dual-key discipline the plan flags as a top risk (lines 510-512): the working
     partition's key is the **world id**, the history partition's key is the **revision
     sha256** — never the same value, never overwriting each other.
8. **Return** the world with its head set. Re-read it so the returned `World` carries the
   advanced head/sim_time: `return w.store().GetWorld(ctx, world.ID)` (store.go:570). (`rev`
   from step 5 already proves the head advanced; returning the fresh `World` is what callers
   /tests assert on.)

**Failure handling:** if any step after CreateWorld fails, return the wrapped error. Do
**not** attempt destructive cleanup (no row/blob/file deletion in v1 — same stance as B2's
`Create`, workspace.go:113/150; an orphan blob or a head-less world row is harmless and
documented). Note the world row may exist head-less if step 5 fails; that is acceptable in
v1 (CreateWorld leaves head NULL by design) and a re-run creates a new world id.

### Mutex / DuckDB-writer seam
`importWorldFromArchive` is the first method to actually take `w.mu` (B2 declared it
unused, workspace.go:41-45). Take it once at the top of the shared core so both `AddWorld`
and `AddWorldBytes` are serialized and there is exactly one lock acquisition per import (do
**not** also lock in the two public wrappers — that would self-deadlock with a
non-reentrant `sync.Mutex`). The two `duckdb` imports run inside the held lock, satisfying
"one DuckDB writer per workspace" (workspace_plan.md line 521).

## Risks / seams

- **Dual-key discipline (the headline risk — workspace_plan.md lines 510-512).** The
  working partition is keyed by **world id**, the history partition by **revision sha256**.
  C1 imports both. Calling `ImportExtractedSave`/`ReplaceExtractedSave` with the wrong
  `save_id` (e.g. seeding history under the world id, or working under the sha256) silently
  breaks C3's "history accumulates / working reflects head" invariant and C4's queries.
  `tb.ExtractTables(saveID, archive)` stamps `SaveID` into every row (normalize.go:15-21,
  and the `SaveID:` field on every row builder), so the partition key is exactly the first
  arg — extract once per key. The `TestAddWorldSeedsDualKeyMirror` test pins it.
- **`ReplaceExtractedSave` deletes by `save_id` then inserts (import.go:113-118).** On a
  fresh world the working delete is a no-op; the history delete is a no-op (new sha256). It
  composes additively, so seeding the history partition does **not** disturb any other
  world's or revision's partitions — this is exactly the property C3 relies on. Do not use a
  bare INSERT path; the delete-then-insert keeps re-imports (a re-run of the same bytes)
  idempotent.
- **Single DuckDB writer.** The two `duckdb` imports must run under `w.mu`. `ReplaceExtractedSave`
  opens its own connection + `BEGIN/COMMIT` (import.go:97-122) and `db.SetMaxOpenConns` is a
  DuckDB concern, but cross-method serialization is the workspace's job (workspace_plan.md
  line 521). Take the lock in the shared core only (non-reentrant mutex — never double-lock).
- **Ordering: registry/blob before DuckDB.** Record the world + blob + revision (durable
  SQLite + content-addressed bytes) before the mirror import. If the DuckDB import fails
  after the revision is recorded, the registry is the source of truth and the mirror can be
  re-seeded later (C2's Load re-imports the working partition); the inverse (mirror written,
  no revision) would be a mirror with no provenance. A blob written without a following
  revision is a harmless orphan (loadedsave.go:589-591) that G3's GC reclaims.
- **`RecordRevisionAdvancingHead` parent is nil here (first revision only).** C1 is the only
  AddWorld path and always passes `ParentID: nil` (the world has no prior head). C3's commit
  passes the current head as parent. Do not invent a parent in C1.
- **`ScriptSHA256` must be a valid 64-hex digest** (A2/`RecordScriptRun` validates it).
  Reusing `ref.SHA256` (the content digest) satisfies this and gives the import run a
  meaningful, content-tied id. Do not pass an empty or non-hex string.
- **Blob bytes vs parsed archive identity.** `AddWorld` reads the file bytes separately from
  parsing; assert `ref.SHA256 == archive.SHA256` so a file mutated between parse and read is
  caught loudly rather than producing a revision whose blob disagrees with its mirror.
- **No reparse budget regression.** C1 parses exactly once (`ParseFile`), extracts twice
  (pure transform, not a parse), writes one blob, records one revision, and runs two DuckDB
  imports (working + history) — consistent with the churn DoD spirit
  (workspace_plan.md lines 548-549: one import per AddWorld; here "one import" = the world's
  initial dual-key seed). `AddWorldBytes` adds one temp-file write + the single parse, no
  extra parse.
