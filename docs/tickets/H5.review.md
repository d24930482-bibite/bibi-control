## int64<->uint64 normalization (focus item 3) — CONFIRMED a read-back artifact
Migrations declare UBIGINT columns (compressed_size, uncompressed_size). The appender
writes `int64` for unsigned fields (cellValue: `int64(v.Uint())`, after the overflow
guard so the value is always non-negative); DuckDB returns `uint64` on `Scan(&any)`.
So the golden (via fieldValue) yields int64(n) and the DB read-back yields uint64(n)
for the SAME n. `cellValueKey` collapses both to `i:%d` BY VALUE. This is a pure
DuckDB read-back artifact, NOT masking an H5 divergence, because: (a) both Go
extraction paths are proven byte-identical by TestFieldExtractorMatchesFieldValue;
(b) the uint64 only ever appears on the DB side, never from either extractor; (c) the
+1 mutation above still FAILED despite the normalization, proving a real value change
on these rows is caught. Sound.

for AppendRow(...driver.Value). No redundant reflection or per-cell kind switch
remains. No obvious further Go-side headroom — legitimate PASS.

## Metadata-derived column set (focus item 5, hard constraint) — PASS
Extractors are built from `fields []fieldSpec` (origin: `tb.NormalizedFieldSpec` via
`fieldSpecs`) and the struct's static `reflect.Type` resolved by `FieldByName`. No
hand-maintained parallel column list, no per-table extractor table — the extractor is
purely a function of the generated field type. memory: sqlref-generation-philosophy
honored.

## -race — SKIPPED (per policy + herder)
Purely sequential change: typed extractors, no new goroutines, no shared mutable
state, no t.Parallel. The DuckDB driver's checkptr failure under -race is a known
pre-existing false-positive that reproduces on main. A race rebuild would only
re-validate pre-existing code at multiplied cost.
