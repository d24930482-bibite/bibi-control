# M5 — Brain node↔synapse join navigation

## Goal

Add edge navigation to the existing brain element handles so a user can follow a
wire without a manual id-join. Concretely, on the `*ArrayElement` handles produced
by `b.nodes` / `b.synapses` (and the egg equivalents), expose four new read-only
attributes:

- `syn.source` → the **node** whose `NodeIndex == syn.node_in` (the synapse's input endpoint).
- `syn.target` → the **node** whose `NodeIndex == syn.node_out` (its output endpoint).
- `n.inputs`   → the synapses whose `node_out == n.node_index` (edges feeding this node).
- `n.outputs`  → the synapses whose `node_in == n.node_index` (edges leaving this node).

"Done" = these four attrs resolve correctly using the **logical `NodeIndex`** (the id
synapses reference), NOT the positional `NodeRowIndex` array slot; a synapse endpoint
that references a non-existent `NodeIndex` fails loudly (the §4/§5 loud-miss rule), never
silently returns `None`; `inputs`/`outputs` return iterable handle results over the
matching synapse elements; and existing read/append/delete behavior on the collections is
unchanged. Navigation only — no `topo()`, reachability, cycle detection, or `.where` on
the returned results (those are user-space walks / §4, explicitly out of scope).

Tier: medium — subtle invariant (NodeIndex vs NodeRowIndex; a fixture where the two
coincide would let a wrong lookup pass) + new from-scratch keyed-lookup logic; no direct
fan-out. exec=sonnet, review=opus.

## Grounding (must get right)

From `saveparser/thebibites/normalize_types.go:296-325` and
`saveparser/thebibites/normalize_metadata.go:280-301,406-492`:

- `BrainNodeRow` has TWO index-like fields: `NodeRowIndex` (friendly column
  `node_row_index`, the array slot — exposed today as the element's `index` attr) and
  `NodeIndex` (friendly column `node_index`, `sqlref:"Index"`, the LOGICAL id synapses
  reference). The join keys on `node_index`, never `node_row_index`.
- `BrainSynapseRow` has `NodeIn` (`node_in`) and `NodeOut` (`node_out`), each a logical
  `NodeIndex` value. A synapse is directed: `NodeIn → NodeOut`, carrying `Weight`/`Enabled`.
- `TypeName` is the activation function, not a role — irrelevant to the join.
- The graph may be recurrent (cycles, self-loops possible). v1 ships only navigation, so
  cycles are a non-issue; do not add or assume acyclicity.

Both `bibite` and `egg` kinds register `nodes`/`synapses` (eggs carry no stomach) via
`entitySubCollections` (`script/thebibites/attr_registry.go:329-339`), so the navigation
must be kind-generic, not bibite-only.

## Files to change

1. **`script/thebibites/subcollection.go`** — the only product file with new behavior.
   - `(*ArrayElement).Attr` (currently `:204-216`): add cases for the four navigation
     names. Dispatch is gated on which sub-collection the element belongs to so a synapse
     element only answers `source`/`target` and a node element only answers
     `inputs`/`outputs` (a stomach element answers none). Use `e.spec.table` /
     `e.spec.attr` to distinguish (e.g. `e.spec.attr == "synapses"` vs `"nodes"`).
   - `(*ArrayElement).AttrNames` (`:218-226`): append the navigation names for the
     matching element kind so `dir()` and any future generated cheatlist see them.
   - New helper methods on `*ArrayElement` (or small free functions taking `ls`,
     `kind`, `entryName`, `spec`) that perform the lookups:
     - `source`/`target`: read this synapse's `node_in`/`node_out` int64 (via the
       existing `elementAttrs["node_in"].fieldIndex` reflect path, mirroring how
       `deleteBuiltin` reads the guard field at `:242`), then resolve the node whose
       `node_index == that value` and return a fresh `*ArrayElement` for the `nodes`
       sub-collection of the SAME entity. A miss is a loud error
       (`fmt.Errorf("synapse.%s: no node with node_index %d", name, idx)`).
     - `inputs`/`outputs`: read this node's `node_index`, scan the entity's synapse rows,
       collect those whose `node_out`/`node_in` equals it, and return an iterable handle
       result (see "Returned types" below).

2. **`script/thebibites/loadedsave.go`** — add a memoized `NodeIndex → node array slot`
   lookup so `source`/`target` are not an O(N) scan per call.
   - Add a lazily-built index keyed by `(table, entryName, nodeIndexValue) → array
     ordinal` (or `→ reflect.Value`), modeled exactly on the existing `subRowIdx` /
     `geneIdx` pattern (`:51-66`, `buildSubRowIndex` `:278`): a `sync.Once` + a map field
     on `LoadedSave` (`:37-66`), built once from `subRowsFor(nodesTable, entryName)`.
   - Expose a method like `(ls *LoadedSave) nodeByIndex(kind, entryName string, nodeIndex
     int64) (reflect.Value, int64, bool)` returning the node row + its array ordinal (so
     the caller can build an `*ArrayElement` with the correct `index`). The nodes table
     name comes from the `nodes` sub-collection spec for that kind
     (`subCollectionRegistry()[kind]["nodes"].table`).
   - Reuse `subRowsFor` to source the rows; read each row's `node_index` via the nodes
     spec's `elementAttrs["node_index"].fieldIndex`. If two nodes share a `node_index`
     (malformed save), keep first-wins and do not panic — navigation is read-only.

   No change to `saveparser/thebibites/normalize_types.go` is required (it is already
   correct; it is listed in §5 only as the grounding source for the field semantics —
   confirm during implementation that `NodeIndex`/`NodeIn`/`NodeOut` are present and
   leave it untouched).

## Returned types (the one design decision)

`syn.source` / `syn.target` return a single `*ArrayElement` over the `nodes`
sub-collection of the same entity, with `index` set to the resolved node's array
ordinal (so `node.index`, `node.delete()`, and all node attrs keep working on the
returned handle). This reuses the existing handle verbatim — no new type.

`n.inputs` / `n.outputs` return MULTIPLE synapse handles. Do NOT return a full
`*ElementCollection` (its `rows()` re-reads ALL of the entity's synapses from
`subRowsFor` and ignores any filter — it has no predicate field, so it cannot represent
"the subset feeding this node"). Two acceptable options, executor picks the simpler that
passes review:

- **(preferred) Return a Starlark `*starlark.List` of `*ArrayElement` synapse handles.**
  Iterable, `len()`-able, indexable — covers every in-scope use (`for s in n.inputs`,
  `len(n.inputs)`). Each handle carries the correct synapse `index`, so `.weight`,
  `.source`, `.delete()` all work. Simplest, no new type, no stale-filter trap.
- (alternative) A minimal new in-memory iterable wrapper holding the pre-filtered
  `[]reflectValueAt`. Only choose this if a List proves insufficient — it adds a type for
  no current benefit.

Explicitly OUT OF SCOPE: `.where(...)` on the `inputs`/`outputs` result. The §5 example
`out.inputs.where("enabled")` assumes §4's unified element-collection `.where`, which does
NOT exist on `ElementCollection` today (verified: subcollection.go exposes only
`append`/`count`; the M4 merge `1e194f0` unified gene/setting keyed lookup, not brain
`.where`). Adding in-memory `.where` is a §4 task — DEFER it. In-scope filtering is
user-space: `[s for s in n.inputs if s.enabled]`. Note this deferral in the completion
summary.

## Approach (steps)

1. Confirm field/column names in `normalize_metadata.go` (already verified: nodes →
   `node_index` logical + `node_row_index` slot; synapses → `node_in`/`node_out`). Leave
   the parser untouched.
2. Add the memoized `nodeByIndex` lookup on `LoadedSave`, copying the `subRowIdx`
   `sync.Once` + lazy-build idiom (`loadedsave.go:270-299`). Read `node_index` through the
   nodes spec's `elementAttrs` reflect path; key the map by `(table, entryName)` then by
   `nodeIndex`.
3. In `subcollection.go`, add the four `Attr` cases gated by `e.spec.attr`
   (`"synapses"` → `source`/`target`; `"nodes"` → `inputs`/`outputs`). Read the local
   endpoint id from `e.row.FieldByIndex(e.spec.elementAttrs[col].fieldIndex).Int()`,
   matching the existing reflect-read style.
4. `source`/`target`: call `nodeByIndex`; on hit build `&ArrayElement{ls, kind, entryName,
   spec: nodesSpec, row: nodeRow, index: nodeOrdinal}`; on miss return a loud error.
5. `inputs`/`outputs`: iterate `subRowsFor(synapsesTable, entryName)`, match
   `node_out`/`node_in == n.node_index`, build an `*ArrayElement` (synapses spec) per
   match with its true array ordinal, append to a `*starlark.List`, freeze nothing
   (handles are immutable views). Return the list.
6. Update `AttrNames` for both element kinds.
7. Do NOT touch `api/web/app.js`: `AC_TYPE_MEMBERS` (`app.js:1715-1734`) has no
   `synapse`/`node`/`element_collection` entry today, so there is nothing to keep in sync;
   the new attrs surface via `AttrNames()` and will flow into §2's generated cheatlist when
   that lands. Adding a hand entry now would create the exact second-copy drift the SQL-ref
   philosophy forbids.

## Risks / seams

- **Perf-review: `inputs`/`outputs` scan all of the entity's synapses per call, and
  `source`/`target` must not rebuild the node→index map per call.** Memoize the
  `NodeIndex → node` lookup on `LoadedSave` (`nodeByIndex`, lazy `sync.Once`) so iterating
  every synapse of a brain is O(N) total, not O(N²). Brains are small and not unbounded,
  so this is a contained concern, but the per-element rebuild shape is the classic trap —
  the reviewer should confirm the lookup is built once, not per `source`/`target` call.
- **NodeIndex vs NodeRowIndex** is the core correctness trap. `node_row_index` is the
  array slot (the `index` attr); `node_index` is the logical wire id. The join keys on
  `node_index`. A test on a fixture where the two happen to coincide would pass with the
  wrong field — the test plan forces them to differ.
- **Loud miss, not silent None.** A dangling endpoint must error (§4/§5 rule). Returning
  `(None, nil)` would reintroduce the silent-`None` papercut §3 just removed.
- **Handle `index` must be the real array ordinal**, not the `node_index`, or
  `node.delete()` / re-navigation on the returned handle would target the wrong row. The
  delete guard at `subcollection.go:240-248` relies on `index` being the array slot.
- **Do not return a bare `*ElementCollection` for inputs/outputs** — its `rows()` re-reads
  the full unfiltered set (`subcollection.go:70-77`), silently discarding the filter.
- **Kind-generic**: dispatch off `e.spec.attr`, not a hardcoded bibite assumption, so eggs
  get the same navigation; stomach elements answer none of the four.
- **Recurrence / self-loops**: a synapse with `node_in == node_out` is legal; `source` and
  `target` both resolve to the same node, and that node appears in both its own `inputs`
  and `outputs`. Don't special-case or dedup — just report what the wiring says.
