## Spec / plan fidelity
The source spec (docs/workspace_plan.md#g3, lines 493-498) literally says
"refcount=0 and no full revision needs them". The plan deliberately identifies that
wording as the load-bearing trap (recorded G2 carryover): G2's eviction flips
tier/blob_present only and leaves refcount=1, so a refcount-gated query MISSES the
real orphan. The plan corrects the orphan condition to "no blob_present=1 row needs
the sha" and the implementation follows the plan, not the stale spec wording. This is
the correct reading of the spec's intent (reclaim the orphans G2 can leave) and is
the central correctness point. No dropped/narrowed/contradicted requirement.

## Correctness — high-stakes items verified

1. ORPHAN CONDITION IS blob_present, NOT refcount. `OrphanedBlobs` (store.go) selects
   rows with `blob_present = 0 AND sha256 NOT IN (SELECT sha256 ... WHERE
   blob_present = 1)`, collapsed to one representative per sha via `MIN(id)`. It does
   NOT reuse/widen the refcount-gated `UnreferencedBlobs`. `TestStoreOrphanedBlobs`
   constructs an evicted (refcount stays 1, asserted via rawRefcount) revision, proves
   `UnreferencedBlobs` MISSES it, proves `OrphanedBlobs` CATCHES it, and proves a
   still-present sha and a dedup-shared (full+mirror) sha are excluded, plus the
   single-representative collapse. Exactly the required proof.

2. NEVER DELETE A LIVE BLOB. `GCUnreferencedBlobs` double-gates: the SQL orphan set
   AND an under-lock `shaHasFullReference(ctx, rev.SHA256)` re-check (TOCTOU defense)
   immediately before each delete; a re-introduced full reference => Skipped, no delete.
   Inline refs are skipped (not counted). `blobstore.Delete` is idempotent
   (gcerrors.NotFound => nil, verified in fsstore.go). Blobstore delete failures are
   collected as soft `DeleteErrors` and the sweep continues; a real store/DB error from
   `OrphanedBlobs`/`shaHasFullReference`/`Has` aborts and is returned. `TestGCSkipsLiveBlob`
   proves the shared sha is neither listed nor deleted; the head/kept-full blobs survive
   in `TestGCUnreferencedBlobsReclaimsOrphan`.

3. NO RESURRECTION / NO DOUBLE REFCOUNT. `PromoteReappearedBlob` gates on
   `blobs().Has(ctx, ref)` first: absent bytes => returns (0, nil), row stays
   mirror_only (`TestPromoteReappearedBlobRefusesAbsentBytes`). On present bytes it
   promotes each mirror_only row via `PromoteRevision`, which UPDATEs tier/blob_present
   ONLY, gated `WHERE id=? AND tier='mirror_only'` — no refcount touch.
   `TestPromoteReappearedBlobRestoresMirror` re-reads refcount before/after and asserts
   it is unchanged (no-double-refcount proof) and count==1.

4. CRASH-SAFE ORDERING. GC path writes NO catalog state; it only deletes bytes the
   committed catalog already marks unneeded, and Delete is idempotent, so a crash
   mid-sweep / a re-run are no-ops. `TestGCIsIdempotent` proves second run reports
   BytesDeleted==0 with no DeleteErrors. Consistent with G2: catalog durable before/
   independent of byte deletes; a crash leaves reclaimable orphans, never a dangling
   full revision.

5. SINGLE-WRITER. Both methods take `w.mu` for the whole body; called helpers
   (`shaHasFullReference`, store/blobs methods) do not re-take `w.mu`. No call into
   EvictWorldHistory/ReconcileBlobs from the GC path. Nil-workspace and nil-ctx guards
   present.

## Minor notes (non-blocking)
- The GC loop adds an explicit `blobs().Has` stat before delete (beyond the plan's
  reliance on Delete's NotFound=>nil), so an already-absent orphan is tallied as
  Skipped rather than re-counted. This strengthens idempotency accounting; the plan's
  asserted invariants (idempotent second run BytesDeleted==0, no DeleteErrors) still
  hold. `Has` for a size-mismatched stray object returns false => GC conservatively
  skips it (won't delete a wrong-size object) — safe, not a regression.
- `TestGCSkipsLiveBlob` proves the live-blob guard via the SQL exclusion path; the
  under-lock `shaHasFullReference` TOCTOU re-check (defense-in-depth, fires only when
  SQL passes but a concurrent ref appeared) is verified by reading, not by a dedicated
  concurrency test. Acceptable: the plan frames the re-check as defense-in-depth over
  the SQL, and the load-bearing "never delete a live blob" outcome is proven.
