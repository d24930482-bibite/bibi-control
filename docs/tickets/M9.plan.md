## Goal

┬¦9 lists seven gaps. The load-bearing one is **"No supervisor"**: a node process that
dies leaves its row at `status="running"` AND leaves the runtime in `w.nodes`, so
`activeNodeForWorldLocked` keeps the world bound to a corpse and **wedges a fresh
`StartNode` for that world forever** (the only recovery today is dropping the row, which
orphans nothing because the process is already dead but still requires manual UI action).

"Done" for v1 means: when a node's OS process exits (crash, external kill, or graceful
stop the workspace didn't drive), a **supervisor reaps it** ŌĆö removes it from `w.nodes`,
drops its log ring, and flips the persisted row to a terminal status ŌĆö so (a) the world
becomes immediately re-startable, (b) `node.wait` returns a clean `node_exited` verdict
instead of an opaque IPC error, and (c) `node.status` reflects the post-death reality
within the same run. The supervisor is built on `control/controller.go` (today an empty
stub) and owned by the `Workspace`, started in `Create`/`Open` and stopped in `Close`.

The remaining ┬¦9 bullets (log persistence, real `?follow` streaming, a reconnect/reattach
endpoint, IPC event-loss backpressure, UI affordances for ingest/evict/load/unload) are
**deferred** ŌĆö see "Scope: deferred" with the rationale for each. They are either
independent of the supervisor or strictly larger follow-ups; v1 ships the seam they build on.

Tier: medium ŌĆö touches the shared `w.nodes`/status-row contract and adds a concurrency
seam (a reaper goroutine racing `StartNode`/`StopNode`/`Close`), but it is additive
supervision over an existing, well-tested lifecycle and every effect is covered by a
deterministic test using a real short-lived process. No data-shape or security change.
Reaper-vs-StartNode ordering is the one subtle invariant; an Opus reviewer guards it.

## Scope: v1 (in this ticket)

1. **Supervisor (`control/controller.go`)** ŌĆö the dead-process reaper. This is the spine
   every other v1 item hangs off.
2. **`node.wait` sees death** ŌĆö predicate ANDs only sim_time/paused/autosave today; add a
   death check so a mid-wait exit returns a clean verdict, not an opaque IPC error.
3. **Status-row reconciliation on reap** ŌĆö a reaped node's persisted row flips
   `running -> exited` (or `crashed`), so `PersistedNodes`/`handleNodesInfo` and a fresh
   `Open` see the truth instead of a stale `running`.
4. **`node.status` freshness** ŌĆö the handle caches the row captured at creation; after a
   same-run reap (or stop/kill) `node.status` lies. Re-read the live verdict.

## Scope: deferred (NOT in this ticket ŌĆö record as follow-ups)

- **Log persistence + real `?follow` streaming** (`node_logs.go`, `handleNodeLogs`,
  `app.js:657`). Persisting the ring and a true SSE/long-poll follow is a self-contained
  observability feature with its own storage decision; the 2s poll keeps working. Independent
  of the supervisor. *Deferred ŌĆö separate ticket.*
- **Reconnect/reattach endpoint** (`app.js:868` toast). Re-attaching to a still-live but
  detached process needs a new HTTP endpoint + `noderuntime.Connect` re-dial + active-set
  re-registration, and is only meaningful for processes the supervisor did NOT reap (i.e.
  still alive). Larger than v1 and orthogonal to reaping the dead. *Deferred.*
- **IPC event backpressure** (`ipc/session.go:181`). Unsolicited events drop when the
  128-buffer is full. We consume no unsolicited events today (all node commands are
  request/reply), so this is latent, not active. Changing the drop policy risks the read
  loop's liveness. *Deferred ŌĆö needs a consumer first.*
- **UI affordances for `ingest_autosave`/`evict_history`/`load`/`unload`**
  (`app.js:1697,1700`). Pure frontend wiring over bindings that already exist; no supervisor
  dependency. *Deferred ŌĆö UI-polish ticket.*

## Risks / seams

- **Reaper vs StartNode ordering (the core race).** `StartNode` drops `w.mu` around
  `noderuntime.Start`, then re-acquires it and inserts into `w.nodes`. The watcher is
  registered AFTER the insert (step 3), so a process that dies between Start and insert is
  handled by the existing post-Start re-check, not the reaper ŌĆö never register a watcher for a
  runtime that isn't in `w.nodes`. The `reapNode` same-runtime guard (compare the stored `*Runtime`
  pointer, not just the id) prevents reaping a NEW node that reused a logical id after a fast
  stop+restart.
- **Reaper vs StopNode/KillNode double-write.** Both can flip status. `StopNode` must `Cancel`
  the watcher; but the watcher goroutine may already be mid-`onExit` when Cancel arrives ŌĆö the
  `sync.Once` makes `onExit` fire at most once, and the same-runtime guard makes a late reap a
  no-op if `StopNode` already deleted the entry. Last-writer-wins on `SetNodeStatus` is
  acceptable (both terminal); just never let a stale reap overwrite a NEW node's `"running"`.
- **Close ordering.** `Close` must `supervisor.Stop()` (join all watchers) BEFORE the manual
  `w.nodes` drain, otherwise a watcher fires `reapNode`, which takes `w.mu`, while `Close`
  iterates `w.nodes` (also expects to be the sole mutator) ŌĆö at minimum a concurrent-map-write
  panic. Stop-then-drain serializes them.
- **No import cycle.** `workspace` imports `control`; `control` must NOT import `workspace`.
  Keep the supervisor callback-based (it knows only `nodeID` + an exit channel).
- **`automation.go` rebase.** This file was edited in the BASE commit; anchor all edits on the
  named functions (`waitBuiltin`, `nodeValue.Attr` status case, `waitResultDict`), not the ┬¦9
  line numbers, which have drifted.
- **Deferred items must stay deferred.** Do not start log persistence, `?follow` streaming, the
  reattach endpoint, or the IPC backpressure change ŌĆö they are out of scope and each is its own
  ticket. Shipping the supervisor seam is the whole job.

Perf: none. The supervisor is one goroutine per active node (active nodes are few and
bounded by one-per-world); no hot path, no per-query work, no unbounded scan. No perf flag.
