### The fix (verified, not just trusted)
`script/thebibites/sql.go`: `(*LoadedSave).Query` now calls `scopedQuery(query)`,
which wraps the user SQL with one shadowing CTE per normalized base table, each
filtered to this save's working partition:

	WITH bibites AS (SELECT * FROM bibites WHERE save_id = '<saveID>'),
	     bibite_body AS (SELECT * FROM bibite_body WHERE save_id = '<saveID>'),
	     … (one per normalized table) … <user query>

DuckDB binds a CTE name ahead of the catalog table of the same name, so a bare
un-joined `FROM bibites` resolves to the per-world CTE — scoping is enforced BY
CONSTRUCTION, the caller never writes a JOIN. The table list comes from
`allNormalizedTableNames()` (new in `attr_registry.go`), a `sync.Once`-cached
projection of `tb.NormalizedTables` (generated metadata, no hand-maintained list);
every `NormalizedTableSpec` carries `{Column:"save_id"}` (normalize_metadata.go), so
the `WHERE save_id = …` is valid for all shadowed tables. saveID is single-quoted
with doubled quotes for defense in depth. The old prepend-only `working_saves` CTE
(which required a caller JOIN) is gone.

I verified the fix empirically (temp tests, removed):
- The executor's `TestAutomation_WorldQueryScopedToOpenWorld` passes: two worlds
  (A=1027, B distinct; unscoped total 3224); opening A and running the un-joined
  `SELECT count(*) FROM bibites` returns A's 1027, NOT the combined total.
- Adversarial robustness check (my own): the CTE shadowing scopes NESTED references
  too — `SELECT count(*) FROM (SELECT * FROM bibites) t` returns 1027, and a
  `bibites JOIN bibite_body USING (save_id, entry_name)` stays within A (1027, no
  cross-world rows). DuckDB resolves the CTE name at every reference depth, so this
  is robust, not just a top-level FROM patch.
- The canonical un-joined tests (`OpenQueryWorkingCopy`, `OpenMutateCommitAdvancesHead`)
  are restored to the plan's form (no mandatory JOIN); the masking JOINs are gone.

Minor (non-blocking, noted): a user query that itself begins with `WITH` would
collide with the prepended CTE and error loudly — a clean Starlark error, never a
silent leak. This is identical to E1's `HistoryQuery`/`workspace.query` prepend
idiom (the accepted sibling pattern), so it is consistent and out of scope here.
