# F1 — cross-world transfer: graft MECHANICS + SAFE-BUT-CONSERVATIVE species handling (RE-PLAN after split)

Source: `docs/workspace_plan.md#f1`. Deps: **C4** (done). Drives: **F2** (expose via
workspace surface) and **F3** (cross-world species REMAP). Fan-out = 2.

> **This is a RE-PLAN.** F1 already shipped a reviewed implementation
> (`savemutator/thebibites/transfer.go`, `transfer_identity.go`, `transfer_test.go`,
> `workspace.go`). The ticket was then **split**: F1 keeps the graft mechanics +
> *fail-loud* species handling; the full species **remap** (allocate a fresh dest

## Tier
**Tier: high — silent-conflation invariant a green test missed once already + 2 dependents
(F2, F3).** Set `--model-exec opus --model-review opus`.
Rationale: the species rule is exactly the kind of correctness property a passing test can
miss — the *original* F1 shipped green with a species bug (`activeSpeciesList` add that
silently conflates a grafted bibite into the destination's unrelated same-linear-id species;
see `memory/bibites-id-spaces.md` and `docs/tickets/F1.review.md:37-41` which "verified" the
now-wrong behavior). speciesID is a **per-world LINEAR** id space, so a dest world
independently reuses the same small ids for *biologically different* species — a graft that
adopts the dest's coincidental same-id species is corruption on the common case. The new rule
inverts that behavior to fail-loud, and a too-weak verifier could re-bless a quietly-wrong
"it appended something" outcome. F2 and F3 both build directly on this surface, so a quiet
bug propagates. Net: keep exec **and** review at opus.

**Perf:** No perf flag. `AppendEntry`/`reconcileGraftIdentity` run **once per graft** (a
host/script-initiated transfer), not per request/row/event. The dest-scan helpers
(`dstBibiteWithBodyID`, `dstBibiteBodyIDs`) are O(dest entries) over an in-memory parsed
archive, invoked once per graft — not a hot path and not over unbounded/growing data. No
N+1, no tight-loop allocation. Assessed and cleared.

## Goal
"Done" = F1 ships the cross-world **graft mechanism** (settings copy, array-element append,
whole-entry bibite/egg graft) with **safe-but-conservative, fail-loud species handling**:
the graft **refuses loudly** whenever the grafted entity carries a `genes.speciesID` that
cannot be linked into the destination *without remap* — i.e. when the id is already present
in the destination (cannot prove it is the same species → conflation risk) **or** when
linking it would require fabricating/importing a species record into the destination. F1
**never silently adopts** the destination's coincidental same-id species. This conservatively
**refuses most species-bearing transfers ON PURPOSE**; **F3** lifts the restriction by
remapping (fresh dest species id + import the species record + rewrite refs). All other
identity guards (body.id collision, dangling child refs, fresh non-colliding entry name,
kind re-validation, deep clone) stay exactly as shipped.

F1 is the **savemutator-layer mechanism only**. It imports **only**
`saveparser/thebibites`, opens a source + destination `*Session`, collects from the parsed
source archive, and **stages** onto the destination `Session`. It does **not** commit, does
**not** advance a world head, and does **not** touch DuckDB / revisionstore / blobstore /
the `workspace/` package. Wiring to `workspace.transfer(...)` is **F2**.

## Scope boundary (load-bearing — unchanged from the original plan)
- `savemutator/thebibites` imports **only** `tb "…/saveparser/thebibites"`. Keep it that way.
  No `database/sql`, `duckdb`, `revisionstore`, `blobstore`, or `workspace` import.
- "collect from source" = read the source's parsed archive JSON, **not** a DuckDB query.
- F1 stops at **staging** onto the dest `Session`. No `CommitWorld` / head advance. (The
  dest `Session.Commit(path)` used in tests is the single-archive reparse, not the world head.)
- **Cross-world species REMAP is F3** (`deps: F1`). F1 must NOT attempt: allocating a fresh
  dest species id, importing/fabricating a species record into `speciesData.json`, or
  rewriting `genes.speciesID`. F1's job for any unremappable species is to **refuse**.

## What to CHANGE — the species rule (the ONLY behavioral change)

### The bug being removed
Current `reconcileGraftIdentity` (transfer_identity.go:48-54) calls `ensureActiveSpecies(sid)`
which **adds** a missing `genes.speciesID` to the destination's `activeSpeciesList`
(transfer_identity.go:124-144). Because speciesID is a **per-world linear** id, this is wrong
two ways:
- **Absent id** → "add it to activeSpeciesList" registers a species id with **no species
  record** behind it (no `recordedSpecies` entry) and no remap — a half-merged/fabricated
  species. Doing this safely is exactly F3's job (import the record).
- **Present id** → the *current* code no-ops ("already active"), silently adopting the
  destination's unrelated same-linear-id species — the conflation bug.

### The new F1 rule: FAIL-LOUD on any species that needs remap
Replace `ensureActiveSpecies` with a refusal. In `reconcileGraftIdentity`, the species branch
becomes:

1. Read the grafted entity's species via `entitySpeciesID(value)` (session.go:544 — reads
   `genes.speciesID`, works for both bibite and egg). If the entity carries **no** species id
   (`ok == false`), the graft proceeds (nothing to link) — this is the only non-refusing path.
2. If the entity **does** carry `sid`, **refuse the graft loudly**. Both sub-cases refuse,
   for distinct, named reasons:
   - **`sid` already present in the destination** (detect with `activeSpeciesIndexOf(dstSpecies.JSON, sid) >= 0`, session.go:605 — present in `activeSpeciesList`; you may additionally treat "any existing dest entity carries `sid`" as present): refuse with a message naming the conflation risk, e.g.
     `transfer: append entry: grafted genes.speciesID %d already exists in the destination; cannot prove it is the same species (speciesID is a per-world linear id), so linking would conflate distinct species — cross-world species remap is F3`.
   - **`sid` absent from the destination**: linking would require fabricating/importing a
     species record (no `recordedSpecies` entry exists for it). Refuse with a message naming
     that, e.g.
     `transfer: append entry: grafted genes.speciesID %d is absent from the destination; safely linking it requires importing the source species record (cross-world species remap is F3)`.
3. **Never** add to `activeSpeciesList`, never fabricate a record, never adopt the dest's
   same-id species. Delete `ensureActiveSpecies` entirely (it is only called from
   `reconcileGraftIdentity`; confirm no other caller via grep before removing).

**Ordering / atomicity (close the review's noted deviation while here):** the new species path
is a pure *check* with no staging, so move it **before** `nextEntryName` + `StageAppendBibite`
alongside the other reject-before-stage guards. With `ensureActiveSpecies` gone there is no
longer any "stage species then maybe fail" path, so a rejected graft leaves the destination
with **0 staged ops** by construction (no longer relying on `Apply`'s atomicity to undo a
staged species add). Verify the dest is unstaged in tests.

### Helper to fetch the dest species entry
`reconcileGraftIdentity` needs the destination `speciesData.json`. Use
`t.dst.Archive().Entry(SpeciesEntryName)` (operation.go:31 `SpeciesEntryName =
"speciesData.json"`). Decide the policy when the dest has **no** species entry or it is
**undecoded**: with the new fail-loud rule, an entity that carries a species id should still
**refuse** (you cannot prove linkage either way) — prefer refusing over silently grafting a
species-bearing entity into a save with no species table. Document the choice in a comment.
(An entity with no species id still passes — step 1.)

## Risks / seams the executor must not break
- **Do not re-introduce the conflation bug.** The species path must **refuse**, never add to
  `activeSpeciesList`, never adopt a same-id dest species, never fabricate a `recordedSpecies`
  record. If the implementation seems to need to "make the species work," that is **F3**, not
  F1 — refuse instead.
- **Conservative refusal is intended.** After this change, essentially every species-bearing
  graft refuses (both absent and present ids fail). That is the correct F1 behavior; do not
  "soften" it to make a transfer succeed.
- **Reject before stage.** With `ensureActiveSpecies` removed, the species check is a pure
  guard; run it (and the body.id / child-ref guards) before any `StageAppendBibite` /
  `StageAppend`, so a rejected graft leaves the dest with **0** staged ops by construction —
  do not regress to relying on `Apply` atomicity to unwind a staged species add.
- **Keep the rest intact.** body.id collision (random-int32 near-never net), dangling
  child-ref rejection, fresh non-colliding `nextEntryName`, kind re-validation at apply, and
  the three-layer deep clone are all correct and reviewed — do not touch them.
- **Package boundary unchanged.** Only `saveparser/thebibites` imported. No commit/head
  semantics. New code stays in the four existing files; merge-clean vs C3/`workspace/`.
- **Error specificity.** Each refusal names the offending `speciesID` and the distinct reason
  (already-present-conflation vs absent-needs-import) and points to F3, so a caller (F2) sees
  exactly why a graft was refused and a future implementer knows where the lift lives.
