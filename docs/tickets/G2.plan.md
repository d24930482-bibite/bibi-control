# G2 — EvictWorldHistory: blob-evict, keep mirror, crash-safe

**Tier: high — irreversible blob deletes gated on the G1 refcount invariant +
crash-safety ordering + 2 downstream dependents (G3 re-promotion/GC, G4 guardrails)
read the catalog/blobstore drift this op produces.**
`--model-exec opus --model-review opus`.

## Goal

Ship the memory-reclamation op for Track G: an `EvictWorldHistory(ctx, worldID, policy)`
on `*workspace.Workspace` (plus the per-revision `EvictRevisionBlob` it loops over) that,
for each eviction *candidate*, demotes the revision to `mirror_only` in the SQLite catalog
and then deletes the blobstore bytes **only when no live reference remains**. "Done" means:

1. A non-head, single-ref revision selected by the policy ends up `tier=mirror_only`,
   `blob_present=0`, its DuckDB **history partition still fully queryable**, and its
   blobstore object **gone**.
2. A world **head** and a **clone/dedup-shared** blob (refcount > 1) are **refused** for
   eviction (never deleted) — the head/shared-blob safety net.
3. The ordering is **crash-safe**: the catalog flip (`tier`/`blob_present`) is durably
   committed (SQLite fsync) **before** any byte is deleted; the blobstore delete is
   **idempotent**; and a startup `ReconcileBlobs` repairs catalog-vs-blobstore drift in
   both directions. A crash between the commit and the byte-delete leaves a recoverable
   `mirror_only`-row-with-bytes-present state that reconcile (or a re-run) cleans up — and
   **never** a deleted-but-still-referenced blob and **never** a stranded head.
4. History rows are **never** deleted (mirror_only stays queryable); there is **no**
   mirror→blob path (non-rematerialization stays structural).

## Approach

### 1. Blobstore idempotent delete (the destructive primitive)
Add `Delete` to the interface and `FSStore`. This is the only new byte-destructive code;
keep it dumb and idempotent. The workspace holds the concrete `*blobstore.FSStore`
(`w.blobsStore`) but reaches it via the `blobstore.Store` interface through `w.blobs()`
(workspace.go:95) — adding `Delete` to the interface keeps that path working.

### 2. `EvictRevisionBlob` on the workspace (per-revision crash-safe sequence)
The exact ordering — **catalog durable before bytes deleted**:

1. Call `w.store().EvictRevisionBlob(ctx, revisionID)` (G1). This runs in one SQLite tx
   that (a) refuses a head with `ErrRevisionIsHead`, (b) refuses refcount>1 with
   `ErrBlobStillReferenced`, (c) flips `tier='mirror_only', blob_present=0`, and
   **commits (fsync)**. Returns those typed errors up to the caller — the loop maps them
   to "refused", never to a delete.
2. **Only after** that commit succeeds, re-read the revision (`RevisionByID`) to get its
   `BlobRef`, then call `w.blobs().Delete(ctx, ref)`.
   - The catalog already says `mirror_only`/`blob_present=0`, so even if Delete fails or
     the process dies here, the durable truth is "bytes may be gone"; the only residue is
     possibly-still-present bytes, which `ReconcileBlobs` (and `UnreferencedBlobs`/G3) treat
     as reclaimable, never as a live blob.
   - Inline refs: Delete is a no-op; the bytes live in the row's `inline_blob` and are part
     of the catalog, not the blobstore. (`mirror_only` still means "not loadable" via G4's
     `blob_present` guard at working_set.go:64; we are demoting the tier regardless of
     inline-ness, consistent with G1's flip.)
3. A Delete error after a successful catalog commit is **logged/returned as a soft error in
   `EvictResult`, not a hard failure that implies the catalog is wrong** — the revision is
   correctly `mirror_only`; the orphan bytes are G3/reconcile's to sweep. Surface it so the
   caller knows, but do not roll back the catalog (there is nothing to roll back to — we
   *want* it evicted).

**Why this order is the crash-safe one (spell it out for the executor):** the destructive
step (byte delete) is last and idempotent; everything that *gates* it (head-check,
refcount-check) and everything that *records the decision* (the tier flip) is durable in
SQLite **before** the first byte is touched. So the only crash windows are:
- before the SQLite commit → nothing changed; re-run evicts cleanly.
- after the SQLite commit, before/within the byte delete → row is `mirror_only`, bytes may
  still exist → `ReconcileBlobs` leaves the row `mirror_only` (it does NOT re-`full` a row
  whose blob happens to still be on disk *if that blob is unreferenced*; see step 4 rule)
  and G3/`UnreferencedBlobs` reclaims the orphan. No live blob is ever deleted because the
  refcount>1 gate ran and committed first.

### 3. `EvictWorldHistory` (candidate selection + head/shared protection)
1. Resolve candidates under `w.mu` (single-writer; same whole-body-lock shape as
   `importWorldFromArchive`, world.go:81, and `Load`, working_set.go:35). Read the world
   (`GetWorld`) for its current `head_revision_id`, and `RevisionsForWorld(ctx, worldID)`
   (store.go:463, already ordered by insertion id = lineage order).
2. Build the candidate set from the policy:
   - `KeepLastN(n)`: all revisions except the last `n` in lineage order **and** except the
     head (head is always in the kept tail, but assert/skip it explicitly — defense in
     depth).
   - `OlderThanSimTime(t)`: revisions whose recorded sim_time < t (sim_time lives on the
     world head, not per-revision today; if per-revision sim_time is unavailable, scope this
     constructor to "by created_at" or document the limitation and fail loudly rather than
     guess — **executor: confirm what time field is queryable per revision before
     implementing OlderThanSimTime; KeepLastN is the must-ship policy, OlderThanSimTime is
     secondary**).
3. For each candidate call `w.EvictRevisionBlob` (step 2). Tally into `EvictResult`:
   `ErrRevisionIsHead`/`ErrBlobStillReferenced` → increment refused counters and continue
   (never abort the sweep); a real DB error → abort and return it.
4. **Do not touch DuckDB.** History partitions are keyed purely by `save_id` (= revision
   sha256) with **no `blob_present` coupling** (import.go:132 deletes by save_id only). So
   "keep the mirror" is achieved by *not writing DuckDB at all* in this op — the history
   rows simply survive. Document this explicitly: G2 is a catalog+blobstore op, never a
   mirror op.

### 4. `ReconcileBlobs` (startup drift repair — both directions)
Run after `Open` (callable by the host; do **not** auto-wire it into `Open` in this ticket
unless trivial — expose it and let the test drive it). For each side:
- **`full`/`blob_present=1` whose bytes are missing** (`!w.blobs().Has(ref)`): demote with
  `w.store().EvictRevisionBlob`? No — that re-checks head/refcount and would refuse a head.
  Instead use a direct `revisionstore` demote that mirrors the flip but is reconcile-only:
  add `(*Store) MarkMirrorOnly(ctx, revisionID)` (unconditional `tier='mirror_only',
  blob_present=0`) **or** reuse `EvictRevisionBlob` and treat its `ErrRevisionIsHead` as
  "leave full, the blob is genuinely missing for a head → that is corruption, surface it
  loudly". **Decision for the executor: a head with missing bytes is unrecoverable
  corruption — reconcile must FAIL LOUDLY (typed error), not silently demote a head.** For
  non-head full-but-missing rows, demote to `mirror_only` (a normal post-crash-mid-evict
  state).
- **`mirror_only` whose bytes reappeared by hash** (`w.blobs().Has(ref)` true) **and the
  bytes are still referenced by a `full` revision of the same sha256**: re-`full` via
  `w.store().PromoteRevision` (store.go:973, idempotent). Re-promotion is G3's main job;
  here reconcile only repairs the narrow "we crashed before deleting and the bytes are
  legitimately still needed" case. **Do not re-`full` a `mirror_only` row whose bytes are
  present but unreferenced** — those are orphans for G3 to GC, and re-`full`ing them would
  resurrect an intentionally-evicted revision.
- Return a `ReconcileResult` (demoted count, promoted count) for assertions.

1. **`TestEvictWorldHistoryKeepsMirrorDeletesBlob`** — AddWorld, commit the world 2–3×
   (so there are ≥2 non-head revisions), `EvictWorldHistory(KeepLastN(1))`. Assert:
   the older revisions are `tier=mirror_only`, `blob_present=0` (`RevisionByID`); their
   blobstore objects are **gone** (`blobs.Has(ref) == false` via a test accessor or a
   direct `blobstore.NewFSStore(blobsRoot)` re-open + `Has`); the **head** revision is

   revision's sha256 partition **still returns its rows** (prove "keep mirror" — reuse the
   query path from query_test.go).
2. **`TestEvictRefusesHead`** — `EvictWorldHistory` never demotes the head; `EvictResult`
   reports it refused; head blob still present and loadable (`OpenWorld` succeeds).
3. **`TestEvictRefusesSharedBlob`** — two revisions sharing a sha256 (refcount>1, e.g. a
   no-op recommit or an AddWorld of identical bytes / explicit `IncBlobRef`): eviction is
   refused for the shared candidate, the blob is **not** deleted (`Has == true`).
   This is the load-bearing "never delete a live blob" proof.
4. **`TestEvictThenLoadIsNotRematerializable`** — after evicting a non-head revision,
   attempting to use its bytes fails: `Load`/`OpenWorld` on a world whose head was
   *forced* to that evicted revision returns the blob-present error (working_set.go:64);
   or, simpler given head-protection, directly `blobs.Get(evictedRef)` errors. Assert the
   no-mirror-fallback property.
5. **`TestReconcileBlobsAfterCrashMidEvict`** — the crash-safety proof. Simulate the crash
   window: (a) catalog-flip-committed-but-bytes-still-present → call `MarkMirrorOnly`/the
   G1 flip on a non-head revision **without** deleting bytes, then `ReconcileBlobs`, assert
   the row stays `mirror_only` and the now-unreferenced bytes are reported by
   `UnreferencedBlobs` (G3's to sweep), and crucially **no live/head blob was deleted**;
   (b) `full`-but-bytes-missing non-head row (delete the object out-of-band) → `Reconcile`
   demotes it to `mirror_only`; (c) `full`-but-bytes-missing **head** → `Reconcile` returns
   a typed corruption error (fail loud, do not silently demote a head).
6. **`TestFSStoreDeleteIdempotent`** (in `blobstore/fsstore_delete_test.go`) — Put a
   non-inline blob, `Delete`, `Has==false`; `Delete` again → nil (idempotent); `Delete` an

## Risks / seams

- **Crash-safe ordering is the whole ticket.** The catalog tx (G1's `EvictRevisionBlob`,
  which fsyncs on commit) MUST commit before the first `blobs.Delete`. Never delete bytes
  inside the same logical step as, or before, the catalog flip. The byte delete must be
  idempotent (NotFound→nil) so a re-run after a mid-evict crash is safe.
- **Never delete a live blob.** The refcount>1 gate lives in G1's `EvictRevisionBlob` and
  is committed before any delete. Do not add a second, looser delete path that bypasses it.
  The shared-blob test (#3) is the regression guard — it must stay.
- **Head protection is double-gated** (policy never selects the head + G1 refuses it). Keep
  both; do not rely on the policy alone.
- **Keep mirror = do not write DuckDB.** History partitions have no `blob_present` column
  (import.go), so they survive automatically. Adding any DuckDB delete here would silently
  destroy the queryable mirror — the exact failure G2 must not cause. Test #1 asserts the
  history rows survive.
- **No mirror→blob fallback** (structural non-rematerialization). Reconcile re-`full`s only
  when bytes are *genuinely present and still referenced*; it must not resurrect an
  intentionally-evicted, now-orphaned blob. Test #5(a) guards this.
- **Single-writer discipline:** take `w.mu` for the eviction sweep body, exactly like
  `importWorldFromArchive`/`Load`. `w.mu` is non-reentrant — do not call a `w.mu`-taking
  method from inside the locked body (the per-revision `EvictRevisionBlob` workspace method
  should have an unlocked internal core the locked sweep calls, mirroring the
  `Load`/`OpenWorld` split, working_set.go:146).
- **Parallel-ticket overlap (flag for the herder):**
  - **D3** (planning, reload/ingest) adds *new* methods to `workspace` (likely a new
    `reload.go`/`ingest.go`) and to `script` — no file collision with `workspace/eviction.go`.
  - **F1** (reviewing, cross-world transfer) lives in `savemutator/thebibites` — no overlap.
  - The only shared *edited* files are `blobstore/blobstore.go` (+Delete on the `Store`
    interface) and `revisionstore/store.go` (+`MirrorOnlyRevisions`/`FullRevisions`/optional
    `MarkMirrorOnly`). Neither D3 nor F1 edits those per their scope, so no serialization is
    required — but if D3's plan turns out to add a blobstore or revisionstore method, the
    herder should serialize the two executors on those two files. **No product-code change
    to F1/D3 files is in this plan.**
