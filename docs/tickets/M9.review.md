## The prior FAIL is fixed

- **`handleNodesInfo` now derives liveness from the shared helper.** The inline
  verdict ladder is gone; `handlers_nodes.go:87` calls
  `ws.NodeLiveness(r.Context(), row.NodeID)` and stores the result directly in

  now gated on `Liveness == "running" || "crashed"`, re-checking `ws.Node()` only
  on that path. No separate ladder remains.
- **Vocabulary is now identical across both surfaces.** Both `handleNodesInfo`
  and Starlark `node.status` route through `Workspace.NodeLiveness`, which returns
  exactly `running` / `crashed` / `stopped` / `exited` / `detached`:
    - active + process running  → `running` (was HTTP `alive`).
    - active + exited/failed    → `crashed`.
    - inactive + persisted `stopped`/`exited`/`crashed` → that status
      (was HTTP always `detached`).
    - inactive + persisted `running` (not yet reconciled / stale Open) or no row
      → `detached`.
  `ProcessState` is a `string` type (`ipc/types.go:8-14`), so the handler's
  `st.Process.State == ipc.ProcessExited` and the helper's `== "exited"` compare
  the same underlying values — no skew between the two state checks.
- **Doc comments corrected.** `NodeLiveness` (node.go:410-422) now truthfully
  states it is called by both `handleNodesInfo` and Starlark `node.status`. The
  `reapNode` comment (node.go:361-377) no longer claims a `*Runtime`-pointer
  same-runtime guard it never had; it now describes the two actual guards
  (`sync.Once` per Watch + the `s.entries[nodeID] == e` cleanup guard). The
  `nodeInfoResponse` / handler doc comments document the shared-helper vocabulary.
