## Resolved open questions (Â§11)

- **Surface form.** Use the subscript `b.nodes["name"]` (NOT a `b.node("name")` accessor),
  matching the Â§4 collection idiom that genes already follow (`b.genes["X"]` via
  `starlark.Mapping`). No new builtin/accessor is added to `Entity`.
- **String vs int key on one collection â€” the load-bearing seam.** Starlark's `getIndex`
  (`go.starlark.net@v0.0.0-20260324133313/starlark/eval.go:682-710`) is a type switch that
  checks `Mapping` **before** `Indexable`. A value cannot be both and get both dispatches:
  once `ElementCollection` implements `Mapping`, **every** subscript (int or string) routes
  to `Get`, and `Indexable.Index` is never consulted. Therefore implement ONLY `Mapping`
  (a `Get(k)` method) and have that one `Get` branch on the key's Starlark type:
  - `starlark.String` key â†’ Desc fold lookup (this ticket's feature).
  - `starlark.Int` key â†’ positional array access (the array ordinal, i.e. the element's
    `index` attr / `NodeRowIndex` slot), with bounds checking and a loud out-of-range error.
  Today `ElementCollection` is NOT `Indexable` (no `Index(int)` method â€” verified: only
  `Iterate`/`Sequence`/`HasAttrs`), so `b.nodes[0]` does not work at all yet. Adding the int
  branch in `Get` makes positional subscript newly available too; this is additive, with no
  prior int-subscript contract to break. Do NOT add `Indexable` â€” it would be dead (Mapping
  shadows it) and is a trap for a future maintainer.
- **Scope: nodes only, not all sub-collections.** `Desc` is a node concept. Gate the string
  branch on `c.spec.attr == "nodes"`: a string key on `synapses`/`stomach` is a loud error
  ("synapses has no name key"). The int branch may apply to any sub-collection (uniform
  positional access) â€” keep it general; only the stringâ†’Desc branch is nodes-specific.
- **Cross-brain collections.** Out of scope: the only cross-brain node surface is the
  spanning `workspace.nodes` `EntityCollection` (query-backed, aggregate-only, no element
  handles â€” `script/thebibites/collection.go`, `workspace/spanning.go`), which has no
  subscript and is untouched. Desc uniqueness is therefore only required *within one brain*,
  which Â§11's grounding confirms holds. No cross-brain handling needed.
- **Hidden-node names.** `Hidden0/1/â€¦` are ordinary `Desc` values; they resolve through the
  exact same path with zero special-casing.

## Risks / seams

- **Mapping shadows Indexable (the central seam).** Implement `Mapping.Get` only and route
  BOTH key types through it; do not add `Indexable`. If a future change adds `Indexable`
  expecting int subscript to use it, it will be silently dead. The int branch living inside
  `Get` is intentional and must stay there.
- **Returned handle parity.** The desc/int `Get` must construct the `*ArrayElement` with the
  identical field set the iterator uses (`ls, kind, entryName, spec, row, index`) â€” a handle
  missing `index` or with a wrong ordinal would break `.delete()`'s stale-guard and M5 nav.
  Get the ordinal from the same source the iterator does (`c.rows()` ordinals), not from
  `NodeIndex`.
- **Ordinal vs NodeIndex.** Desc resolves to the *array ordinal* (the `index` attr /
  `NodeRowIndex` slot), NOT the logical `NodeIndex`. `foldLookup` returns the value stored in
  `byName`, so store the ordinal there. Do not confuse this with M5's `nodeByIndex` (keyed by
  logical NodeIndex). Both indexes coexist; keep them distinct.
- **Empty Desc.** Skip empty `Desc` rows when building the fold map so `nodes[""]` is a clean
  miss, not an accidental hit on an unnamed node.
- **Per-brain scope.** The fold map is keyed (table, entryName); never merge brains. Â§11's
  grounding guarantees per-brain Desc uniqueness but the build still first-wins defensively on
  a within-brain duplicate (consistent with `nodeByIndex`).
- **No mutation surface.** Do not add `HasSetKey`; string-keyed node writes are Â§5/out of
  scope. Adding it here would imply a write contract this ticket does not define.
