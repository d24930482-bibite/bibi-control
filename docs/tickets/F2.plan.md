# F2 — Expose cross-world transfer via the workspace surface; replace the DSL P3 stub

## Goal

`workspace.transfer(...)` becomes a real, bound builtin on the host-trusted automation
surface. A script selects WHAT to move with the **object DSL** — a collection/where over an
opened source world (`src.bibites.where("energy > 100")`) or a single `Entity` — and names the
**destination world**; the binding grafts the selected bibite/egg entries into the destination
world under the hood via the merged F1/F3 transfer engine (`savemutator/thebibites` `NewTransfer`
+ `CollectEntry` + `AppendEntry`, with identity reconcile + per-world species remap), then commits
the destination through the existing head-advancing commit path (`CommitWorldLoaded` /
`applyWorldCommit`). "Done" =
1. `workspace.transfer(<collection-or-entity>, dst=<worldID or world handle>)` runs end-to-end,
   advances the destination world head to a new revision whose blob contains the grafted entities,
   and re-seeds the dual-key DuckDB mirror (working + history) for the destination.
2. Species remap is exercised end-to-end through the script surface (a grafted bibite whose
   source `genes.speciesID` collides with an unrelated dest species lands under a FRESH dest
   species id, with the source species record imported — never conflated).
3. The "P3 stub" slot is replaced: `transfer` is removed from automation.go's deferred
   `(nil,nil)` set and bound for real; `docs/DSL_completion_plan.md` #8/P3 is marked implemented.
4. The user NEVER writes SQL/JOINs to transfer; selection is purely the collection/Entity DSL
   (the dsl-not-raw-sql contract).

## Key architectural facts (verified in the worktree — deps merged)

- F1/F3 transfer engine (`savemutator/thebibites/transfer.go`, `transfer_identity.go`,
  `workspace.go`) is a **pure-archive** coordinator over a `(src *Session, dst *Session)` pair.
  `NewTransfer(src, dst)` (transfer.go:45) validates both wrap an archive. `CollectEntry(entryName)`
  (transfer.go:138) deep-clones a whole bibite/egg entry off the SOURCE archive into a
  `CollectedElement`. `AppendEntry(element)` (transfer.go:204) reconciles identity, REMAPS species
  across the per-world-linear id space (bibites-id-spaces memory), allocates a fresh entry name,
  and **stages** `StageAppendBibite` onto `dst` (the destination Session). It never commits.
- The destination Session that the transfer stages onto MUST be the destination world's
  `*LoadedSave.session`, because the workspace commit path (`CommitWorldLoaded` →
  `thebibites.CommitLoadedWorld`, commit.go:113 / commit_world.go:129) commits exactly
  `ls.session` and advances the head. `ls.session` is **unexported** (`loadedsave.go:45`) — a thin
  exported seam in `script/thebibites` is required to (a) hand the transfer the source/dest
  Sessions off two `*LoadedSave` handles and (b) drive Collect→Append for a set of entry names.
- The source selector is the object DSL: `src.bibites.where(...)` is an `*EntityCollection`
  (`collection.go:74`) carrying the source world's `ls` + a resolved set of source `entry_name`s
  (`entryNames()`, collection.go:258, via `matchingEntryNames`, sql.go:935). A single `Entity`
  (`entity.go:15`) carries the source `ls` + one `entryName`. Both are reached through the
  automation `saveValue`/`Save` wrappers (automation.go:415, save_value.go:13) whose underlying
  `*LoadedSave` is the loaded SOURCE world working copy.
- `world.open()` returns a `saveValue{ws, worldID, save:*thebibites.Save}` (automation.go:283).
  `s.bibites.where(...)` returns the bare `*thebibites.EntityCollection` (save_value.go:41) — so
  the value the script passes to `transfer(...)` is a `*thebibites.EntityCollection` or
  `*thebibites.Entity`, NOT a workspace wrapper. The binding must accept those concrete types.
- Commit machinery to reuse verbatim: `Workspace.CommitWorldLoaded(ctx, worldID, RunOptions{})`
  (commit.go:113) takes `w.mu`, reads current head as parent (TOCTOU guard), calls
  `thebibites.CommitLoadedWorld` (one WriteArchive + `RecordRevisionAdvancingHead`, self-refs the
  blob — no double IncBlobRef), then `applyWorldCommit` reparses the committed blob and re-seeds
  BOTH dual-key mirror partitions. The transfer binding stages onto the dst `ls.session` then
  calls this exact method — no new commit logic.
- Lock discipline (load-bearing): `OpenWorld` takes `w.mu` itself and `w.mu` is NON-reentrant
  (working_set.go:144, commit.go:31). `CommitWorldLoaded` already does the open-before-lock dance.
  The transfer binding must resolve both source and dest working copies via `OpenWorld` (each
  acquiring its own lock) and stage onto the dst session BEFORE calling `CommitWorldLoaded`
  (which re-resolves the same cached dst handle and commits the session we staged onto).
- Per-world-query-scoping memory: the transfer does NOT query DuckDB for selection — the source
  entry names come from the collection's already-resolved push-down (`entryNames()`), which is
  scoped to the source world's `ls` by construction. No cross-world leak surface here.

## Files to change

1. **`script/thebibites/transfer_bridge.go`** (NEW — the exported seam over the unexported
   `ls.session`; keeps `savemutator` import out of `workspace`'s transfer binding).
   - `func TransferEntries(srcLS, dstLS *LoadedSave, entryNames []string) (int, error)`:
     - Build `src := srcLS.session` access and `dst := dstLS.session` via existing unexported
       field; call `mutator.NewTransfer(srcSession, dstSession)`.
     - Guard the source session is in a readable (clean / not-Applied) state: read selection off
       the SOURCE archive's parsed JSON (transfer.CollectEntry reads `Archive().Entry(name).JSON`),
       so the source must have valid projections — fail loud if `srcLS.session.State()==StateApplied`.
     - For each `entryName`: `el, err := tr.CollectEntry(entryName)`; on err return (loud);
       `if err := tr.AppendEntry(el); err != nil` return (loud — atomicity is per-AppendEntry by
       construction, see transfer.go:200-203). Increment `dstLS.stagedOps` per staged graft so the
       eventual `CommitLoadedWorld` `willWrite` gate (`ls.stagedOps > 0`, commit_world.go:189) fires
       and the commit is not treated as a no-op.
     - Return the count of grafted entries.
   - Add small exported accessor(s) ONLY as needed to reach `ls.session` from the bridge file —
     since the bridge lives IN package `thebibites`, it reads `ls.session` directly; no new public
     accessor leaks the Session type outside the package. The function signature exposes only
     `*LoadedSave` + entry names + count.
   - NOTE: keep the `dstLS.stagedOps++` consistent with how other structural stages bump it
     (`entity.go:173` does `e.ls.stagedOps++` after `stageEntityDelete`). Match that pattern.

2. **`script/thebibites/save_value.go`** (small addition): export a way for the workspace binding
   to extract the underlying `*LoadedSave` from a selector value. The selector the script passes
   is an `*EntityCollection` or `*Entity` (both hold unexported `ls`). Add:
   - `func (c *EntityCollection) SourceLoadedSave() *LoadedSave` and
     `func (c *EntityCollection) EntryNames() ([]string, error)` (delegates to `entryNames()`),
   - `func (e *Entity) SourceLoadedSave() *LoadedSave` and `func (e *Entity) EntryName() string`,
   - restrict the collection accessor to bibite/egg kinds (the only AppendEntry-eligible kinds);
     reject others loudly.
   (These are the minimal exports that let `package workspace` read the selection without touching
   `ls` internals or importing `savemutator`. Put `EntryNames`/`SourceLoadedSave` next to the
   types they belong to — collection.go and entity.go — rather than save_value.go, to keep the
   diff method-scoped; either file is acceptable.)

3. **`workspace/transfer.go`** (NEW — the workspace Go method driving the bridge + commit).
   - `func (w *Workspace) Transfer(ctx, srcLS *thebibites.LoadedSave, entryNames []string,
     dstWorldID string) (revisionstore.Revision, error)`:
     - Validate: non-nil w/srcLS, non-empty dstWorldID, non-empty entryNames (empty selection =
       loud no-op error OR documented committed=false; prefer a clean "nothing selected" outcome
       that does NOT advance the head — mirror the no-op contract).
     - `dstLS, err := w.OpenWorld(ctx, dstWorldID)` (acquires its own lock — do NOT hold w.mu).
     - Reject src==dst world (same `*LoadedSave` / same worldID) loudly — a self-transfer would
       double-stage onto a session it also reads from.
     - `n, err := thebibites.TransferEntries(srcLS, dstLS, entryNames)` — stages grafts onto the
       dst session.
     - `return w.CommitWorldLoaded(ctx, dstWorldID, thebibites.RunOptions{})` — commits the staged
       grafts, advances the dst head, re-seeds the dual-key mirror. (CommitWorldLoaded re-resolves
       the SAME cached dst handle we staged onto — verified commit.go:124-130.)
   - Lock note in the doc comment: TransferEntries stages but takes NO workspace lock; the single
     `CommitWorldLoaded` call is the only DuckDB/SQLite writer and it owns `w.mu`. Staging mutates
     only the in-memory dst session, which is the cached handle CommitWorldLoaded commits within
     the same hermetic automation cycle (no concurrent Unload/Load — same assumption E2 relies on,
     commit.go:99-103).

4. **`workspace/automation.go`** (bind the builtin; remove transfer from the deferred set).
   - Add `"transfer"` to `workspaceValue.AttrNames()` (line 70) and a `case "transfer"` in

## Risks / seams

- **Staged-session-vs-commit identity (load-bearing).** `TransferEntries` MUST stage onto the
  EXACT `*LoadedSave` that `CommitWorldLoaded` will commit — i.e. the cached `w.worlds[dstWorldID]`
  handle returned by `OpenWorld`. Resolve dst via `w.OpenWorld` ONCE and pass that pointer to both
  the bridge and the commit (CommitWorldLoaded re-resolves the same cached handle on the fast path,
  commit.go:124-130). If an Unload/Load ran between staging and commit the staged grafts would be
  lost — within one hermetic automation cycle that does not happen (same assumption E2/s.commit
  relies on). Do NOT detach/copy the LoadedSave.
- **Lock discipline.** `OpenWorld` and `CommitWorldLoaded` each take the non-reentrant `w.mu`
  themselves; `TransferEntries` takes NONE. `Transfer` must call OpenWorld → TransferEntries
  (lock-free in-memory staging) → CommitWorldLoaded, never nesting `w.mu`. Mirror the exact shape
  in commit.go:49-62.
- **Species-remap conflation (silent corruption).** The remap correctness lives in the F1/F3
  engine (transfer_identity.go) and is unit-tested; F2's job is to REACH it through the script
  surface without bypassing it. Do NOT add any "fast path" that stages an entry append directly
  (e.g. calling `StageAppendBibite` from the bridge) — always go through `tr.AppendEntry`, which
  owns the species remap + identity reconcile. A bridge that skips AppendEntry silently conflates
  species (the exact F1 v1 bug; bibites-id-spaces memory).
- **AppendEntry atomicity boundary.** AppendEntry guarantees 0-staged-ops on a SINGLE rejected
  graft (transfer.go:200-203), but the bridge loops over many entries. If graft k fails after
  grafts 0..k-1 staged, the dst session holds partial stages. Decide and document: fail the WHOLE
  Transfer loudly on the first AppendEntry error and do NOT call CommitWorldLoaded (so nothing is
  committed — the partial stages die with the in-memory session when the cycle ends / the handle is
  reloaded). This keeps "transfer is all-or-nothing at the COMMIT boundary": either all selected
  grafts commit as one revision, or none do. Make the test for a mid-loop failure assert the dst
  head did NOT advance.
- **Source archive read state.** `CollectEntry` reads `srcLS.session.Archive().Entry(name).JSON`.
  If the source world has staged structural ops (Apply not run), projections may be stale but the
  per-entry JSON is what CollectEntry uses — confirm CollectEntry reads `entry.JSON` (it does,
  transfer.go:139-145) and guard against a source session already in `StateApplied` (a committed/
  consumed handle) with a loud error rather than grafting stale bytes.
- **Selector type surface.** The script passes the BARE `*thebibites.EntityCollection` /
  `*thebibites.Entity` (save_value.go returns them directly), not a workspace wrapper — the
  binding's type switch must match the concrete `thebibites` types. A `GroupedCollection`,
  `ElementCollection`, or non-bibite/egg kind must be rejected loudly (no SQL escape hatch — the
  dsl-not-raw-sql contract; the selector is the object DSL only).
- **Perf-review: NONE.** `transfer` is once-per-call and bounded by the user-selected set, then a
  single `CommitWorldLoaded` (one WriteArchive + one dual-key re-import). The Collect→Append loop
  is O(selected entries), each an in-memory JSON clone/stage — no DB round-trip per entry, no scan
  over retained history, no N+1. A large selection grafts proportionally many entries, but that is
  the user's explicit request, not redone-per-call hidden work, so it is not a hot/unbounded path.
  No perf flag.

## Model tiering

Tier: **high** — exec opus / review opus.
Signals: **shared contract** (new public Starlark `workspace.transfer` surface + new exported
`script/thebibites` selection accessors + a new `Workspace.Transfer` Go method that E3/H1 depend
on); **persistence / data shape** (commits a new dst revision, advances the head, mutates the
save's entries + speciesData, re-seeds the dual-key DuckDB mirror); **subtle invariant** (species
remap must not conflate — a passing "count went up" test can MISS a silently-conflated species;
plus all-or-nothing at the commit boundary across a multi-entry loop); **fan-out** (E3 spanning
collections + H1 depend on F2). Any one of persistence / subtle-invariant / shared-contract alone
would round up; together they are unambiguously high. Perf flag: none (see above), so review=opus
is set by the tier, not the perf lens.
