## Per-instruction findings
1. **NULL write.** `fromStarlarkNullable` added (None→`(nil,true,nil)`, else delegates).
   `fromStarlark` ITSELF unchanged — still rejects None (grep confirms it is the only
   None-rejecting helper; `TestFromStarlarkNullable` regression-guards this). `setRowField`
   has a leading `goVal == nil` branch (`reflect.Zero`) staging nil → JSON null. NULL
   round-trip test mutation-checked via the suite. Read-back caveat is ACCURATE not a bug:
   `Entity.Attr` (entity.go:51-55) returns None only for an ABSENT 1:1 row; a null write
   zeroes a PRESENT row so it reads back 0/""/false — exactly what the doc comment states
   and the test asserts. Committed JSON null is the contract; in-run zeroed read-back is
   documented best-effort.
2. **`b.delete()` returns `MakeInt(1)`** (was `None`), matching `bulkDelete`'s
   `len(names)` count contract; `TestEntityDeleteReturnsCount` asserts type+value.
3. **`species_id` guard.** `validateSpeciesID` scans `ls.tables.Species`
   (`HasSpeciesID && SpeciesID == n`), wired into scalar `SetField`, `bulkSet`
   (once, before query — even on zero-match predicate), and `bulkSetExpr` (per-row,
   post-compute). Mutation #2 proves all three reject a nonexistent id; nothing stages.
   `species_id = None` is correctly rejected as not-nullable on a separate path
   (`TestSetFieldSpeciesIDNullRejected`), before the null short-circuit, so it cannot slip
   past as an absence-clear. Composes with the existing non-negative shape rule.
4. **THE KEY INVARIANT — structural stale-read is loud at EVERY stage site.** Enumerated
   all 11 `stagedOps++` sites and all `session.Stage*` calls. The 10 structural sites
   (entity delete, `bulkDelete`, zone delete/append, pellet delete/append, subcollection
   append/delete, transfer_bridge dst-graft + src-move) each call `markStructuralStaged()`.
   The ONE `stagedOps++` without it is `mirror.go:123` `stageScalarSet` — the mirrored
   scalar-set path, which CORRECTLY must not arm the flag. `ls.query` refuses loudly at its
   head (`structuralReadError`) before touching DuckDB, covering Query/save.sql/where/
   aggregates which all route through it. Mutation #1 proves a missed site is caught;
   mutation #3 proves the scalar path's read-after-write stays open.
5. **Behavior-change test fallout.** The 3 reordered tests
   (`TestDeleteInRunReadErrorsButVisibleAfterCommit`, `TestPelletCloneAppendNotMirrored`,
   `TestSubCollectionStructuralNotVisibleInRun`) now read BEFORE the structural edit
   (asserting it still works + dbOpenCount==1) and assert the read AFTER fails loudly with
   a "structural edit" substring — a strictly STRONGER contract than the old silent-stale
   equality, and they retain the `flushStmtCount==0` / structural invariants. The dropped
   `none value` case in `TestSetCleanErrors` is correct: None is now valid on a writable
   scalar (covered by the new null test); the non-scalar LIST reject remains. Not weakened.
6. **Scope.** `api/web/app.js` untouched; M1 read-path builders
   (`identityTable`/`fromClause`/`spanningFromExpr`/`rewritePredicate`) untouched in sql.go;
   `settings_value.go` (ergonomics bulk-write surface) untouched. Cut honored.
