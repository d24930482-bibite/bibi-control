# C4 Review — Query (working copy + full history + whole workspace), read-only enforced

Re-review after a FAIL bounce. Prior FAIL: the leading-keyword gate accepted any
statement starting with WITH and only checked for `;` chaining, so
`WITH x AS (SELECT 1) DELETE FROM save_archives` executed and deleted a retained
history-partition row (mirror corruption). Verified the fix independently; did not
trust the green suite or the prior review.

Scope: only `workspace/query.go` + `workspace/query_test.go` changed (confirmed via
`git diff main...HEAD --name-only`). No edits to `world.go`/`workspace.go`/`node.go`/
`revisionstore`/`duckdb`. `go build ./...` and `go test ./...` both green.

## 1. The new scanner IS the trust boundary — attacked from both sides

The leading-keyword gate was replaced with `scanForbidden` (query.go:281): a
whole-statement, depth-agnostic, comment/literal/identifier-aware bare-verb scanner.
`rejectChained` is fully folded in (no leftover references). I read it byte-by-byte and
exercised it directly via the unexported `ensureReadOnly` plus an end-to-end
no-corruption probe through the public `Query`/`HistoryQuery` API (probe tests written,
run, and removed — worktree left clean).

FALSE NEGATIVES — all REJECTED (errors.Is ErrReadOnlyQuery), DB row count UNCHANGED:
- Original bypass `WITH x AS (SELECT 1) DELETE FROM save_archives`.
- CTE-body mutation `WITH x AS (DELETE FROM save_archives RETURNING *) SELECT * FROM x`
  and multi-CTE `WITH a AS (SELECT 1), b AS (UPDATE t ... RETURNING *) SELECT ...`.
- Comment-hidden verbs: `/* */DELETE`, `--\nDELETE`, `/* a */ /* b */ DELETE`,
  and the non-nesting block comment `/* outer /* inner */ DELETE` (DuckDB block comments
  do not nest; the scanner closes at the first `*/` exactly as DuckDB does, so DELETE is
  exposed to both — no scanner-vs-engine divergence to exploit).
- Chained `SELECT 1; DELETE`, `SELECT 1;DELETE`, `SELECT 1 /* c */ ; /* c */ DELETE`.
- Verbs glued to punctuation/whitespace: `(DELETE`, `\nDELETE`, `\tDELETE`,
  `DELETE    FROM` (extra whitespace), case variants `dElEtE`/`select 1 ; delete`.
- ATTACH/DETACH/COPY/PRAGMA/INSTALL/LOAD/CALL/VACUUM/CHECKPOINT/EXPORT all rejected.

The bare-token mechanism is sound: punctuation/whitespace falls to the `default` byte
advance, then the next alpha byte starts a fresh `[A-Za-z0-9_]` token that is matched
exact-equal (uppercased) against `forbiddenVerbs`. A leading `WITH`/`SELECT` is required
first, and a verb anywhere thereafter (any depth) trips the gate — a legitimate
read-only query never carries one as a bare token.

End-to-end: `countBySaveID(save_archives, <sha>)` is identical before and after the
attack batch through both `Query` and `HistoryQuery` (verified live, not just via a
returned error). The in-tree `TestReadOnlyRejectsMutations` asserts the same
(baseline==after, errors.Is on both entry points).

FALSE POSITIVES — all ACCEPTED (forbidden words inside literals/identifiers/larger
tokens not flagged): `SELECT 'delete me'`, `SELECT 'DROP TABLE t'`, `SELECT "drop" FROM t`,
`SELECT "update", "delete" FROM t`, `WITH x AS (SELECT 1) SELECT * FROM x`,
`SELECT update_count`, `SELECT create_time, deleted_at`, escaped-quote literal
`'it''s a delete'`, trailing `;`, leading/trailing whitespace, and forbidden words inside
`--` / `/* */` comments. No over-blocking of legitimate read-only queries.
