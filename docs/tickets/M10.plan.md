# M10 — Workspace memory & storage lifecycle

Source: `docs/missing.md` §10. Base: `264cefd` (M9 merged — see Close() note in Risks).

## Goal

§10 lists six gaps. Several are pure resource-management ergonomics; three are
*correctness* (crash-recovery / mirror-vs-head drift / cross-store invariant). "Done"
for M10 ships the three correctness fixes as small, crash-safe, well-tested additions
that REUSE the existing single-writer + dual-key machinery, and **explicitly defers**
the three ergonomics bullets (flagged below so the herder gates the scope cut). No
existing behavior changes for the deferred items; the persisted reclamation layer
(G1/G2/G3) and the M9 supervisor drain are left untouched.

## Scope decision (v1 vs deferred — FLAGGED)

**v1 (ships):**
- **§10 bullet 5 — working-partition drift self-heal.** A failed `ReplaceExtractedSave`
  on commit (`commit.go:215-220`) leaves the working partition lagging head with no
  on-demand reproject. Add `ReprojectWorking(ctx, worldID)` that re-seeds the working
  partition from the *current head blob*, reusing the exact dual-key import the commit
  path already runs. This is correctness: the mirror is a rebuildable projection that
  currently has no rebuild lever.
- **§10 bullet 3 — `ReconcileBlobs` auto-wire into `Open`.** Post-crash catalog↔blobstore
  drift repair currently depends on the host remembering to call it (`eviction.go:271-283`,
  `workspace.go:199`). Add an opt-in `OpenAndReconcile` (Open + one ReconcileBlobs pass)
  and wire the daemon's lazy open to use it, so crash-recovery happens on first touch.
- **§10 bullet 6 — cross-store linkage check.** DuckDB `save_id` partitions and
  revisionstore `sha256` rows have no *joint* invariant; byte deletion is decoupled from
  refcount bookkeeping (crash-safe by ordering, correctness rides on a GC pass). The
  actual delete ordering is ALREADY correct (`evictRevisionBlobLocked` commits the
  catalog flip before any byte delete; G3 sweeps orphans). What is missing is a *verifier*
  that can detect a violated invariant. Add `VerifyStorageConsistency(ctx)` (a read-only
  doctor pass) that asserts: every `full`/`blob_present=1` revision's bytes are present;
  no `blob_present=1` revision has refcount < 1; no orphan sha (all-mirror_only) still has
  bytes UNLESS a concurrent GC has yet to run (reported, not failed). This makes the
  "correctness rides on GC running" implicit contract explicitly checkable.

**Deferred (FLAGGED — these §10 bullets are intentionally NOT in M10):**
- **§10 bullet 1 — in-memory working-set cap/LRU/sweeper** (`workspace.go:48`,
  `working_set.go:120-129`). Pure resource ergonomics; `w.worlds` holds only `*LoadedSave`
  values that own no closeable resource (ls.db is the shared handle). Bounded growth, not
  a correctness bug. Defer to a dedicated ticket.
- **§10 bullet 2 — `DeleteWorkspace` own-handle eviction** (`registry_ops.go:79-104`).
  The sole caller (`api/handlers_workspaces.go:103-128`) ALREADY evicts + Close()s the
  cached handle before calling the free function (verified: the prose contract is honored).
  Folding eviction into the free function needs the daemon's cache, which the function
  has no handle to. Correctness-neutral today; defer.
- **§10 bullet 4 — own-handle admin variants** of gc/evict/reconcile (`eviction.go:101,204,283`,
  `gc.go:47,124`). Pure ergonomics (avoid a full `Open` for maintenance). Defer.

Rationale for the cut: the task brief prioritizes correctness (cross-store linkage,
working-partition drift) over resource-management ergonomics (in-memory cap/LRU), and the
deferred bullets are each either already-honored-by-the-caller or non-correctness.

## Files to change

- `workspace/working_set.go` (or a new `workspace/reproject.go`) — **new** method
  `(*Workspace) ReprojectWorking(ctx, worldID string) error`. Re-seeds the working
  partition (save_id = worldID) from the current head revision's blob.
- `workspace/workspace.go` — **new** func `OpenAndReconcile(ctx, root, id string) (*Workspace, ReconcileResult, error)`:
  calls `Open` then `w.ReconcileBlobs(ctx)`; on reconcile error it Closes the just-opened
  handle and returns the error (no leaked handle). `Open` itself is left unchanged (so
  existing callers/tests that want plain Open keep it; the auto-reconcile is the opt-in
  variant). Do NOT inline reconcile into `Open` — `ErrHeadBlobMissing` must be able to
  surface as an explicit, recoverable signal, and several tests construct deliberately
  drifted states then call Open.
- `workspace/eviction.go` (or new `workspace/verify.go`) — **new** read-only method
  `(*Workspace) VerifyStorageConsistency(ctx) (StorageConsistencyReport, error)` plus the
  `StorageConsistencyReport` result type. Pure SELECTs + blobstore `Has` stats; writes
  nothing.
- `api/daemon.go` — `ws()` (lazy open, `daemon.go:45-58`): switch `workspace.Open` →
  `workspace.OpenAndReconcile`. The reconcile result is logged/discarded; a reconcile
  error fails the open (it means unrecoverable corruption — `ErrHeadBlobMissing` — which
  the caller must see). Keep the whole sequence under the existing single `d.mu` hold.

## Approach

### 1. `ReprojectWorking` (bullet 5)
Mirror the proven head-materialize + dual-key import already written TWICE in this package:
- Resolve head exactly like `Load` (`working_set.go:38-70`): `GetWorld` → nil-head guard →
  `RevisionByID(*HeadRevisionID)` → `BlobPresent` guard (`notRematerializable`) → `blobs().Get`.
- Reparse the head bytes into a fresh `*tb.Archive`. **Reuse `reparseCommitted`**
  (`commit.go:257-282`) — it already does temp-file + `tb.ParseFile` and returns an archive
  whose typed projections are fresh. Call `w.reparseCommitted(ctx, rev.BlobRef)`.
- Re-seed ONLY the working partition: `working := tb.ExtractTables(worldID, archive)` then
  `duckdb.ReplaceExtractedSave(ctx, w.duck(), working)` — identical to `commit.go:214-220`
  and `node_reload.go:374-377`. Do NOT touch the history partition (`CopySavePartition`):
  reproject is a working-partition-only repair; head's history partition already exists
  (it was written when the head revision was first recorded). This is the load-bearing
  difference from the commit path.
- Lock discipline: hold `w.mu` for the whole body (single DuckDB writer), the same
  whole-body-lock shape `Load`/`EvictWorldHistory` use. `reparseCommitted` and the duckdb
  calls never re-take `w.mu` (non-reentrant). Drop the cached `w.worlds[worldID]` handle at
  the end (like `applyWorldCommit:241`) so the next `OpenWorld` lazy-reloads from the
  now-consistent working partition — the cached `*LoadedSave` is unaffected by a DuckDB
  re-seed, but dropping it keeps the "working set reflects head" contract uniform.

### 2. `OpenAndReconcile` + daemon wiring (bullet 3)
- `OpenAndReconcile` = `Open(ctx, root, id)` then `w.ReconcileBlobs(ctx)`. On reconcile
  error: `_ = w.Close()`, return `(nil, ReconcileResult{}, err)` so a failed reconcile
  leaks no handle (mirror the partial-failure cleanup style in `Open` itself).
- `ReconcileBlobs` is already safe to call on a freshly-opened handle: it only reads the
  registry + stats blobs under `w.mu`, never touches DuckDB or the (empty) working set.
- Daemon `ws()`: replace the `workspace.Open` call. Log the `ReconcileResult` (Demoted/
  Promoted) if non-zero; surface a reconcile error as the open error (it is
  `ErrHeadBlobMissing` = unrecoverable corruption the operator must see). Whole
  check-open-store stays under one `d.mu` hold (the "never open twice" invariant).

### 3. `VerifyStorageConsistency` (bullet 6)
Read-only. Reuse existing store queries — add no new SQL if the existing surface suffices:
- Full-but-missing: iterate `store().FullRevisions(ctx)` (`eviction.go:297`), `blobs().Has`
  each → any miss is an invariant violation (record `BytesMissing`, distinguishing head
  via `store().IsRevisionHead` since a head miss is unrecoverable per `ErrHeadBlobMissing`).
- Orphan-bytes-present: iterate `store().OrphanedBlobs(ctx)` (`store.go:655`), `blobs().Has`
  each non-inline → present bytes are a *reclaimable orphan* (record `OrphanBytesPresent`),
  which is EXPECTED between an evict/crash and the next GC pass, so it is REPORTED not
  failed. This is the explicit form of "correctness rides on a GC pass running".
- refcount floor: the schema CHECK (refcount >= 0) already enforces non-negativity; a
  `blob_present=1` row at refcount 0 would be a self-ref bug. If `store()` exposes a cheap
  way to read refcounts per the full set, count them into `RefcountAnomalies`; if it does
  not, do NOT add raw SQL — note in the report struct doc that refcount integrity is
  covered by the in-tx self-ref (`store.go:1329` `incBlobRefTx`) and the schema CHECK, and
  scope this field to the data the existing queries already return. Prefer reusing
  `Revision.Refcount` already carried on the rows `FullRevisions`/`OrphanedBlobs` return.
- The report is a value struct (counts + a slice of typed findings) returned to the
  caller; `VerifyStorageConsistency` returns a non-nil error ONLY on a store/blobstore I/O
  failure, NOT on a found inconsistency (findings live in the report) — same pattern as
  `EvictResult.DeleteErrors`.

## Risks / seams

- **M9 Close() ordering (base note).** M10 does NOT touch `Close()` or the working-set
  drain. `ReprojectWorking` only mutates DuckDB + drops `w.worlds[worldID]` under `w.mu`;
  it adds no goroutine and does not race the supervisor. Leave `Close()`'s M9
  supervisor-stop-then-drain sequence (`workspace.go:283-295`) exactly as-is.
- **`w.mu` is non-reentrant.** `ReprojectWorking` and `VerifyStorageConsistency` must take
  `w.mu` themselves and call only lock-agnostic helpers (`reparseCommitted`, duckdb funcs,
  `shaHasFullReference`, store reads) — never a method that re-takes `w.mu`. Mirror the
  caution already documented in `eviction.go:113-117` / `gc.go:55-59`.
- **Reproject must not write history.** The single most likely executor error is copying
  the `CopySavePartition` call from the commit/ingest path. Reproject is working-partition-
  ONLY; writing history would either no-op (head sha already seen) or, worse, only matters
  if it diverges — the test `TestReprojectWorkingDoesNotTouchHistory` pins this.
- **`OpenAndReconcile` must Close on reconcile failure** or it leaks the DuckDB/registry/
  blob handles the just-succeeded `Open` allocated. The leak test
  (`TestOpenAndReconcileHeadMissingFailsLoud` re-opening) guards this.
- **Do not change `Open`'s signature/behavior.** Many tests call plain `Open`; the
  auto-reconcile is strictly the new opt-in variant + the daemon switch.
- **Verify is read-only.** It must issue no UPDATE/DELETE — a found inconsistency is a
  reported finding, never a repair (repair is reconcile/GC's job). Keeping verify
  side-effect-free is what makes it safe to run anywhere, including in tests asserting a
  deliberately-drifted state.
- **No perf flag.** Reproject is an on-demand repair, OpenAndReconcile runs once per
  workspace-open (not per query — the daemon caches the handle), and Verify is a manual
  doctor pass. None is a per-request/per-row hot path; reconcile's O(full+mirror) scan is
  the same shape the already-shipped G2/G3 sweeps run at admin cadence.

## Tier

Tier: **high** — crash-recovery + mirror-vs-head reprojection + cross-store
refcount/blob-present invariant are all subtle-invariant / crash-safety / shared-mirror-
contract signals (any one alone is a high-risk signal; three together). Executor and
reviewer both `opus`: do not hand the storage-lifecycle correctness code to the weaker
model, and stop any silent invariant error at review. No fan-out (deps=[], no dependents)
and no perf hot path, so no perf flag.
