# H5 — speed DuckDB append: kill per-row reflection (134k rows 564ms -> ~150ms target)

## Goal
Make the single in-memory DuckDB append for a large save materially faster (the
~564ms / 134,796-row append on the 3.1MB fixture, last link in the H2/H3/H4 import
perf chain) **without changing one byte of what lands in any table**. Done means:
the per-row append path no longer does a `reflect.Value.Kind()` type-switch per cell
(`fieldValue`); instead each table precomputes — once — a typed cell-extractor per
column derived from the generated metadata, so the per-row loop does a direct typed
read. The column set stays 100% derived from `tb.NormalizedTables` / dbtable tags
(no hand-maintained parallel list). The `uint64 > maxDriverInt64` overflow error and
pointer-deref/nil->NULL behavior are preserved exactly. `go test ./...` stays green,
`-race` clean for `duckdb`, and the append wall-time drops toward ~150ms.

## Key finding from the driver (shapes the approach — read before coding)
The ticket hypothesizes "~134k per-row cgo calls." That is **not** what the current
driver does. In `github.com/duckdb/duckdb-go/v2@v2.10503.1`:
- `Appender.AppendRow` (appender.go:373) → `appendRowSlice` already **batches into a
  Go-side `DataChunk`** and only crosses cgo once per full chunk
  (`appendDataChunk` → `mapping.AppendDataChunk`, every `GetDataChunkCapacity()`
  = 2048 rows). So cgo is already amortized (~66 chunk flushes, not 134k calls).
- The remaining **per-cell** cost is all Go-side and triple-dispatched:
  1. OUR `fieldValue` (import.go:405) — a `reflect.Value.Kind()` switch per cell
     (~1.3M calls/import).
  2. Boxing each result into `driver.Value` (an `any`) — interface conversion.
  3. The driver's own `chunk.SetValue` (data_chunk.go:60) → per-column `setFn`
     closure → `setNumeric[any,T]` / `setBool` (vector_setters.go:35,88) which do
     **another** type-switch on the boxed `any` per cell.
- **There is NO public columnar/data-chunk append on `Appender`.** Its `chunk` field
  and `appendDataChunk` are unexported; `SetChunkValue[T]` (data_chunk.go:83) works on
  a `DataChunk` *you* build but the Appender will not accept an externally-built chunk.
  `NewQueryAppender`/`NewTableAppender` still ingest via `AppendRow`. So ticket item
  #2 ("batch the cgo append / columnar vector append") is **not reachable** through
  this driver's public API at this version — document that and do not attempt it.

Conclusion: the addressable win is ticket item **#1** — eliminate OUR per-cell `Kind`
switch (and shrink boxing) by precomputing a typed extractor per column. Items #2 and
#3 are out of scope for this ticket (record why, below). Item #1 alone removes one of
the three per-cell dispatches and the redundant `Kind()` reflection; expect a solid
fraction of the 564ms, though the driver's own `setFn`+`setNumeric[any,T]` dispatch
(#3, in the dependency, unmodifiable here) sets a floor we cannot cross from Go.

## Files to change
- `duckdb/import.go` — the only product file.
  - REPLACE the per-cell `Kind` switch path. Today: `appendRows` →
    `buildFieldIndices` (keeps the index paths) → per row `rowValuesInto` →
    per cell `row.FieldByIndex(idx)` + `fieldValue(...)`.
  - NEW shape: build, ONCE per table, a `[]columnExtractor` where each entry pairs
    the precomputed `[]int` index path with a typed extractor function chosen by the
    field's static `reflect.Type` (not re-derived per row). Per row, call
    `extractor(row)` directly — no `Kind()` switch per cell.
  - `fieldValue` stays as the single source of conversion truth but is reduced to a
    fallback / is folded into the typed extractors (see Approach). `maxDriverInt64`
    and its overflow error MUST remain.
- `duckdb/import_test.go` — add the verification (value-identity + perf) tests below.
- No changes to generated metadata (`saveparser/thebibites/normalize_metadata.go`),
  row types (`normalize_types.go`), or migrations. The extractor is keyed off the row
  struct field's `reflect.Type`, which is itself derived from the generated metadata.

## Approach (steps)
1. **Build typed extractors once per table.** Extend `buildFieldIndices`
   (import.go:362) — or add a sibling `buildColumnExtractors(elem, fields)` — to
   return, per column, a closure `func(reflect.Value) (driver.Value, error)` that
   already knows the field's `reflect.Type`. Resolve the field with `FieldByName`
   (as today) to get both the `[]int` index path AND `sf.Type`; pick the extractor by
   `sf.Type.Kind()` (and pointer-element kind) at **build time**, so the per-row call
   has no switch.
2. **Typed extractor set — must cover exactly the kinds `fieldValue` handles** (the
   full universe in `normalize_types.go` is `string, float64, int64, int, bool,
   uint64`, plus pointer wrappers — confirmed by a field-type tally):
   - non-pointer `Bool` → `v.Bool()`
   - non-pointer `String` → `v.String()`
   - non-pointer signed (`Int,Int8..Int64`) → `v.Int()`
   - non-pointer unsigned (`Uint,Uint8..Uint64`) → guard `v.Uint() > maxDriverInt64`
     → **return the SAME error string** `fmt.Errorf("uint value %d overflows int64", ...)`,
     else `int64(v.Uint())`.  ← load-bearing; do not reword or wrap.
   - non-pointer `Float32,Float64` → `v.Float()`
   - **pointer of any of the above** → if `IsNil()` return `(nil, nil)` (NULL); else
     deref once and apply the element extractor. `fieldValue` loops on pointer-to-
     pointer; replicate that (loop deref) for exact parity even though current structs
     are at most one level.
   - default/unsupported kind → `fmt.Errorf("unsupported kind %s", kind)` — same as
     `fieldValue`'s default arm.
   The cleanest implementation that guarantees parity: keep ONE tiny conversion
   primitive that the extractors share, OR keep `fieldValue` and have the build step
   bind the precomputed `[]int` so the per-row hot loop is `fieldValue(row.FieldByIndex(idx))`
   minus the per-cell `Kind()` recomputation. **Either is acceptable** as long as the
   per-row path no longer recomputes the kind dispatch from scratch and output is
   byte-identical. Prefer the closure-per-column form (matches the ticket's stated
   "closure-per-column keeps the metadata-driven design" and avoids a giant switch in
   the hot loop). Keep the closure construction defensive: it reads `reflect.Value`s,
   so a wrong static-vs-dynamic kind must still error, not panic.
3. **Wire it in.** `rowValuesInto` becomes: deref the row pointer chain (unchanged
   nil-row error: `"duckdb: %s row is nil"`), require struct (unchanged error), then
   `for i, ex := range extractors { buf[i], err = ex(row) }`. Keep the
   `buf []driver.Value` reuse (import.go:311) and the existing AppendRow loop,
   appender create/Close/Clear error handling, and `CloseWithCancel(ctx)` exactly —
   only the per-cell extraction changes.
4. **Do NOT touch** the appender lifecycle, the BEGIN/COMMIT/ROLLBACK transaction in
   `ReplaceExtractedSave`, `deleteSaveRows`, or `CopySavePartition`/`copySaveSelectList`
   (H3's set-based history copy). They compose unchanged.
5. **Out of scope, record in code comment + summary:** (a) columnar/chunk append —
   the v2.10503.1 driver exposes no public path (Appender's chunk is unexported);
   (b) the overwrite DELETE (~174ms) — H3 already removed the second *extract*; the
   remaining DELETE in `deleteSaveRows` is a separate concern and lower priority; do
   not change it here.

## Tests (exactly these; reuse existing helpers)
File: `duckdb/import_test.go` (helpers already present: `repoRoot`, `countRows`,
`dumpPartition`, `scanAll`, `rowSetsEqual`, fixture path pattern).

1. **Value-identity unit test — `TestFieldExtractorMatchesFieldValue`** (fast, no DB):
   For each kind in the universe (bool, string, int/int8/16/32/64, uint/uint8/16/32/64
   including a value `> maxDriverInt64` to assert the **overflow error**, float32/64,
   and a `*T` set vs nil for at least one of each family), build the new extractor and
   assert its `(driver.Value, error)` is identical to calling the old `fieldValue` on
   the same `reflect.Value`. This is the direct fieldValue-vs-new comparison the
   Perf-review asks for and pins the overflow + nil->NULL behavior. Use a small
   throwaway struct with one field of each type so `buildColumnExtractors` is exercised
   through the real build path.
2. **Materialization-identity test — `TestAppendMaterializationUnchanged`** (fixture):
   Parse `testdata/saves/the-bibites/autosave_20260301021357.zip` via `tb.ParseFile`,
   `tb.ExtractTables`, import into an in-mem DB (new path). Compare against a golden
   the test computes from the SAME extracted save: for every `tb.NormalizedTables`
   table, `dumpPartition` the imported rows and assert they equal the values produced
   by running the extracted struct rows through the OLD `fieldValue` extraction
   (or, simplest and equally strong: import the same save twice into two DBs — one
   built at the pre-H5 commit is not available in-process, so instead assert
   `dumpPartition` rows match a row-by-row recomputation via `fieldValue` over the
   in-memory `ExtractedSave`). Every column of every row byte-identical; covers all
   ~40 tables / 134k rows. This is the "speedup must not change what lands" gate.
3. **Perf test — `TestAppendPerf`** (model on `TestCopySavePartitionPerf`,
   import_test.go:231; `t.Skip` under `-short`): split migrations out
   (`ApplyMigrations` timed separately, ~15ms), time `ReplaceExtractedSave` of one
   `ExtractTables` partition on the fixture before/after via two builds is not
   in-process; so this test just LOGS the new append wall-time (and asserts row counts
   via `countRows` per table) — the before/after numeric comparison is the reviewer's
   job per the Perf-review line. Keep it a logged measurement + a correctness floor,
   not a flaky hard threshold.

## Risks / seams (the executor must not break these)
- **DATA-INTEGRITY BOUNDARY — the review crux.** The new extractor MUST produce a
  byte-identical `driver.Value` vs `fieldValue` for every type, including: pointer
  deref + nil->NULL, all int/uint/float widths and bool/string, and the
  `uint64 > maxDriverInt64` **error** (returned, not wrapped/swallowed). Any divergence
  is silent DuckDB corruption. The unit test (#1) is the guard — it must assert the
  error case, not just happy values.
- **Metadata-derived column set (hard constraint, memory: sqlref-generation-philosophy).**
  Extractors are keyed off the row struct field's `reflect.Type`, itself from the
  generated `NormalizedFieldSpec` order. Do NOT introduce a hand-written per-table
  column list or per-table extractor table — it would drift from the schema.
- **Static-vs-dynamic kind.** Pick the extractor by the field's static `reflect.Type`
  at build time, but the closure still operates on a runtime `reflect.Value`; guard
  so a mismatch errors rather than panics (mirrors `fieldValue`'s default arm).
- **Build-once, not per-row.** The extractor slice is built once per table in
  `appendRows` (alongside `buildFieldIndices`), reused for every row. Re-deriving it
  per row would defeat the entire ticket.
- **Don't touch the transaction / appender lifecycle / H3 copy.** Only the per-cell
  extraction changes.
- **Driver dependency floor.** The driver's `setNumeric[any,T]` still type-switches
  the boxed value per cell (#3 above) and is unmodifiable here; the Go-side win is
  removing OUR redundant `Kind()` switch and reducing boxing — set expectations
  accordingly (material drop, not necessarily all the way to 150ms in one ticket).
