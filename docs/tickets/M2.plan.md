# M2 — Generated cheatlist / quick-reference page

Source: `docs/missing.md` §2 ("No cheatlist / quick-reference page"), **including** its
"Caveat — the root is deeper" paragraph. This is the FINALE ticket: it documents the
now-settled DSL surface. Deps: M1 (merged `7fe7671`: spanning genes/nodes/synapses kinds +
all-worlds `world_id` wrapper), M4 (merged `1e194f0`: `genes.get`/`["X"]` keyed-lookup
contract, removed `gene("X")`), M5 (merged `7bc040f`: brain nav `syn.source`/`target`,
`n.inputs`/`outputs`). All three are merged into this worktree's base; their surface MUST
be reflected.

Tier: medium — additive + read-only (no product behavior changes: a generator, a generated
artifact, a cross-check test, and a static HTML page). Clears every high-risk signal
(no persistence/concurrency/security/invariant change; the only "contract" touched is the
in-repo AC table, and the generator+test make drift LOUDER, not riskier). But it does NOT
clear every low-risk signal: it adds from-scratch generation+parsing logic (novel logic),
and a silently-wrong cheatlist is worse than none (it teaches a stale surface as truth), so
a strong reviewer should verify the generated content actually matches the live bindings.
Round up from low → medium on "novel generation logic whose mistakes a passing test can
miss." `--model-exec sonnet --model-review opus`.

Perf-review: none. Generation runs ONCE at `go generate` / test time over a fixed set of
static string slices (~10 types), not on any request/query/event/row hot path. No loop over
growing data, no N+1, no per-keystroke cost (the page is static HTML served by the existing
`go:embed`, no JS rebuild of the vocab). No perf flag.

## Goal

Ship an always-available, **generated** cheatlist/quick-reference for the object DSL so the
surface is no longer discovered by trial and error. "Done" =

1. a new served page (`api/web/cheatlist.html`, picked up by the existing `//go:embed web`
   in `api/daemon.go`, linked from the notebook UI) showing: the type → members graph (one
   line per DSL type), the read/write/iterate distinctions (the `genes["X"]` loud vs
   `genes.get("X")` tolerant split — note `gene("X")` is GONE per M4), spanning-vs-single-world
   (`workspace.bibites` aggregate-only; open a world to iterate), and ~6 copy-paste recipes;
2. the type→members half of that page is **GENERATED from the Go bindings' static
   `AttrNames()`**, NOT hand-maintained — emitted by a small `cmd/gen_dsl_cheatlist` tool
   wired via `//go:generate` (the project's established generation idiom, mirroring
   `cmd/gen_thebibites_schema`); and
3. a Go test fails if the JS autocomplete table (`AC_TYPE_MEMBERS` in `api/web/app.js`) OR
   the generated cheatlist drifts from the live `AttrNames()` — so the "second hand copy
   drifts like an allowlist" failure mode (the SQL-ref generation philosophy, project
   memory) is caught by CI, not by a user hitting a stale entry.

This is intentionally a partial attack on the §2 caveat's "deeper root" — see the scope
decision below. It anchors BOTH the cheatlist and the existing AC table to the Go bindings
(killing the "generate from a hand-typed source" inheritance) **for the statically-listed
top-level types**, and explicitly defers the full fold of the highlighter's second vocab
copy and the data-derived element handles.

### Scope decision (v1) — FLAG these at approval; the herder gates scope cuts

The §2 caveat poses two versions:

- **(a)** emit a generated cheatlist from the existing **hand-typed** `AC_TYPE_MEMBERS` — easy,
  but inherits a stale source (the very drift §2 warns about; `AC_TYPE_MEMBERS.settings`
  already lists a phantom `zones`, see "Drift found" below). REJECTED — it documents a copy,
  not the truth.
- **(b)** go to the real root: make `AC_TYPE_MEMBERS` itself generated from `AttrNames()` AND
  fold the highlighter's second vocab copy (`SL_METHODS`/`SL_BUILTINS`/`SL_KEYWORDS`) into the
  same source. Philosophy-correct but a much larger change (it rewrites how app.js builds its
  vocab, touching the highlighter and the AC pools, on a file M7 is also editing).

**RECOMMENDED v1 line — between (a) and (b), closer to (b)'s root without (b)'s blast radius:**
generate the cheatlist + AC `AC_TYPE_MEMBERS` block FROM the Go `AttrNames()` for the
**statically-listed top-level types** (`workspace`, `world`, `session`/save, `node`,
`collection` both read-only and mutable branches, `settings`, `setting_scope`, and the
static element handles `gene`/`setting`/`synapse`-nav and node-nav names), and add a Go
**cross-check test** that fails on ANY drift between the live `AttrNames()`, the generated
cheatlist, and the JS `AC_TYPE_MEMBERS`. Source-of-truth = Go bindings; the JS table becomes
a checked mirror.

**DEFERRED (call out in the completion summary):**

- **Fold the highlighter's second vocab copy** (`SL_METHODS`/`SL_BUILTINS`/`SL_KEYWORDS` +
  the manual kwarg top-ups in `AC_METHODS`/`AC_TOP_LEVEL`) into the generated source. This is
  the larger half of caveat-(b). v1 leaves the highlighter vocab hand-typed but ADDS a
  cross-check test arm asserting every generated member name appears in `SL_METHODS` (so a new
  binding member that the highlighter forgot is caught) — drift-detection without the rewrite.
- **Generate the DATA-DERIVED handle surfaces** — `Entity`/`ArrayElement`/`Setting`/`Settings`
  whose `AttrNames()` are computed from `attrRegistry()`/`subCollectionRegistry()`/
  `elementAttrs` (per save-format metadata), not static literals. These are ALREADY derived
  from the generated metadata (not a second hand copy), so they are not the drift root; the
  cheatlist documents their SHAPE (named typed scalar handle: `.name`/`.type`/`.value`, nav
  attrs) and points at the live `dir()` for the full per-format column list, rather than
  freezing a column list into a doc that churns per Bibites version (save-format-churn memo).
- **The full §4 unify / §7 write surface** — out of scope; the cheatlist documents the surface
  AS MERGED (M4's keyed-lookup contract), not a future one.
- Auto-regenerating app.js's `AC_TYPE_MEMBERS` in CI: v1 generates a `.gen.js` fragment +ALSO
  cross-checks the committed `AC_TYPE_MEMBERS`; wiring the generator to rewrite app.js in place
  is deferred (keeps the generator's only product artifact the cheatlist; the AC table is
  cross-checked, not overwritten, to avoid churning a file M7 edits).

### Generation runs in Go (not JS) — load-bearing decision

Generate in **Go**, at `go generate` / `go test` time, NOT in JS in the browser:

- The source of truth is the Go `AttrNames()`. A Go generator can call them directly (the
  static ones are pure functions returning literals — see "Static vs dynamic" below) and parse
  the JS `AC_TYPE_MEMBERS` for the cross-check. A JS generator would have no access to the Go
  bindings and would re-encode them (the exact second-copy drift §2 forbids).
- It matches the project's ONLY existing generation idiom: `//go:generate go run
  ../../cmd/gen_thebibites_schema` (see `saveparser/thebibites/normalize_types.go:3`), which
  parses Go source and emits Go. Reuse that shape.
- The cheatlist page is then a STATIC `.html` committed under `api/web/` and served verbatim by
  the existing `//go:embed web` (`api/daemon.go:21`) — zero browser-side build, zero new runtime
  vocab construction, no per-keystroke cost.

## Static vs dynamic AttrNames (the seam that makes Go-side generation tractable)

The top-level DSL types the cheatlist documents return `AttrNames()` as **static string
literals** — trivially enumerable by the generator and the cross-check test:

| type (JS key)   | Go value / file                                   | `AttrNames()` form |
|-----------------|---------------------------------------------------|--------------------|
| `workspace`     | `workspaceValue` (`workspace/automation.go:73`)   | static literal |
| `world`         | `worldValue` (`workspace/automation.go:414`)      | static literal |
| `session`       | `saveValue`→`Save` (`script/thebibites/save_value.go:59`) | static literal |
| `node`          | `nodeValue` (`workspace/automation.go:681`)       | static literal |
| `collection`    | `EntityCollection` (`script/thebibites/collection.go:211`) | static, **two branches** (read-only spanning vs mutable single-save) |
| `settings`      | `Settings` (`script/thebibites/settings_value.go:62`) | static literal |
| `setting_scope` | `SettingScope` (`settings_value.go:134`)          | static literal |
| `gene` handle   | `Gene` (`script/thebibites/gene.go:44`)           | static literal |
| `gene_collection`| `GeneCollection` (`gene.go:121`)                 | static literal (`["get"]`) |
| `setting` handle| `Setting` (`settings_value.go:231`)               | static literal |

DATA-DERIVED (deferred — documented by shape, not enumerated): `Entity`
(`entity.go:64`, from `attrRegistry()`+`subCollectionRegistry()`), `ArrayElement`
(`subcollection.go:238`, from `elementAttrs` + the M5 nav attrs `source`/`target` on
synapses, `inputs`/`outputs` on nodes), `Settings` material scopes, `Zone`/`Pellet`.

**Drift found (proof the generated cross-check is needed):** `AC_TYPE_MEMBERS.collection`
(`app.js:1719`) is the **mutable** branch (`set`/`set_expr`/`delete`), but a spanning
`workspace.bibites` is the **read-only** branch — the AC table can't express both. And
`AC_TYPE_MEMBERS.settings` (`app.js:1721`) lists `zones`, but `Settings.AttrNames()`
(`settings_value.go:62`) returns only `["independent","material","simulation"]` and
`Settings.Attr` has no `zones` case (`zones` lives on `session`, `save_value.go:60`) —
**a phantom member the cross-check must flag**. Decide at approval whether v1 (i) FIXES the
JS to match Go (remove phantom `zones`; split `collection` into a documented read-only vs
mutable note) or (ii) only REPORTS the drift in the test. Recommended: **(i) fix** — it is a
one-line app.js correction and leaving a known-wrong table defeats the ticket.

## Tests

The headline test is the **drift cross-check** (this is what makes the cheatlist "generated,
not a second copy" real). Put it where it can reach all three artifacts.

- `script/thebibites/cheatlist_test.go` (or a new `api/cheatlist_test.go` if it needs the
  embedded FS) — **TestCheatlistMatchesBindings** (fixture-free, fast, pure-string):
  1. Read the canonical model from `DSLCheatlistTypes()` (the live `AttrNames()`).
  2. Parse the `<!-- GENERATED -->` fenced member section out of `api/web/cheatlist.html`
     (the same artifact the generator emits) and assert it equals the canonical model — this
     fails if someone hand-edits the page or forgets to re-run `go generate`.
  3. Parse `AC_TYPE_MEMBERS` out of `api/web/app.js` (a small regex/JSON-ish extraction;
     the object is a flat `key: [..strings..]` literal — see `app.js:1715-1725`) and assert
     each key's members match the canonical model for the types AC covers (allowing
     `collection` to be the mutable branch by construction, asserted explicitly). This is the
     test that flags the phantom `settings.zones` until fixed.
  4. (Highlighter drift arm) assert every generated member name is present in `SL_METHODS`
     (parsed from `app.js:1578-1585`) — the deferred-fold drift detector.
- `cmd/gen_dsl_cheatlist` round-trip: a test (or the test above) that runs the generator into
  a temp buffer and asserts its output equals the committed `cheatlist.html` fenced section —
  the standard "generated file is up to date" guard (mirror how the repo verifies
  `normalize_metadata.go` is current, if such a test exists; otherwise this is the new guard).
- `api/web_test.go` (extend `TestServesWebIndex`): add `/cheatlist.html` to the served-asset
  loop (line ~64) so the embed + serving is proven, and assert `index.html` links to it.

Command (from the worktree root):
```
go generate ./...        # regenerate the page from the live bindings
go test ./script/thebibites/... ./workspace/... ./api/...
```
The cross-check is the load-bearing test: run it AFTER `go generate` (a stale committed page
must fail the test, proving the generator is the source of truth). These packages are NOT
import-bound for these tests (no `AddWorld`/save import needed — the static `AttrNames()` and
string parsing are pure), so they are cheap; do not add a fixture-loading test.

## Risks / seams (do not break)

- **The point of the ticket is to NOT hand-maintain.** Every member name on the page MUST flow
  from `AttrNames()` via `DSLCheatlistTypes()`. If the executor types a member list into the
  HTML by hand "to save time," the cross-check test must still catch it (it parses the fenced
  section back and diffs against the live bindings) — keep that test the gate, not a comment.
- **`collection` is two surfaces.** A single `AC_TYPE_MEMBERS.collection` cannot represent both
  the read-only spanning branch and the mutable single-save branch. The page MUST show BOTH
  (spanning = aggregate/where/group_by only; single-save adds set/set_expr/delete) and the
  cross-check must assert AC's `collection` equals the MUTABLE branch by construction (not
  treat the missing spanning members as drift).
- **The `workspace.nodes` asymmetry (M1).** `workspace.nodes` is the host process-node
  builtin, NOT the brain-node aggregate (`workspace/automation.go:88-91`). The page's spanning
  section must say so explicitly; do NOT let the generator's flat "workspace has a `nodes`
  member" line imply it is the brain aggregate. The recipes for all-worlds brain stats use
  `world.nodes` / `workspace.synapses`.
- **M4 removed `gene("X")`.** Do not document `b.gene("X")` anywhere — it is gone. The gene
  surface is `b.genes["X"]` (loud), `b.genes.get("X")` (tolerant), `b.genes["X"]=v`, iterate,
  `len`. The handle reads `.name`/`.type`/`.value` (`gene.go:44`).
- **M5 nav attrs are conditional** (`subcollection.go:244-249`: `source`/`target` only on
  synapses, `inputs`/`outputs` only on nodes). If the page documents element nav, scope it to
  the right handle; do not list `source` on a node.
- **Save-format churn (project memory).** Per-entity/per-element column lists
  (`Entity`/`ArrayElement`/`Setting`) come from generated metadata that churns per Bibites
  version. The cheatlist documents their SHAPE and points at live `dir()`; it must NOT freeze
  a column list into the page (that would re-create the drift this ticket fights, one layer
  down). This is why those data-derived surfaces are explicitly deferred from generation.
- **Generated-file staleness.** The committed `cheatlist.html` and the generator MUST agree;
  the round-trip test is the tripwire. The executor must run `go generate ./...` and commit the
  result — note this in the completion summary.
- **Do not leave a build artifact in the repo** (orchestrator gotcha, project memory): the
  generator is run via `go run`/`go generate`; do not `go build` a binary into the repo root.
