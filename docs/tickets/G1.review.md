## Plan → implementation: an intentional, correct deviation
The plan's literal mechanism was "insert at schema default 0, then `+1 WHERE sha256`".
Taken literally that BREAKS the load-bearing per-hash dedup invariant: for a hash already
shared by N rows at value V (V>=1, since they each self-ref'd), inserting the new row at 0
then running `UPDATE ... +1 WHERE sha256` bumps the N pre-existing rows to V+1 but the new
row only to 1 — unequal counts across rows sharing one sha256.

The executor recognized this and seeded the new row's refcount in the INSERT with
`COALESCE((SELECT MAX(refcount) FROM save_revisions WHERE sha256 = ?), 0)` BEFORE the
shared `incBlobRefTx` `+1`. So the new row first matches the existing sharing rows, then
all sharing rows (new + pre-existing) are raised by one together, ending equal at V+1.
This is a correctness IMPROVEMENT over the literal plan, and it satisfies the plan's
actual stated contract (Goal: "+1 for every row sharing that hash"; new-test requirement:
both dedup-sharing rows read 2). The doc comment honestly documents the MAX-seeding. Not
drift.

## Load-bearing seams checked
- Per-hash dedup invariant: new hash -> 1; two revisions sharing one BlobRef -> BOTH read
  2 (TestStoreRecordEstablishesBlobRef asserts rawRefcount on both row ids == 2).
  Inductively MAX over equal-valued sharing rows = the common value; the subquery runs
  pre-insert (excludes the new row), incBlobRefTx runs post-insert (includes it).
  IncBlobRef/DecBlobRef are whole-set UPDATEs that preserve cross-row equality. Sound.

  the seeding subquery and incBlobRefTx(ctx, q, ...) run on the same tx q and commit
