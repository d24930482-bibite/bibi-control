# B1 — LoadedSave seam: explicit world id + injected DB

Source: `docs/workspace_plan.md#b1` (lines 252–266). Base: `main` (B3 already merged —
`Globals(ls)` already hands out a `Save` via `open()`; there is no `save` global).

## Goal

Add a constructor on `LoadedSave` that the future `workspace/` package drives: it parses a
save file but takes an **explicit, stable world id** as the partition key and an **injected,
shared per-workspace `*sql.DB`** instead of deriving `saveID` from `archive.SHA256` and opening
a fresh in-memory DuckDB on first query. The existing standalone `Load(path)` path is untouched.
"Done" = a `LoadInto(path, worldID string, db *sql.DB) (*LoadedSave, error)` exists; the returned
`LoadedSave` has `ls.saveID == worldID` and `ls.db == db`; the lazy `openDB` short-circuits the
in-memory import because `ls.db != nil`; the in-run query and mirror write-through paths are
correctly scoped to `save_id = worldID` against the injected DB; and a focused test proves a
mutation→query round-trip works through the injected handle while the standalone `Load` behavior
and the churn DoD are unchanged. **B1 does not import any rows into the injected DB** — that is
C1/C3's job (history + working partitions). B1 only wires the seam.

## Approach

1. **Extract a shared builder.** Today `Load` (loadedsave.go:120–138) does: `tb.ParseFile` →
   pick `saveID` (`archive.SHA256`, fallback `archive.FileName`) → build the `LoadedSave` literal
   (`path`, `saveID`, `archive`, `tables: tb.ExtractTables(saveID, archive)`,
   `session: mutator.NewSession(archive)`, `willCommit: true`) → `ls.buildAccess()`. Pull the
   "from a parsed archive + chosen saveID, build the struct + buildAccess" part into a private
   helper, e.g.:

       func newLoadedSave(path, saveID string, archive *tb.Archive, db *sql.DB) *LoadedSave

   It sets every existing field exactly as `Load` does today, plus `db: db` (nil for the
   standalone path), and calls `buildAccess()`. `Load` then becomes: parse → derive `saveID` →
   `return newLoadedSave(path, saveID, archive, nil), nil` — byte-identical behavior.

2. **`LoadInto`.** Implementation:
   - `archive, err := tb.ParseFile(path, nil)` (same as `Load`; loadedsave.go:121).
   - Validate `worldID != ""` and `db != nil`; return a wrapped error otherwise (the workspace
     always supplies both — a nil/empty here is a programming error worth surfacing loudly, in
     keeping with the save-format-churn "fail loud/localized" stance).
   - `return newLoadedSave(path, worldID, archive, db), nil`.
   - **Key the projection by `worldID`, not the file hash.** Because `newLoadedSave` passes
     `saveID` to `tb.ExtractTables(saveID, archive)` (loadedsave.go:133 → normalize.go:9), the
     in-memory `ExtractedSave` rows and `ExtractedSave.Archive.SaveID` (normalize.go:16) are
     stamped with `worldID`. This is required for consistency: the mirror locator flush
     (`flushMirrorColumn`, mirror.go:171) and the query `whereClause`/`fromClause` (sql.go:237–238,
     223) all filter `save_id = ls.saveID = worldID`, so the in-memory rows that back mirror
     locators must carry the same key as the DuckDB rows C1/C3 seeded under `save_id = worldID`.

3. **Injected DB short-circuits the in-memory import (no new code needed).** `openDB`
   (loadedsave.go:379–395) already early-returns `ls.db` when `ls.db != nil` *before* calling
   `duckdb.OpenAndImport(ctx, "", ls.tables)`. With `LoadInto` setting `ls.db = db`, the first
   analytical query reuses the shared handle and never imports in-memory. The world's rows are
   expected to already be present under `save_id = worldID` (seeded by C1/C3); B1 does not import.
   Note `dbOpenCount` stays 0 on the injected path because the import branch is skipped — keep
   that as-is (the counter measures the in-memory lazy-open, which the injected path bypasses);
   the seam test asserts this so the contract is pinned.

4. **Mirror scoping is already correct.** `flushMirrorColumn` scopes its UPDATE with
   `<table>.save_id = ?` bound to `ls.saveID` (mirror.go:171–172). With `ls.saveID = worldID`,
   write-through lands only on the working partition. No change to `mirror.go`. (This confirms the
   plan's correction of the draft "key fix" — it is already done.)

5. **Leave `Load`, `RunAndCommit`, `runLoaded`, `Globals`, `recordRevision` untouched.** Parent
   threading (`RecordRevisionAdvancingHead`, the always-nil `ParentID` at loadedsave.go:582–583)
   and `WorldID` on `RevisionInput` are **A2/C3 scope**, not B1. B1 is purely the load-time seam.

## Risks / seams

- **Dual-key discipline.** B1 sets the *working-partition* key only (`ls.saveID = worldID`). It
  must NOT touch history partitions (keyed by revision sha256) — B1 does no import at all, so this
  is safe by construction. Do not "helpfully" call `ImportExtractedSave` inside `LoadInto`; the
  workspace owns when/where rows are imported (and confusing working vs. history keys is the
  headline failure mode in the plan's risks section).
- **`ExtractTables` must be keyed by `worldID`.** If `LoadInto` ever derived `saveID` from the
  hash (like `Load`) while seeding the DB under `worldID`, in-run reads/mirror would query a key
  that has no rows. The shared builder makes the key a single parameter to avoid this drift.
- **Shared DB is caller-owned.** `LoadInto` must not `Close` the injected `db` (the workspace
  reuses it across worlds). Nothing in `LoadedSave` closes `ls.db` today; keep it that way.
- **Single-writer.** The injected DuckDB handle is shared across worlds in one workspace; mutating
  ops must be serialized by the workspace mutex (B2 concern). B1 introduces no concurrency itself
  but must not assume exclusive ownership of `db`.
- **Counter semantics.** `dbOpenCount` is incremented only on the in-memory open branch
  (loadedsave.go:388). The injected path intentionally leaves it 0; existing churn assertions in
  commit_test.go stay valid because they use the standalone `Load`. Do not move the increment
  above the `ls.db != nil` early return.
- **Package serialization point.** `script/thebibites/` is a known contention scope (B1/B3/C-track
  all touch it). Keep this diff method-scoped and additive (one new constructor + one private
  builder extraction + one new test file) so it layers cleanly on B3.
