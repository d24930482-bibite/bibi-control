## Non-negotiables — all verified IN CODE
1. Selection is the object DSL, never SQL/JOINs. transferBuiltin's type switch accepts ONLY
   *thebibites.EntityCollection / *thebibites.Entity; everything else (grouped/element collection,
   string) is rejected loudly. EntryNames() delegates to entryNames() -> matchingEntryNames(), which
   is ENGINE-generated SQL over the user's .where predicate, scoped to the source ls by
   construction. No raw-SQL/internal-schema path is exposed to the caller.
   Tests: TestAutomation_TransferNonEntitySelectorRejected (grouped + string both rejected).
2. Every grafted entry routes through tr.AppendEntry. transfer_bridge.go::TransferEntries loops
   CollectEntry -> AppendEntry per name and NEVER calls StageAppendBibite directly. The species
   remap + identity reconcile therefore always fire (no F1-v1 conflation bypass).
3. All-or-nothing at the commit boundary. On any CollectEntry/AppendEntry error TransferEntries
   returns the error; Transfer returns WITHOUT calling CommitWorldLoaded, so partial in-memory
   stages die with the cycle and the dst head never advances.
   Test: TestTransferAllOrNothingOnBadEntry (one valid + one missing entry) independently asserts
   the dst head ID is unchanged AND the dst working bibite count is unchanged (==3).
4. Stages onto the exact cached dst *LoadedSave, never nests w.mu. Transfer does
   OpenWorld (own lock; fast-path returns the cached w.worlds[id] handle) -> TransferEntries
   (lock-free, in-memory) -> CommitWorldLoaded (takes w.mu, re-resolves the SAME cached handle via
   OpenWorld). Self-transfer is caught by pointer equality (dstLS == srcLS) because OpenWorld
   returns the same cached handle for the same world id.
   Test: TestAutomation_TransferSelfTransferRejected (loud "same world" + head unmoved).

## Perf lens
Plan carries "Perf-review: NONE" and the surface confirms it: transfer is once-per-call, O(selected
entries) of in-memory JSON clone/stage with no per-entry DB round-trip, then a single
CommitWorldLoaded (one WriteArchive + one dual-key re-import). No N+1, no per-call work scaling with
retained history. The species-id allocator (dstSpeciesIDUsage) re-scans the dst species table + live
entities + staged ops per graft — O(entries) per graft, O(n^2) over a multi-graft loop — but the
loop is bounded by the user's explicit selection (not data growing over the system's life) and runs
entirely in memory; not a hot/unbounded path. No perf failure.
