## Spec vs plan vs implementation
Source spec `docs/missing.md §11` read in full; plan `docs/tickets/M11.plan.md`
read in full. The plan faithfully covers the spec (subscript surface, foldLookup
reuse, loud miss/ambiguity, per-brain Desc source-of-truth, hidden names with no
special-casing, cross-brain out of scope) and the implementation matches the plan
point for point. No dropped, narrowed, or contradicted requirement.

Verified seams:
- **Mapping-shadows-Indexable (central seam).** Confirmed against the actually
  vendored starlark (`go.starlark.net@v0.0.0-20260613233743`, `eval.go:681-708`):
  `getIndex` switches `Mapping` before `Indexable`, so both int and string
  subscripts route to `Get`. `Indexable` is correctly NOT implemented; the int
  branch lives inside `Get`. A `(nil,false,nil)` from `Get` yields starlark's
  `key %v not in %s` — the loud miss. (Plan cites an older starlark hash for line
  refs; behavior is byte-identical in the vendored version, immaterial.)
- **Returned-handle parity.** `Get` builds `&ArrayElement{ls, kind, entryName,
  spec, row, index}` with the identical field set the iterator uses
  (`subcollection.go:228`), so `.delete()` stale-guard and M5 nav work.
- **Ordinal correctness.** `buildNodeByDescMap` stores `byName[Desc]=ordinal`
  (array position in `subRowIdx[table][entryName]`); `Get` indexes `c.rows()`
  (same `subRowsFor` source, same array order, `index=int64(i)`). Ordinals align;
  `nodes["Accelerate"].index == 0` confirmed by test.
- **node_desc field.** Generated column for `BrainNodeRow.Desc` is `node_desc`
  (`normalize_metadata.go:285`); read via `elementAttrs["node_desc"].fieldIndex`,
  no hardcoded field name.
- **Scope gate.** String key gated on `c.spec.attr == "nodes"`; non-nodes string
  key is a loud error (gate is data-independent, correct for synapses/stomach).
- **First-wins / empty-Desc skip / nil-byName miss** all present per plan.
