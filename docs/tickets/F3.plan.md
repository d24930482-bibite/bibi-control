# F3 — Cross-world species REMAP on whole-entry graft

Tier: high — subtle-invariant + data-shape (per-world LINEAR speciesID; remap must
allocate a fresh non-colliding id, import a species RECORD, and rewrite refs on BOTH
the bibite and any egg without conflating the dest's coincidental same-id species).
Fan-out: F2 (1 dependent) consumes this through the workspace surface, so a silent
conflation bug propagates into the capstone. exec=opus, review=opus.

Perf: none. The remap runs once per `AppendEntry` (one graft), entirely in-memory over
the already-parsed dest archive. The scans it adds (max over `activeSpeciesList`,
`recordedSpecies[*].speciesID`, dest entity species ids; one linear search of source
`recordedSpecies` for the matching record) are O(species + entries) per graft, not per
row/query, and there is no DB/IPC round-trip. No Perf-review flag.

## Goal

`transfer.AppendEntry` of a species-bearing bibite/egg must SUCCEED with a correct
cross-world species remap instead of F1's loud refusal. "Done" means: grafting an entity
whose `genes.speciesID` is **absent** from the dest, or **present as a DIFFERENT
biological species** (the colliding-linear-id case F1 refuses), produces a destination
where (1) the grafted entity carries a FRESH dest species id that collides with nothing
in the dest, (2) the source species RECORD has been imported into dest
`speciesData.json#recordedSpecies` under that fresh id, (3) the fresh id is in
`activeSpeciesList`, (4) the dest's original same-id species (its records, its members)
is byte-for-byte untouched, and (5) BOTH a grafted bibite and a grafted egg get their
`genes.speciesID` rewritten. The species-less graft and all non-species identity guards
(body.id collision, dangling child refs) keep their existing F1 behavior. The change
must stay inside `savemutator/thebibites` (only `saveparser/thebibites` imported).

## Approach

All mutation is staged through the **dst Session** (never by writing `entry.JSON`
directly), so it rides the existing `Apply()` atomicity: a failure anywhere leaves 0
applied bytes. The whole remap is a pure read of the in-memory parsed archives plus
`StageAppend`/`StageSet` ops; it does not commit or advance a head (that is F2).

### 1. Allocate a fresh, non-colliding dest species id

New helper `(t *transfer) freshDstSpeciesID() (int64, error)` reading `t.dst.Archive()`:
- Read `speciesData.json` (`SpeciesEntryName`); if absent/undecoded, fail loudly (we
  cannot import a record without a species table — same condition F1 named).
- Compute `maxUsed` = the max over ALL of:
  - `activeSpeciesList` (reuse `activeSpeciesIndexOf`'s read pattern — iterate the list),
  - every `recordedSpecies[*].speciesID` (linear scan of the records array),
  - every dest bibite/egg entry's `genes.speciesID` (reuse the existing scan shape in
    `dstEntityHasSpecies` — generalize it to return the set/max so the conflation guard
    and the allocator share one traversal).
- Read `nextSpeciesID` (the engine's monotonic counter) if present.
- `fresh = max(nextSpeciesID, maxUsed+1)`. Using `max(...)` (not just `nextSpeciesID`)
  is the invariant guard: a save whose `nextSpeciesID` is stale/behind an in-use id must
  still get a non-colliding allocation. This is the load-bearing correctness step.

### 2. Locate + import the source species record

New helper `sourceSpeciesRecord(srcArchive, sid) (any, bool)`:
- Read source `speciesData.json#recordedSpecies` (linear scan, match
  `recordedSpecies[i].speciesID == sid` via `jsonNumberToInt64`). Return a `cloneJSON`
  of the matching record map (deep copy — never alias source bytes into dst).
- Record-absent policy: if the source has a `genes.speciesID` but NO matching record
  (source species table empty/missing the id), FAIL LOUD naming the id. Rationale from
  the ticket: importing the genome/metadata is the point of the remap; silently adding a
  fresh `activeSpeciesList` id with no backing record would leave the dest with an active
  species the engine has no genome for. Refusing here is the safe, named failure (it does
  not regress F1 — F1 refused this case too).

Import staging (new helper `(t *transfer) stageSpeciesImport(record any, freshID int64)`):
- Rewrite the cloned record's `speciesID` to `freshID` (`setJSONPath(record,
  "speciesID", freshID, SetOptions{})`).
- `parentID`: the source record's `parentID` points into the SOURCE linear id space and
  has no dest counterpart. Set it to a self/sentinel value (set `parentID` to `freshID`,
  i.e. treat the import as its own lineage root) so it is NOT a dangling cross-world
  reference. Document this explicitly as the deliberate lineage-root choice — do NOT
  carry the source parentID through, and do NOT attempt to remap an ancestry chain
  (out of scope; cross-reference reconciliation beyond identity/species is excluded by
  the workspace_plan non-goals).
- Stage `Append(SpeciesTarget(), "recordedSpecies", record)` via `t.dst.StageAppend`.
- Stage `Append(SpeciesTarget(), "activeSpeciesList", freshID)` via `t.dst.StageAppend`.
  Both target the same `speciesData.json` entry; `entryUpdate` coalesces them onto one
  cloned working value during `Apply`, so order within the batch is safe (the append
  resolvers re-read the live array each time).
- Optionally bump `nextSpeciesID` to `freshID+1` when the field exists (keeps the
  engine's counter ahead of what we allocated). Use `StageSet` with the existing field
  only (no `CreateMissing`) — a save without the field is left as-is.

### 3. Rewrite the grafted entity's species id

In `AppendEntry`, after `cloned := cloneJSON(element.JSON)` and the identity guard:
- `sid, ok := entitySpeciesID(cloned)`. If `!ok` (species-less) -> skip the whole remap,
  keep the existing clean-graft path (unchanged behavior).
- Else: `fresh, err := t.freshDstSpeciesID()`; `record, ok := sourceSpeciesRecord(...)`
  (fail loud if absent per policy); rewrite `setJSONPath(cloned, "genes.speciesID",
  fresh, SetOptions{})` so the staged payload carries the new id (this covers BOTH
  bibite and egg — both read/write `genes.speciesID`, confirmed by `entitySpeciesID`
  and the EggRow/BibiteRow sqlref tag); then `t.stageSpeciesImport(record, fresh)`.
- THEN allocate `nextEntryName` and `t.dst.StageAppendBibite(payload{JSON: cloned})` as
  today. The species-table appends and the entry append are all staged before any
  `Apply`, so a mid-remap error (e.g. missing source record) returns before
  `StageAppendBibite`, leaving the dst with 0 staged ops — preserving the F1 invariant
  that a rejected graft never half-mutates.

Ordering note: stage the species import BEFORE the entry append, but it does not matter
for `Apply` (set/append ops on distinct entries are independent; `orderElementDeletes`
only reorders deletes). Keep the body.id + child-ref guard FIRST so a non-species
rejection still leaves nothing staged.

### 4. (Optional) identical-species dedup — IMPLEMENT only if cheap & exact

Optional and explicitly lower priority. Only do it with an EXACT identity key: if a dest
`recordedSpecies[*]` is deep-equal (via `jsonValuesEqual`/a structural compare of the
cloned source record with its `speciesID`/`parentID` fields excluded) to the source
record being imported, reuse that dest record's `speciesID` for the rewrite and SKIP the
record append + activeSpeciesList append (but still ensure the id is active). If no clean
exact key is available, ALWAYS allocate fresh — duplicating a record is safe, conflation
is not. If implemented, gate it behind a named helper and a dedicated test; if skipped,
leave a one-line comment pointing back to this section. Do NOT use a fuzzy/lossy match.

## Tests

Run: `go test ./savemutator/thebibites/ -run 'Transfer|Species|NextEntryName' -count=1`
then the full `go test ./savemutator/... -count=1` to prove the shared-fixture change
did not regress the delete-cascade / append tests.

Required cases (in `transfer_test.go`):

1. **Colliding-linear-id REMAP (headline, was the F1 refusal).** Retag source bibite_0
   (body 500) to `genes.speciesID = 2` where dest species 2 is the UNRELATED pair
   bibite_1/bibite_2. `AppendEntry` must SUCCEED. After Commit assert:
   - grafted `bibites/bibite_3.bb8` exists; its `genes.speciesID` is the FRESH id
     (not 2, not in the pre-graft `[1,2,3]`).
   - `activeSpeciesList` == `[1,2,3,<fresh>]`.
   - a new `recordedSpecies` entry exists with `speciesID == <fresh>` and template.genes
     matching the SOURCE species-2 record (proves the record was imported, not the dest's
     species 2 reused).
   - the dest's ORIGINAL species-2 record (template.genes / names) is unchanged, and
     bibite_1/bibite_2 still carry `genes.speciesID == 2` (no conflation).
2. **Absent-id REMAP.** Source bibite_0 carries species 7 (absent from dest `[1,2,3]`).
   `AppendEntry` succeeds; grafted entity gets a fresh id, source record for 7 imported,
   `activeSpeciesList` gains the fresh id, dest records 1/2/3 untouched.
3. **Egg remap rewrites genes.speciesID on the egg.** Graft source `eggs/egg_0.bb8`
   (species 8); assert the appended egg entry's `genes.speciesID` is the fresh id and the
   source-8 record was imported. (Proves the rewrite covers eggs, not only bibites.)
4. **Allocator dodges a stale `nextSpeciesID`.** Build a dest whose `nextSpeciesID` is
   LOWER than an in-use id (e.g. activeSpeciesList contains 3 but nextSpeciesID is 1);
   assert the fresh id is `> max in-use`, never colliding. Locks in the `max(...)` guard.
5. **Source record absent -> loud, dst unstaged.** Source carries `genes.speciesID` but
   its `recordedSpecies` has no matching record; `AppendEntry` fails loudly naming the id
   and leaves `len(dst.StagedOperations()) == 0` / `State()==clean` (reuse
   `assertDestUnstaged`, but note its `[1,2,3]` activeSpeciesList assertion still holds
   for a refused graft).
6. **Species-less still grafts cleanly, table untouched.** Keep the existing sub-test
   (activeSpeciesList stays `[1,2,3]`, no record added).
7. **No species table -> loud.** Dest with no `speciesData.json` and a species-bearing
   graft fails loudly (the import has nowhere to land). Mirrors the F1 named refusal.
8. (If dedup implemented) **Byte-identical source species reuses the dest id**, no new
   `recordedSpecies` row, fresh id NOT allocated.

Update the existing `TestTransferAppendEntryRefusesSpecies` doc + sub-tests: the
"absent" and "colliding" sub-tests now assert SUCCESS+remap (or move to new test funcs

## Risks / seams

- **The conflation invariant is the whole ticket.** The fresh id must beat the max over
  activeSpeciesList AND recordedSpecies ids AND every live dest entity's species id —
  reusing only `nextSpeciesID` (which can be stale) or only `activeSpeciesList` (which
  can omit a record-only id) would reintroduce the exact F1 silent-conflation bug. Test
  case 4 exists to catch this.
- **Both bibite AND egg.** The rewrite is on `genes.speciesID`, shared by both kinds;
  test case 3 guards the egg path. Do not special-case bibites.
- **Atomic-before-staging.** The remap must compute the fresh id and locate the source
  record (failing loudly if absent) BEFORE any `StageAppendBibite`/`StageAppend`, so a
  rejected graft leaves 0 staged ops (the F1 invariant the refusal tests assert). Stage
  the species-table appends and the entry append together; never write `entry.JSON`
  directly (that bypasses `Apply` atomicity).
- **parentID is cross-world and must NOT dangle.** The source record's `parentID` points
  into the source id space. Reset it (to the fresh id = self-root) on import. Carrying it
  through unremapped is a latent dangling reference; remapping an ancestry chain is out of
  scope.
- **Shared fixture churn.** `parseSpeciesArchive` is used by the delete-cascade tests in
  `append_delete_test.go`. Only ADD `recordedSpecies` records and `nextSpeciesID`; keep
  ids `[1,2,3]` and the member bibites/egg so `TestStageDeleteReconcilesActiveSpecies`
  and friends still pass.
- **Number typing.** Ids read from parsed JSON are `json.Number`; compare via
  `jsonNumberToInt64` and write the fresh id as `int64` so `setJSONPath` →
  `normalizeJSONValue` stores a value the round-trip re-parse reads back as the same id.
- **Scope fence.** Stay in `savemutator/thebibites`; the only allowed import is
  `saveparser/thebibites` (alias `tb`). No DB, no revision store, no head advance, no DSL.
