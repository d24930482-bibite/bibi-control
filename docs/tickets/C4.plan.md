Source: `docs/workspace_plan.md#c4` (the "Track C — C4 — Query …" section, lines 334-350),
read against the locked **dual-key DuckDB mirror** decision (lines 43-56), the
broadest-scope rule "a whole-workspace query spans every world in the one workspace's
DuckDB file" (lines 72-74), and the read-only-enforcement risk (lines 348-350, 524-526).
C4 depends on **C1 only** (`deps: C1`; DAG line 136 `C1 --> C4`) — it builds on C1's
retained per-revision history partitions and does **not** consume C2's loaded working set
(C2 is a sibling, not a dependency).

Tier: **medium — shared per-workspace DuckDB contract + read-only-enforcement invariant
(a stray non-SELECT corrupts sibling worlds' partitions) + E1/F1 fan out from C4's query
API; but it is additive (a new file of read-only methods), reuses existing helpers
(`scanRowsToDicts`-shape conversion, `QuoteIdent`, `RevisionsForWorld`), and the
correctness property is directly testable.** exec=sonnet, review=opus.

## Goal

"Done" = `workspace/` grows three **read-only** query Go methods on `*Workspace`, layered
on C1's dual-key mirror without touching `world.go`:

1. **`HistoryQuery(ctx, worldID, sql) ([]map[string]any, error)`** — runs a caller SELECT
   over the **retained per-revision history partitions** of one world (the partitions C1
   keyed by revision `sha256`), scoped so only that world's revisions are visible, with
   each result row attributable to its revision (sha256 / sim_time / tier). Enables
   time-series across a world's revisions. Excludes the mutable working partition.
2. **`Query(ctx, sql) ([]map[string]any, error)`** — the **broadest scope**: runs a caller
   SELECT over **all** history partitions in this one workspace's DuckDB file (every world,
   no world filter), with revision→world/sim_time/tier attribution available via a JOIN.
   No multi-workspace federation.
3. **Read-only enforcement** — both methods (and a shared validator) **reject any non-SELECT
   statement** before it touches the shared read-write DuckDB handle, because a stray
   `UPDATE`/`DELETE`/`INSERT`/`ATTACH`/`PRAGMA`/`COPY` in raw SQL would corrupt sibling
   worlds' partitions and diverge the mirror from the blob source-of-truth.

Plus a **`mirror_saves` catalog** materialized **inside the workspace DuckDB** (a native
DuckDB table, not a SQLite ATTACH — see "Catalog: DuckDB-native, not ATTACH" below) that
maps `save_id (= revision sha256) → world_id, tier, blob_present`, refreshed from the
SQLite registry so history/workspace queries can `JOIN mirror_saves` for world/tier
attribution. C4 ships **only** these Go methods + the catalog; the Starlark
`world.history_query` / `workspace.query` bindings are **E1**, and the loaded-working-copy
`world.query` path is **C2** — C4 does not implement either.

## Approach

### Catalog: DuckDB-native table, not a SQLite ATTACH (deviation from the spec, justified)
workspace_plan.md lines 53-56 sketch ATTACH-ing a SQLite `mirror_saves` catalog into the
workspace DuckDB. **Do not do the cross-engine ATTACH.** Reasons, concrete to this repo:
- There is **no `mirror_saves` SQLite table** — A1/C1 deliberately deferred it (C1 plan
  lines 44-45: "Do not add a `mirror_saves` table — it is a projection, materialized
  later"). It would be a SQL `SELECT sha256 AS save_id, world_id, tier, blob_present FROM
  save_revisions`, i.e. a pure projection of `save_revisions` — exactly the data
  `RevisionsForWorld`/`ListWorlds` already return in Go.
- The workspace already holds **one read-write DuckDB handle** (`w.duckDB`,
  workspace.go:39). DuckDB takes a process/file lock; attaching the SQLite file via the
  `sqlite_scanner` extension needs network/extension install and a second engine in-proc.
  Materializing a small native DuckDB table from the Go-side registry rows is simpler,
  has no extension dependency, and keeps "SQLite = source of truth" intact (the table is a
  rebuilt cache, never authored in DuckDB).

So: **`refreshMirrorCatalog(ctx)`** (called at the top of `HistoryQuery`/`Query`, under
`w.mu` because it writes the shared DuckDB handle) rebuilds a DuckDB table
`mirror_saves(save_id TEXT, world_id TEXT, tier TEXT, blob_present BOOLEAN, sim_time
DOUBLE)`:
1. `mirrorCatalogDDL` = `CREATE TABLE IF NOT EXISTS mirror_saves (...)`.
2. `DELETE FROM mirror_saves` then bulk-insert one row per revision across all the
   workspace's worlds: `for _, world := range w.store().ListWorlds(ctx, w.ID())` → `for _,
   rev := range w.store().RevisionsForWorld(ctx, world.ID)` insert `(rev.SHA256,
   rev.WorldID, rev.Tier, rev.BlobPresent, …)`. Wrap in one DuckDB `BEGIN/COMMIT`
   (mirror the `ReplaceExtractedSave` transaction shape, import.go:97-122) so the catalog
   is never half-rebuilt.
   - **sim_time per revision**: do **not** read it from SQLite (`save_revisions` has no
     sim_time column — only `worlds.sim_time` for the head; schema.sql:33/40-55). The
     per-revision sim_time is already in each history partition's mirror as
     `scenes.simulated_time` keyed by `save_id` (= sha256; migration line 30-35). So
     populate `mirror_saves.sim_time` from a sub-select
     `(SELECT simulated_time FROM scenes WHERE scenes.save_id = ? AND has_simulated_time)`
     **or** simply omit `sim_time` from `mirror_saves` and let history queries
     `JOIN scenes` themselves. **Prefer: include `sim_time` in `mirror_saves`** filled from
     the world's denormalized head value only when the revision IS the head, else NULL —
     no, cleaner: **fill it via the `scenes` sub-select per sha256** so every revision (not
     just the head) carries its own sim_time. Document the chosen source in a comment.
   - Refreshing on every query is cheap (revision counts are small: one row per commit).
     It also guarantees the catalog reflects tier flips from a concurrent G2 eviction.

### `ensureReadOnly(query string) error` — the enforcement gate
A lightweight, literal-aware leading-keyword check (reuse the tokenizer discipline already
in `script/thebibites/sql.go:validateExprSafety` lines 594-651 as the *model*, but C4 owns
its own copy in `query.go` — do not import the unexported sql.go helper):
- Trim leading whitespace and SQL comments; the first significant token must be a SELECT
  (or `WITH` … that resolves to a SELECT — accept a leading `WITH`).
- Reject a statement terminator `;` that is followed by any further non-comment,
  non-whitespace token (a second statement) — a single SELECT may end in a trailing `;`.
- Reject if the leading keyword is any of `INSERT UPDATE DELETE CREATE DROP ALTER ATTACH
  DETACH COPY PRAGMA INSTALL LOAD SET CALL EXPORT IMPORT BEGIN COMMIT ROLLBACK VACUUM
  CHECKPOINT TRUNCATE REPLACE` (case-insensitive). Return the typed
  `ErrReadOnlyQuery` (a package sentinel, `errors.New("workspace: only read-only SELECT
  queries are allowed")`) wrapped with the offending keyword so E1 surfaces a clean
  Starlark error.
- This is the **primary** gate. (A DuckDB `access_mode=read_only` second handle is NOT
  used: DuckDB single-file locking forbids a concurrent RW + RO handle on the same file,
  and the workspace already owns the RW handle — workspace.go:39, openDuck:221. Statement
  rejection is the correct enforcement for a shared RW handle. Note this trade-off
  explicitly in a comment so a reviewer does not "fix" it into a second handle.)

### `HistoryQuery(ctx, worldID, query string) ([]map[string]any, error)`
1. Validate `worldID != ""`; `ensureReadOnly(query)`.
2. `w.GetWorld(ctx, worldID)` to confirm the world exists (clean not-found error;
   `revisionstore.IsNotFound`).
3. `w.mu.Lock()` for the catalog refresh write, refresh, then `Unlock()` **before** running
   the SELECT (the SELECT is a read; holding the writer mutex across it would needlessly
   serialize reads against C2/D2 mutators). Acceptable that the catalog can be rebuilt by a
   racing query between Unlock and the SELECT — it is idempotent and the SELECT only reads
   it.
4. Run the caller's SELECT **as a CTE-scoped query** so the world filter is enforced
   structurally rather than trusting the caller: wrap the user query so it can only see
   this world's history partitions. Concretely: expose the allowed `save_id`s via the
   catalog and require the caller to `JOIN mirror_saves` — **but C4's contract is the
   caller writes raw SQL against the 46 tables**, so enforce the world scope by giving the
   query a pre-filtered catalog view. Implement: before running, also build (or filter) a
   session-scoped `mirror_saves` so that `HistoryQuery` runs the user SQL with a
   **prepended CTE** `WITH world_saves AS (SELECT save_id FROM mirror_saves WHERE world_id
   = ?)` available, and document that history queries `JOIN world_saves USING (save_id)` to
   scope. (Simpler acceptable v1: run the user SELECT as-is over the full DuckDB but pass
   `worldID` and the catalog so the *test* asserts the user can scope; the structural CTE
   is the preferred shape — pick the CTE prepend, it is one string concat and keeps scope
   honest.) Pass `worldID` as the bound `?` arg.
5. `rows, err := w.duck().QueryContext(ctx, wrapped, worldID)`; `defer rows.Close()`;
   `return scanRowsToMaps(rows)`.

### `Query(ctx, query string) ([]map[string]any, error)`
Same shape, no world filter: validate `ensureReadOnly`, refresh catalog under `w.mu`,
`w.duck().QueryContext(ctx, query)`, `scanRowsToMaps`. The whole workspace = every history
partition in this one DuckDB file. The caller `JOIN mirror_saves` for world/sim_time/tier
attribution.

### `scanRowsToMaps(rows *sql.Rows) ([]map[string]any, error)`
A Go-side analogue of `script/thebibites/sql.go:scanRowsToDicts` (lines 80-115), but
returning `[]map[string]any` (column name → driver scalar) rather than Starlark values —
C4 is the **Go API**; E1 converts to Starlark dicts later via the existing
`fromSQLValue`/`scanRowsToDicts` path. Preserve column order is not required for a map
return; if order matters to a test, also return `[]string` columns — **keep it
`[]map[string]any`** for v1 and let E1 own Starlark conversion. Scan into
`[]any`/`[]*any`, `rows.Columns()`, `rows.Err()` checked.

## Risks / seams

- **Read-only enforcement is the headline correctness property (workspace_plan.md lines
  348-350, 524-526).** The shared per-workspace DuckDB is read-write (one writer handle).
  A non-SELECT slipping through `ensureReadOnly` corrupts **sibling worlds'** history
  partitions and silently diverges the mirror from the blob source-of-truth (harmless in
  the old in-memory-per-run model, dangerous now). The chained-statement and
  comment-leading cases (`/* */ DELETE`, `SELECT 1; DROP …`) are the easy-to-miss
  bypasses — the test table must cover them. **Do not** "improve" this into a second
  `access_mode=read_only` DuckDB handle: DuckDB single-file locking forbids a concurrent
  RW+RO handle on the same file (the workspace already owns the RW handle), so a second
  handle would fail to open. Statement rejection on the RW handle is the intended design.
- **Dual-key discipline (top-3 plan risk, lines 510-512).** History queries must read the
  **sha256-keyed** partitions (immutable, one per revision — the whole point of "full
  history"), never the **world-id-keyed** working partition (mutable head-only). The
  `mirror_saves` catalog keys on `rev.SHA256` (the history key) precisely so a
  `JOIN mirror_saves` selects only history partitions; a query that filters on
  `save_id = worldID` would silently read the working partition and miss all but the head.
  The `world_saves` CTE (`save_id` from `mirror_saves WHERE world_id = ?`) keeps callers on
  the history keys. Tests assert the history sha256s, not the world id, appear in results.
- **Catalog freshness vs tier flips (G2 interaction).** `mirror_saves.tier`/`blob_present`
  are a cache of `save_revisions`. Refreshing on **every** query (rebuild-from-registry)
  keeps it correct after a concurrent G2 eviction flips a revision to `mirror_only` — do
  not lazily build it once. The refresh is the only **write** C4 does to DuckDB, so it (and
  only it) takes `w.mu`; release the mutex before the read SELECT so C4 reads do not
  serialize against C2/D2 mutators.
- **sim_time has no SQLite column.** `save_revisions` carries no `sim_time` (schema.sql);
  only `worlds.sim_time` (head). Per-revision history sim_time MUST come from the mirror's
  `scenes.simulated_time` (keyed by sha256, migration line 34). Reading it from SQLite
  would give the head's value for every revision — wrong for a time-series. The catalog
  builder sources it from `scenes`.
- **C4 deps = C1 only, NOT C2.** Do not call `OpenWorld`/`Load`/the working-set
  `LoadedSave` (C2's surface) — they may not exist when C4 executes. The "working copy
  (current)" bullet in the spec (lines 335-337) is explicitly "Unchanged" and belongs to
  C2's `world.query`; C4 ships only history + whole-workspace + read-only. Returning
  `[]map[string]any` (not Starlark) keeps C4 a pure Go API; E1 owns the Starlark
  `history_query`/`query` bindings and the loaded-working-copy `world.query`.
- **File contention.** C4 is a brand-new `workspace/query.go`; it adds no `Workspace`
  struct fields and edits no existing file. Confirm C2 lands its Load/Unload in a new file
  (not `world.go`) and D2 stays in `node.go`; with that, C4 needs **no execution
  serialization** against C2/D2 — only the trivial struct-method-set merge that all
  workspace tickets share.
