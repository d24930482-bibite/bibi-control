# C3 — Commit advances head + re-import (dual-key)

Source: `docs/workspace_plan.md#c3` (Track C — C3, lines 318-333), read against the locked
**dual-key DuckDB mirror** decision (lines 43-52), the head/parent lineage-atomicity invariant
(lines 519, 67), the churn DoD (lines 548-549), and the C1/C2 plans this builds directly on
(`docs/tickets/C1.plan.md`, `docs/tickets/C2.plan.md`). Depends on **B3** (the `save` global is
gone; `open()` returns a `Save` object — `bindings.go:9`/`save_value.go`) and **C2**
(`Load`/`OpenWorld`/`Unload` working-set lifecycle — `workspace/working_set.go`). Reuses A2/G1's
`RecordRevisionAdvancingHead` (`revisionstore/store.go:285`), which already inserts the revision,
advances head, **and** self-refs the blob in one SQLite tx (`insertRevisionTx` →
`incBlobRefTx`, store.go:337/380) — so C3 must **not** call `IncBlobRef` separately (same as C1,
world.go:127-139).

Tier: **high — exec=opus, review=opus.** Signals (round UP on conflict): (1) **persistence +
monotonic state** — writes a NEW revision and advances `worlds.head_revision_id`, the world's
single source of truth; a wrong parent or a head left pointing at a missing/old revision
silently corrupts lineage. (2) **head/parent atomicity** — must thread `parent = current head`
and advance head as one unit (the recorder gives the SQLite tx; C3 must read the *current* head
*inside* the workspace lock and pass it as parent — a TOCTOU here under the single-writer lock
is the subtle bug). (3) **dual-key DuckDB discipline** (workspace_plan.md top risk, lines
510-512) — the new revision imports under `save_id = newSha` via `ReplaceExtractedSave`
(additive, history accumulates) while the working partition re-seeds under `save_id = worldID`;
confusing the two corrupts history (mutating a hash partition) or loses it (replacing history on
commit). (4) **subtle invariant a passing test can miss** — history *retention* (prior
revisions' partitions must survive a commit) and refcount-per-hash are properties a naive test
won't catch. (5) **fan-out ≥2** — D3 (reload/ingest) and E1/E2 (automation `s.commit()`) both
build on this commit path (deps: workspace_plan.md lines 139/141/144). Every one of these is a
high-risk signal; this is the hardest code on the W-stream critical path.

## Goal

"Done" = the `workspace` package grows a **`CommitWorld(ctx, worldID, program []byte, opts
thebibites.RunOptions) (thebibites.WorldCommit, error)`** method that, against the world's
already-loaded working copy (lazy-loaded via `OpenWorld` if absent):

1. runs the save program (the existing staged-write pipeline — `s.bibites.where().set()`,
   `s.commit()` are unchanged; only how the save is obtained — `open()` on the loaded `ls` —
   changed in B3),
2. when the run staged a mutation, opted into commit, and is not dry-run: produces a
   content-addressed blob (one `WriteArchive`, zero reparses) and records a new revision that
   **threads `parent = the world's current head`** and **advances `worlds.head_revision_id`
   (+ `sim_time`) atomically** via `RecordRevisionAdvancingHead`,
3. updates **both** mirror partitions in the shared per-workspace DuckDB:
   - **history import (retained):** import the new revision under `save_id = newSha` with
     `ReplaceExtractedSave` — additive, so prior revisions' history partitions are left intact
     (history accumulates, one partition per revision sha256),
   - **working re-seed:** refresh `save_id = worldID` rows to the new head so the open world's
     push-down/mirror keep reflecting the head,
4. all under the workspace mutex (single DuckDB writer), and returns the produced revision id +
   sha + staged-op count.

C3 implements **only** the world-commit path. No query (C4), no node ops (D*), no eviction (G*).
A pure-mutation commit performs exactly one `WriteArchive`, zero reparses (one only under
`opts.Verify`), one history import, and one working re-seed — the churn DoD
(workspace_plan.md lines 548-549).

### Decision: do NOT add fields to `script.Result` (deviation from source line 333, justified)
The source sketch says "add `RevisionID int64` + `RevisionSHA string` to `script.Result`".
**Skip that** — `script.Result` (`script/result.go:4`) is a domain-neutral, cross-package

reads `Revision.ID`/`Revision.SHA256` from there. This keeps the standalone path's

## Approach

### `script/thebibites/commit_world.go`

Imports: `context`, `fmt` (+ the existing pattern from run.go), and the in-package types
(`script`, `revisionstore`, `blobstore`, `tb "saveparser/thebibites"`). `time`/`crypto/sha256`
via the existing `sha256Sum` helper in run.go (same package — reuse it; do not redeclare).

```go
// WorldCommit is the result of running a save program against a loaded world copy
// and committing an advancing-head revision. Committed is false when the run was
// pure-analysis / dry-run / opted-out (no revision produced); Revision/History
// are then zero and the caller skips the DuckDB re-import.
type WorldCommit struct {
    Result    script.Result
    Committed bool
    Revision  revisionstore.Revision   // the advancing-head revision (zero when !Committed)
    Applied   *tb.Archive              // ls.session.Archive() after Apply (for ExtractTables)
    SaveID    string                   // ls.saveID == worldID (the working-partition key)
}
```

`func RunAndCommitWorld(ctx context.Context, ls *LoadedSave, program []byte, worldID string,
parentID *int64, blobs blobstore.Store, revs *revisionstore.Store, opts RunOptions)
(WorldCommit, error)`:

1. Guard `revs != nil`, `blobs != nil`, `ls != nil`, `worldID != ""`; default `ctx`. Set
   `ls.dryRun = opts.DryRun` (mirror run.go:56).
2. Run the program against the **already-loaded** `ls`:
   `res, runErr := script.Run(ctx, program, Globals(ls), script.Options{Filename:
   opts.Filename, MaxExecutionSteps: opts.MaxExecutionSteps})`. `Globals(ls)` exposes `open()`
   returning the `Save` bound to *this* `ls` (bindings.go:23-40) — the staged mutations land on
   `ls.session`, the same one `prepareCommit` serializes. **Do not re-`Load`** (that would
   stage onto a throwaway save — the exact bug bindings.go:27-32 warns about).
3. Compute `status`/`errMsg`/`recordedDryRun`/`willWrite` exactly as run.go:66-76
   (`willWrite = runErr == nil && !opts.DryRun && ls.willCommit && ls.stagedOps > 0`).
4. If `willWrite`: `ref, commitErr := ls.prepareCommit(ctx, blobs, opts.Verify)`
   (loadedsave.go:592 — applies staged ops idempotently, serializes via `WriteArchiveTo` with
   NO reparse, `blobs.Put`, optional verify). On `commitErr` set `status="commit_failed"` (mirror
   run.go:90-100).
5. `RecordScriptRun` (store.go:201) with the computed status/staged-ops/dry-run — BEFORE the
   revision (the FK `save_revisions.script_run_id -> script_runs.id`; same ordering as
   run.go:102-111 and world.go:108).
6. Carry `res.StagedOps`/`res.DryRun` (run.go:113-114). Surface `recErr`/`commitErr` with the
   same precedence run.go:116-125 uses.
7. **If committed** (`willWrite && commitErr == nil`): derive `simTime` from the **applied**
   archive: `applied := ls.session.Archive()` (session.go:45 — non-nil after `prepareCommit`'s
   `ensureApplied`); `var simTime *float64; if applied.Scene != nil && applied.Scene.HasTime {
   st := applied.Scene.SimulatedTime; simTime = &st }` (same derivation as world.go:121-125;
   `archive.go` `SceneState.SimulatedTime`/`HasTime`).
8. **Record the advancing-head revision:**
   `rev, err := revs.RecordRevisionAdvancingHead(ctx, worldID, simTime,
   revisionstore.RevisionInput{ ParentID: parentID, WorldID: worldID, SourcePath: ls.path,
   BlobRef: ref, ScriptRunID: run.ID })` (store.go:285). This inserts the revision, advances
   head+sim_time, and self-refs the blob in **one** SQLite tx — do NOT also call `IncBlobRef`.
   `parentID` is the **caller-supplied current head** (the workspace passes it; see below) —
   this is the parent-threading the source requires (workspace_plan.md line 318/519). On a
   first-ever commit with a head already set by C1, `parentID` is non-nil.
9. Return `WorldCommit{Result: res, Committed: true, Revision: rev, Applied: applied, SaveID:
   ls.saveID}`. When not committed, return `WorldCommit{Result: res, Committed: false}` with the
   appropriate error (run-level error still surfaced).

> **Why return `Applied` + `SaveID` instead of importing here:** the DuckDB re-import needs
> `w.duck()` and must run under the **workspace** mutex (single writer) — a `script/thebibites`
> function has neither. So C3 splits cleanly: the script package owns run+blob+revision
> (the churn-bounded `WriteArchive`), the workspace package owns the dual-key DuckDB import
> under its lock. `ls.session.Archive()` is the post-Apply archive `prepareCommit` already
> serialized, so `ExtractTables(key, applied)` in the workspace is a pure transform of the
> exact committed bytes — **zero extra reparse** (churn DoD preserved).

### `workspace/commit.go`

Imports: `context`, `fmt`, the `duckdb`, `revisionstore`, `tb "saveparser/thebibites"`, and
`script/thebibites` packages (all already imported elsewhere in the package).

`func (w *Workspace) CommitWorld(ctx context.Context, worldID string, program []byte, opts
thebibites.RunOptions) (revisionstore.Revision, error)`:

1. Nil/empty guards (`w == nil`, `worldID == ""`, `len(program) == 0`).
2. **Resolve the loaded working copy WITHOUT holding w.mu yet.** Call
   `ls, err := w.OpenWorld(ctx, worldID)` (working_set.go:146) — it lazy-Loads under its own
   lock if absent. **Do NOT call `OpenWorld` while holding `w.mu`** (non-reentrant mutex →
   self-deadlock; same caution as working_set.go:162). Capturing the `*LoadedSave` pointer here
   is safe: it is stable in `w.worlds` and the commit serializes on `w.mu` next.
3. **Take `w.mu` for the whole commit body** — `w.mu.Lock(); defer w.mu.Unlock()` — because
   the DuckDB re-import is the single-writer path (workspace_plan.md line 521) and the
   read-current-head → record-advancing-head sequence must be atomic w.r.t. other mutators on
   this workspace.
4. **Read the world's CURRENT head as the parent (inside the lock).**
   `world, err := w.store().GetWorld(ctx, worldID)` (store.go:596); if `world.HeadRevisionID ==
   nil` return an error (`"workspace: world %q has no head to commit onto"` — a loadable world
   always has a head from C1, so this is a defensive guard, not a normal path).
   `parentID := world.HeadRevisionID` (pass the `*int64` straight through as the parent).
   **This read MUST be under `w.mu`** so no concurrent commit advances the head between the read
   and the record — the TOCTOU guard the high tier is about.
5. **Run + commit via the script core:**
   `wc, err := thebibites.RunAndCommitWorld(ctx, ls, program, worldID, parentID, w.blobs(),
   w.store(), opts)`. On error return it (it already carries the truthful run/commit status).
6. **If `!wc.Committed`** (pure-analysis / dry-run / opt-out / staged 0): return
   `(revisionstore.Revision{}, nil)` — no head moved, no re-import. (Mirror the willWrite gate
   semantics; a no-op commit is not an error.)
7. **Dual-key DuckDB re-import** (the head advanced; `wc.Revision` is the new head):
   - **History partition (retained):** `hist := tb.ExtractTables(wc.Revision.SHA256,
     wc.Applied); duckdb.ReplaceExtractedSave(ctx, w.duck(), hist)` — `ReplaceExtractedSave`
     (import.go:92) deletes-by-`save_id` then inserts; the `save_id` is the **new** sha256, so
     the delete is a no-op and the insert adds a new history partition **without disturbing any
     prior revision's partition** (history accumulates — the immutability invariant,
     workspace_plan.md lines 45-52/320-325). Use `ReplaceExtractedSave` (not
     `ImportExtractedSave`) so migrations are not re-run (they ran on Open, workspace.go:226) —
     same choice C1 made for the history partition (world.go:155-158).
   - **Working partition (re-seed to head):** `working := tb.ExtractTables(worldID,
     wc.Applied); duckdb.ReplaceExtractedSave(ctx, w.duck(), working)` — `save_id = worldID`,
     so the delete drops the previous head's working rows and the insert seeds the new head.
     This is the only place the working partition is replaced; history is never touched by the
     worldID key. (`ls.saveID == worldID == wc.SaveID` — assert `wc.SaveID == worldID` and fail
     loudly if they ever diverge; the load-bearing dual-key invariant.)
8. Return `wc.Revision, nil`.

**Failure handling:** if the DuckDB import fails *after* the revision recorded (step 7), return
the wrapped error — the registry/blob are the durable source of truth and the working partition
can be re-seeded by a later `Load` (C2 re-imports via `LoadInto`'s injected handle... note: it
does NOT re-import; see Risks — so a failed working re-seed leaves the working partition at the
PRIOR head until the next successful commit/Load path re-seeds it; document this as a known,
non-corrupting drift, never a data loss, since the blob+revision are durable). Do **not** attempt
to roll back the SQLite head advance (it is already committed); the mirror is a rebuildable
projection. This matches C1's "registry first, DuckDB last, no destructive cleanup" stance
(world.go:73-79).

### Ordering / atomicity seam (the reviewer's checklist)
- `w.mu` held across: read current head → run program (prepareCommit + record advancing-head) →
  dual-key import. Single writer; no interleaving commit can steal the parent.
- The SQLite head advance is atomic inside `RecordRevisionAdvancingHead` (insert + SetWorldHead
  in one tx, store.go:303-318). C3 adds the **outer** atomicity (read-head-then-record under
  `w.mu`) so the parent is the true current head.
- `OpenWorld` is called BEFORE taking `w.mu` (it takes its own lock). Never nest `w.mu`.

## Risks / seams

- **Dual-key discipline (top risk — workspace_plan.md lines 510-512).** History imports under
  the new **revision sha256**; the working partition re-seeds under the **world id**. They are
  never the same value and never overwrite each other. `ExtractTables(key, archive)` stamps
  `key` into every row's `save_id` (normalize.go:9-21), so the partition key is exactly the
  first arg — extract once per key from `wc.Applied`. `TestCommitWorldHistoryRetained` +
  `TestCommitWorldWorkingPartitionReflectsHead` pin both halves.
- **Parent = current head, read UNDER w.mu (TOCTOU).** The parent must be the world's head *at
  commit time*. Read `GetWorld(...).HeadRevisionID` inside `w.mu`, immediately before
  `RecordRevisionAdvancingHead`, so no concurrent commit advances the head in between. Passing a
  stale parent (e.g. one captured before the lock, or from the `LoadedSave` rather than the
  registry) breaks lineage. The recorder's own tx guarantees insert+advance atomicity; C3 owns
  the read-then-record atomicity.
- **Do NOT double-refcount.** `RecordRevisionAdvancingHead` → `insertRevisionTx` →
  `incBlobRefTx` already self-refs the blob in the same tx (store.go:337-383). C3 must NOT call
  `IncBlobRef` — that would double-count and break G2/G3 eviction. (C1 made the same call,
  world.go:127-130: "no separate IncBlobRef here".)
- **Run against the ALREADY-LOADED `ls`, never a fresh Load.** The program's `open()` must
  return the `Save` bound to the working copy whose `ls.session` `prepareCommit` will serialize.
  `Globals(ls)`'s `openBuiltin` returns `&Save{ls: ls}` for *this* `ls` (bindings.go:23-40); a
  re-`Load` inside the run would stage onto a throwaway and the commit would serialize the
  unmutated copy. This is the exact pitfall bindings.go:27-32 documents.
- **`prepareCommit` does the single WriteArchive, in-memory, no reparse** (loadedsave.go:600-605
  `WriteArchiveTo(&buf, …)`). C3 re-extracts from `ls.session.Archive()` (the same in-memory
  applied archive) for the DuckDB import — a pure transform, **not** a parse — so the churn DoD
  (1 WriteArchive, 0 reparses) holds. Do not write the blob to a temp file and re-parse it for
  the import.
- **Single DuckDB writer.** Both `ReplaceExtractedSave` calls run under `w.mu`. Take `w.mu` only
  in `CommitWorld`'s body; `OpenWorld` (called before) takes its own lock — never nest `w.mu`
  (non-reentrant; working_set.go:162, node.go StartNode).
- **Working re-seed uses `ReplaceExtractedSave`, NOT `ImportExtractedSave`.** Migrations already
  ran on Open; re-running them per commit is wasted work (and C1 deliberately avoids it for the
  history key, world.go:151-158). `ReplaceExtractedSave`'s delete-by-`save_id` on the worldID
  key is exactly the "drop old head rows, seed new head" the working partition needs.
- **Mirror re-seed after a failed import is rebuildable, never data loss.** If a
  `ReplaceExtractedSave` fails after the revision recorded, the head + blob are durable in
  SQLite/blobstore; the working partition may lag at the prior head until the next commit
  re-seeds it. Note: C2's `Load`/`OpenWorld` does **NOT** re-import the working partition (the
  injected handle short-circuits the in-memory import — C2 plan Risks), so a stale working
