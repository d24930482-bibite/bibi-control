# M3 — Case-insensitive name lookups (gene / world / settings-scope / column)

Source: `docs/missing.md` §3. Downstream contract dependent: **M4** (deps: M3) which
states `.get`/`[]` "adopt whatever §3 lands on, applied uniformly." This plan therefore
defines the boundary contract crisply; M4 inherits it verbatim.

## Goal

Every user-typed **name** lookup in the DSL resolves case-insensitively: the user's
argument and the candidate canonical key are both lowercased before comparison, so
`b.genes["diet"]` finds the gene `Diet`, `workspace.world("M")` finds the world `m`, and
`save.settings.material("plant")` finds material `Plant`. Stored canonical case is never
mutated (display/serialization still round-trips the original). Names that differ only by
case resolve **deterministically**: a case-insensitive match that hits more than one
distinct canonical key errors **loudly** (mirrors the existing ambiguous-world handling),
never silently picks one. "Done" = the gene / world / settings-scope name boundaries
fold case + handle collisions loudly through **one shared helper**, the `gene("X")` vs
`genes["X"]` miss asymmetry is documented (not changed — see Scope), and tests prove
case-hit, exact-hit, miss, and case-collision on each surface.

## The contract (what M4/§1/§4 inherit)

1. **Fold at the lookup boundary only.** Lowercase the user arg *and* the candidate key
   for comparison (`strings.ToLower`). Never write the lowercased form back into any
   `GeneRow.GeneName`, `World.Name`, `SettingValueRow.SettingName`, or `OwnerID`.
2. **Deterministic collision = loud error.** When ≥2 distinct canonical keys fold to the
   same lowercase as the query, the lookup returns an *ambiguity error* naming the
   colliding canonical keys; it must never return one arbitrarily. (Same policy as
   `worldByName`'s existing `len(match) > 1` branch.)
3. **Exact case still wins cheaply.** If the query equals a canonical key exactly, that is
   an unambiguous single match and never triggers the collision path (so a save with
   genuine `m`/`M` worlds is still addressable by exact id-or-name as today).
4. **Miss behavior is unchanged by M3** (the `gene` vs `genes` asymmetry):
   - `b.genes["X"]` (mapping `Get`) → not-found ⇒ Starlark `KeyError` (loud). Stays loud.
   - `b.gene("X")` (point builtin) → not-found ⇒ `None` (silent). Stays silent.
   - M3 makes the *match* case-insensitive on both; it does **not** flip either miss
     policy. The asymmetry is resolved later by M4 deleting `gene("X")` entirely
     (tolerant read becomes `genes.get("X")`). M3 documents the split as deliberate
     (a one-line comment update at each site) and stops there.

## Files to change

    (new, near the `geneSet`/`indexSettings` code) that implements the contract:
    exact hit → return; else scan keys, lowercase-compare, collect distinct matches;
    0 → `found=false`; 1 → return it; ≥2 → ambiguity `error` listing the canonical keys.

## Approach (steps, reusing existing utilities)

1. Write `foldLookup` in `loadedsave.go` next to `indexGenes`/`indexSettings`. Reuse
   `strings.ToLower` (already imported pattern in `sql.go:893,1140`). Exact-key fast path
   first, then a single pass collecting distinct case-insensitive matches.
2. Thread it through the three `byName`-keyed sites (gene Get/SetKey, geneBuiltin) and the
   settings `settingRow` (setting-name level + material owner-id level).
3. Update `worldByName` to a case-insensitive scan; leave its 0/1/many switch.
4. Update the three doc comments that assert exact-match / None-vs-KeyError so they state
   the case-insensitive + loud-collision contract (the comments are load-bearing here —
   they are the contract M4 reads).
5. **Column scope (deliberately bounded):** the friendly *column* surface
   (`attr_registry.go`, `resolveColumn` in `sql.go:241`) is **generated metadata** — every
   key is already a lowercase snake_case DuckDB column or a lowercase alias (`position_x`,
   `diet`). §3 lists "column" for completeness, but case-folding the registry-keyed path
   touches every `.where`/`group_by`/aggregate/`Attr` dispatch for ~zero real-world payoff
   (no mixed-case column keys exist) and risks the spanning push-down. **This plan folds
   column lookups via the same boundary discipline but only at the single
   `attrRegistry()[kind][col]` access in `resolveColumn`**: lowercase `col` before the map
   read, since registry keys are already lowercase (so no collision is possible and no
   registry rebuild is needed). This satisfies "no partial rollout reintroduces the
   surprise" at the one entry point without disturbing the generated registry. Do **not**
   case-fold the `Attr(name)` dispatch in `entity.go` (Starlark attribute access is a
   language-level identifier, not a user-name lookup, and Python attrs are case-sensitive).

## Risks / seams

- **Signature ripple on `settingRow`.** Adding an `error` return changes its one caller
  (`SettingScope.Get`, settings_value.go:107). Update both; no other caller (verified by
  grep — only settings_value.go uses it). Keep the `(*row, found, err)` shape so an
  ordinary miss is still `found=false`, distinct from an ambiguity `err`.
- **Collision must be order-independent.** `foldLookup` iterates a Go map (random order);
  the result must NOT depend on iteration order — that is exactly why ambiguity is a loud
  error rather than "first match wins." A happy-path test passes even if you accidentally
  pick-first, so `TestGeneLookupCaseCollisionLoud` is the load-bearing test: it must fail
  if the helper returns a value instead of an error on collision.
- **Don't fold the fixed scope owners.** Simulation/independent owner ids are sentinels
  (`"settings"`/`"independents"`), not user names — folding them is harmless but pointless;
  only the material owner name is user-typed. Keep the sentinels exact to avoid masking a
  future real owner-id table value.
- **Don't mutate canonical case anywhere.** `String()`/`AttrNames`/serialization and the
  mirror `SQLValueRef` (`gene.go:152-159`, keyed by `row.EntryName`/`row.Path`/`OwnerID`)
  must keep using the *stored* canonical strings; `foldLookup` returns the backing index,
  so the canonical row is reached unchanged — do not pass a lowercased name into any
  `SQLValueRef`/mirror locator.
- **Exact-match precedence** (rule 3) protects the legitimate `m`/`M`-coexist case: keep
  the exact fast path before the fold scan in both `foldLookup` and `worldByName`.

## Tier

Tier: medium — shared contract consumed by M4 (1 downstream contract dependent) + a
subtle case-collision invariant a happy-path test misses (pick-first passes). Small,
pattern-following change (worldByName already shows the loud-ambiguity reference), so
`--model-exec sonnet`; the collision invariant + downstream contract need a strong
verifier, so `--model-review opus`. No perf signal (per-entity name indexes are built
once and candidate sets are tiny; no hot path / unbounded scan / N+1).
