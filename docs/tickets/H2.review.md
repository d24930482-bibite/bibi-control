1027 bibites), `-benchtime=1x -count=5 -benchmem`. BASE measured in a fresh detached
worktree at 9bde3b6 with an equivalent probe; branch via `BenchmarkParseFileLargest`.

| metric            | BASE (main)          | branch (H2)         | delta            |
|-------------------|----------------------|---------------------|------------------|
| wall-time / parse | ~1.61–2.04 s         | ~0.69–0.77 s        | ~2.5–2.8x faster |
| bytes allocated   | 1.957 GB             | 755 MB              | ~2.6x less       |
| allocs / op       | 8.00 M               | 7.39 M              | -0.6 M           |

This matches the executor's claims (2.5x wall-time; 1.95GB→755MB) and clears the
"materially lower" bar. The win is real, not a b.N/GC artifact — I used the cold
single-parse probe the herder asked for, not the GC-bound b.N loop. Drivers: the
GOMAXPROCS-bounded worker pool over 12 cores (the ~70% structural win), goccy's faster
decode-to-any, plus the allocation cuts (shared entity scalar slices, presized
apply-target slices, byte-buffer path accumulator, stack-backed key buffer).

## 2. Determinism + race (the correctness gate)

- `go test -race -count=1 ./saveparser/...` — CLEAN. Re-ran from scratch.
- `TestParseFileDeterministicAcrossRuns` — PASS under -race (113s). Trims runs 20→4 under
  -race via the `raceEnabled` build tag, but each of the 4 runs still parses the full
  1027-entry fixture (real concurrency for the detector) AND asserts byte-identity two
  ways every iteration: `reflect.DeepEqual(got, want)` over the whole `*Archive` and
  `bytes.Equal` over `WriteArchiveTo` serialization. The build-tag trick trims only the
  iteration COUNT, never the comparison — not neutered.
- `TestParseFileParallelMatchesSequential` — PASS under -race (37s). Forces GOMAXPROCS=1
  for a true single-worker reference and compares the 12-core parallel parse against it
  (DeepEqual + WriteArchiveTo bytes). This is the strict parallel-vs-sequential pin the
  ticket required. Skips only if GOMAXPROCS<2 (this box has 12, so it ran).

Design audit confirms determinism by construction: workers fill independent `results[i]`
slots and only their own `&a.Entries[i]` (index claimed via `atomic.AddInt64(&next,1)`
starting at -1 → each index owned by exactly one worker); all order-sensitive mutation is
confined to the single-threaded `applyParseResult` loop over `results[0..n-1]` in index
order. `applyParseResult` is never called from a worker.

## 3. Aliasing / escape (the "beyond-plan perf changes" scrutiny)

- **Shared entity scalar slices** (`parse_payload.go`: `result.Scalars = result.Bibite.Scalars`
  etc. instead of copy-append): SAFE. Each entity's `.Scalars` is assigned exactly once
  via `collectScalars` and is never appended-to afterward (child scalars live in separate
  fields — `GeneScalars`/`BodyScalars`/brain-node `.Scalars`). `applyParseResult` copies
  by value via `append(a.Scalars, result.Scalars...)` into `a.Scalars`'s own backing, so
  the shared backing is read-only after parse. The sole retained reference is the stored
  entity's own `.Scalars`; normalize/ExtractTables consume it read-only (no in-place
  append into entity scalar slices — verified). `ParseEntryBytes` shares the same way and
  is also single-shot/read-only.
- **Byte-buffer path accumulator** (`json_scalars.go` `collectScalarsInto`): SAFE. `path`
  is passed by value; each leaf materializes `string(path)` (a copy) — no pointer into the
  reused buffer is retained in any stored Scalar. Parent restores `path[:base]` on its own
  copy each iteration; child reallocations don't corrupt siblings. Produces the same paths
  the old `fmt.Sprintf` code did (confirmed by schema-fingerprint + scalar tests).
- **Stack-backed key buffer** (`keyBuf [16]string`): SAFE. Used only to sort+iterate keys
  locally; no pointer escapes. Heap fallback for >16 keys.
- **presizeFromResults / growCap**: capacity-only. `growCap` keeps `len` unchanged, only
  grows cap; `presizeFromResults` counts exact contributions (one Bibite/Egg + len of
  Pheromones/Scalars/Diagnostics per result). Cannot affect content or append order.
- **scalarCount presize**: exact leaf count (every non-map/array value emits exactly one
  Scalar); at worst an undercount would cost a realloc, never wrong content.

## 4. goccy Marshal / decoder byte-identity (the decoder seam)

Ran a direct probe (goccy v0.10.5 from this module's go.sum vs encoding/json):
- `json.Marshal` for scalars: byte-identical incl. HTML-special chars (`<`,`>`,`&` →
  `<`/`>`/`&`), ` `/` `, escapes, unicode, emoji — 0 mismatches.
- Numbers under `UseNumber()`: `Number.String()` and `Marshal` of the decoded value are
  byte-identical across ints, big ints (1e20), exponents (`1e10`/`1E10` case preserved),
  trailing zeros (`1.0`), `-0.0`, high-precision floats — 0 mismatches. So `Scalar.RawJSON`
  for numbers (built from `v.String()`) is unchanged.
- Decode still uses `UseNumber()` + the strict double-Decode/`io.EOF` trailing check + the
  unchanged `utf8BOM` strip. `json.Number` is consistently goccy's type across
  `json_helpers.go` and `json_scalars.go`, so the `case json.Number:` switches match the
  decoder's output type (no silent default-branch routing).
- Backstop: the checked-in `save_schema_fingerprint.golden` (walks raw JSON of every
  fixture leaf) is UNCHANGED in the diff and `TestSaveSchemaFingerprint` passes; write
  round-trip, scalar-redundancy, and normalize fixture tests pass. RawJSON bytes did not
  shift, so fingerprints/StateSum/stored raw are stable.
