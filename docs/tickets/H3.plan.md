# H3 — Plan: derive the DuckDB history partition via SQL instead of a 2nd Go extract+append

## Goal

`AddWorld`/commit/ingest each import a save into the shared per-workspace DuckDB
**twice**: once `ExtractTables(world.ID, …) → ImportExtractedSave` for the WORKING
partition (keyed by the world id), then a *second* `ExtractTables(ref.SHA256, …) →
ReplaceExtractedSave` for the HISTORY partition (keyed by the revision sha256). The
two row sets are byte-identical except the `save_id` column, so JSON→rows
normalization and the DuckDB Appender both run twice for the same data
(~0.9s each → ~1.8s, the dominant import cost; commit tests 14-20s).

"Done" means: the Go-side extract + Appender import runs **once** (the working
partition), and the history partition is **derived inside DuckDB** by a set-based,
per-table `INSERT … SELECT … <hist_sha AS save_id> … WHERE save_id = <working_key>`
that rewrites only `save_id`. The history rows must be **byte-identical** to today's
double-extract output, differing only in `save_id`. All dual-key catalog,
`history_query`, lazy-load, and per-world scoping behavior is unchanged. The second
extract+import is gone on every import path.

## Risk / tier

Tier: **high** — subtle invariant (history-partition byte-equality across 44
normalized tables; the dual-key mirror is the correctness crux) + shared contract (a
new exported `duckdb` import primitive consumed by 3 workspace callsites) + fan-out
(H3 unblocks **H5** which rewrites the remaining append, and speeds **F2**'s
transfer-commit path). Exec opus / review opus (already set).

Perf-review: flag **SET**. This ticket *is* a hot-path perf change. Concern +
location: `AddWorld`/`applyWorldCommit`/`appendIngestedRevision` currently run the
JSON→rows extract + DuckDB Appender **twice per import** on the import hot path
(per world add, per commit, per autosave ingest); H3 must eliminate the second
extract+import. The reviewer's perf lens must (a) MEASURE the extract+import portion
before/after on the 3.1MB fixture and assert the 2nd extract+import is gone, and
(b) confirm the new history copy is a single set-based pass per table (one
`INSERT … SELECT` round-trip per table, NOT row-by-row and NOT per-revision).

## Established facts (verified in code; do not relitigate)

- **3 double-import callsites**, all in package `workspace`, all sharing the exact
  `ExtractTables(workingKey) → import` then `ExtractTables(histSha) → import` shape:
  - `workspace/world.go:147-158` — `importWorldFromArchive` (AddWorld / AddWorldBytes).
  - `workspace/commit.go:209-222` — `applyWorldCommit` (shared by `CommitWorld` and
    `CommitWorldLoaded`; the latter is F2's transfer-commit path).
  - `workspace/node_reload.go:370-378` — `appendIngestedRevision` (autosave ingest /
    LazyLoad).
- Every one of the **44** normalized tables (`tb.NormalizedTables`, exposed in
  `duckdb` as `allTables = normalizedTableNames()`) has `save_id` as a column
  (`grep` confirms 44/44; the migration declares 44 `CREATE TABLE`s, matching).
  `bibite_mutation_refs` is a **VIEW** over physical tables (migration line 575) — it
  needs NO copy, it follows automatically. `json_scalars` is a physical normalized
  table and IS covered by the per-table copy.
- `ReplaceExtractedSave` (import.go:92) = one DuckDB tx: `deleteSaveRows(save_id)`
  over `allTables` + `insertExtractedSave` (Appender per table). `deleteSaveRows`
  for a never-seen `save_id` is a no-op, so history imports compose additively
  (the immutability invariant).
- The mirror catalog (`workspace/query.go`) keys `mirror_saves` by `save_id` =
  **revision sha256** only (history keys); `refreshMirrorCatalog` reads
  `scenes WHERE has_simulated_time` keyed by `save_id` to source per-revision
  `sim_time`. So the derived history `scenes` rows MUST carry the sha256 `save_id`
  exactly as today, or `HistoryQuery` sim_time and the `world_saves` CTE break.

## Approach

### 1. New `duckdb` primitive: `CopySavePartition` (one DB-side set-based copy)

Add to `duckdb/import.go` an exported function that, in one transaction, replaces a
destination partition with a `save_id`-rewritten copy of a source partition:

```
// CopySavePartition replaces all rows for dstSaveID with a copy of every row for
// srcSaveID across every normalized table, rewriting only the save_id column. It
// is the DB-side equivalent of ExtractTables(dstSaveID, …)+ReplaceExtractedSave
// over a save whose other columns are already materialized under srcSaveID.
func CopySavePartition(ctx context.Context, db *sql.DB, srcSaveID, dstSaveID string) error
```

Implementation, mirroring `ReplaceExtractedSave`'s transaction shape
(import.go:97-124 — Conn, `BEGIN`, deferred `ROLLBACK` unless committed, `COMMIT`):

- Open a `conn`, `BEGIN TRANSACTION`.
- `deleteSaveRows(ctx, conn, dstSaveID)` — reuse the existing helper
  (import.go:130) so the dst partition is cleared first (no-op for a never-seen
  sha256; the additive-history invariant is preserved). This guarantees re-derive
  idempotency identical to `ReplaceExtractedSave`.
- For each `table` in `allTables` (the generated list — do NOT hand-maintain a
  parallel table list; see the *sqlref-generation-philosophy* memory):
  - Build the **column list generically** from `tb.NormalizedTables[i].Fields`
    (`.Column`), quoting each with `QuoteIdent`. This is the same metadata
    `insertExtractedSave` already drives off, so the columns stay in lockstep with
    the schema automatically.
  - Emit one statement per table:
    `INSERT INTO "<t>" (<all cols>) SELECT <all cols, but ? AS save_id> FROM "<t>" WHERE save_id = ?`
    where the `save_id` column in the SELECT list is replaced by the bound
    `dstSaveID` parameter and the WHERE binds `srcSaveID`. Concretely the SELECT
    projects each column by name **except** `save_id`, which is projected as the
    `dstSaveID` bind — preserving column order so the positional `INSERT (cols…)`
    matches.
  - `conn.ExecContext(ctx, stmt, dstSaveID, srcSaveID)`.
- `COMMIT`, set `committed = true`.

Helper: a small `copySaveSelectList(fields []tb.NormalizedFieldSpec) (insertCols,
selectExprs string)` that walks the fields once, quoting columns and substituting
the `save_id` column with `?` in the select expression list. Keep it in `import.go`
next to `fieldColumns`/`QuoteIdent`. Because `save_id` is always present as a column
(verified 44/44) and is the working key in the source rows, the rewrite is exact.

Why `INSERT … SELECT` (not "stamp two save_ids before a single append"): it is the
smallest, most auditable change, it keeps ALL row materialization in the existing
single Appender pass, and it makes byte-equality trivially provable (the DB copies
the columns verbatim; only `save_id` changes). It also avoids touching
`ExtractTables`/`ReplaceExtractedSave`, so every other caller is untouched.

### 2. Rewire the 3 workspace callsites

In each, keep the single working import exactly as today, then replace the second
`ExtractTables(histSha,…) → ReplaceExtractedSave(…)` with
`duckdb.CopySavePartition(ctx, w.duck(), <workingKey>, <histSha>)`:

- **world.go `importWorldFromArchive`** (147-158): keep
  `working := tb.ExtractTables(world.ID, archive)` + `ImportExtractedSave` (migrations
  must still run here — first import). Replace the `hist` block with
  `CopySavePartition(ctx, w.duck(), world.ID, ref.SHA256)`.
- **commit.go `applyWorldCommit`** (199-227): the order today is history-first then
  working. Reorder so the **working** import happens first
  (`ExtractTables(worldID, projected) → ReplaceExtractedSave`), then
  `CopySavePartition(ctx, w.duck(), worldID, wc.Revision.SHA256)` derives history
  from it. Preserve the two distinct error messages / non-rollback semantics
  (the mirror is a rebuildable projection; head+blob already durable). NOTE the
  comment at 184-198 about `reparseCommitted` stays valid — we still extract the
  fresh projection once for the working partition; history is now DB-derived from
  those working rows, so it inherits the same fresh projection.
- **node_reload.go `appendIngestedRevision`** (370-378): same reorder — working
  import first (`ExtractTables(worldID, archive) → ReplaceExtractedSave`), then
  `CopySavePartition(ctx, w.duck(), worldID, ref.SHA256)`.

All three already hold `w.mu` (single DuckDB writer) around this body, so the
working-import-then-copy sequence stays atomic w.r.t. other mutators. No lock,
partition-key, or catalog-schema change.

### 3. Do NOT change

- partition KEYS (working = world id, history = revision sha256),
- the `mirror_saves` catalog schema or `refreshMirrorCatalog` logic,
- `ExtractTables` / `ReplaceExtractedSave` / `ImportExtractedSave` signatures or
  behavior (other callers depend on them),
- `tb.NormalizedTables` / the generated metadata (derive the table+column list from
  it; never fork it).

## Risks / seams (the executor must not break)

1. **History byte-equality (the crux).** The derived history rows must differ from
   the working rows ONLY in `save_id`. The `INSERT … SELECT` must project columns
   **in schema order** with `save_id` substituted in place, not appended — a
   misordered column list silently shuffles values. Test A + B prove this; keep the
   column list generated from `tb.NormalizedTables[i].Fields` so it cannot drift.
2. **`save_id` is the source key, not data.** The source rows are keyed by the
   working partition (`world.ID` for add/ingest; `worldID` for commit). The SELECT
   reads `WHERE save_id = <workingKey>` and writes `<histSha> AS save_id`. Do NOT
   read from the sha256 key (it does not exist yet) — order matters: working import
   FIRST, then copy.
3. **Commit/ingest reorder.** Today history is imported before working; the copy
   approach requires working FIRST (it is the copy source). Preserve each
   callsite's distinct error wording and the "mirror is rebuildable, never rolled
   back against the committed head" semantics (commit.go:212-226).
4. **Additive history immutability.** `CopySavePartition` must `deleteSaveRows`
   the dst (histSha) before inserting — a no-op for a never-seen sha (additive) but
   correct on re-derive — exactly matching `ReplaceExtractedSave`'s delete-then-
   insert, so a prior revision's partition is never disturbed.
5. **Catalog dependence on scenes save_id.** `refreshMirrorCatalog`/`readSceneSimTimes`
   read `scenes` by sha256 `save_id`; the derived `scenes` history rows must carry
   the sha256 `save_id` (Test B covers `scenes`). Do not skip optional/empty tables
   in a way that diverges from `ExtractTables` (an empty source table copies zero
   rows — same as today's `ExtractTables` emitting an empty slice).
6. **Generated-metadata discipline** (*sqlref-generation-philosophy* memory): the
   table list and per-table column list both come from `tb.NormalizedTables` /
   `duckdb.allTables`. No hand-maintained parallel list anywhere.
7. **Single-writer atomicity.** All 3 callsites already hold `w.mu`; do not add a
   second lock or move the copy outside the locked body.
