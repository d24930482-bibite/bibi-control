# A2 — Revisionstore methods + parent threading

Source: `docs/workspace_plan.md#a2` (the "Track A — A2 — Store methods + parent threading"
section, lines 230-248), plus the file-contention note at lines 195-196: **fold G1's refcount
methods into A2** so G1 becomes "wire up call sites" only.

## Goal

"Done" = `revisionstore/store.go` (the package A1 gave us its schema) grows the Go API that the
workspace layer (B2/C-track) and tiering layer (G-track) call against the A1 schema, all
**purely additive** (new structs/methods alongside the existing `RecordScriptRun`/
`RecordRevision`/`RevisionByID`; existing signatures and behavior unchanged). Concretely A2
ships:

1. **Registry CRUD** — `CreateWorkspace`/`GetWorkspace`/`ListWorkspaces`;
   `CreateWorld`/`GetWorld`/`ListWorlds`/`SetWorldHead`/`RevisionsForWorld`;
   `CreateNode`/`GetNode`/`ListNodes`/`BindNode`/`SetNodeStatus`.
2. **Parent threading reconciliation** — extend `RevisionInput` with `WorldID string` (reuse the
   existing `ParentID *int64`), persist `world_id` in `RecordRevision`, and add
   `RecordRevisionAdvancingHead` that inserts a revision **and** advances the world's
   `head_revision_id` + `sim_time` **in one SQLite transaction** (fixing the always-nil parent at
   `script/thebibites/loadedsave.go:582`: the workspace passes the world's current head as
   `ParentID`, then head advances to the new revision atomically).
3. **Blob lifecycle / refcount (folded-in G1)** — `IncBlobRef`/`DecBlobRef(sha256)`,
   `EvictRevisionBlob(ctx, revisionID)`, `UnreferencedBlobs()`, `PromoteRevision(ctx, revisionID)`.
   A2 only mutates the **SQLite catalog** (`tier`/`blob_present`/`refcount`); it does **not** touch
   blobstore bytes or DuckDB (G2 deletes bytes; the workspace owns DuckDB). The blobstore `Store`
   interface (`blobstore/blobstore.go:20`) has no `Delete`, by design — A2 must not add one.

A2 is **revisionstore-only**: do **not** edit `script/thebibites/loadedsave.go` (that's B1/C3's
job to call the new recorder), do **not** create the `workspace/` package (B2), do **not** create
a `mirror_saves` table (it is a projection materialized later — A1 plan and workspace_plan.md
lines 53-56/223-226 forbid a physical table). uuid is already a **direct** dependency
(`go.mod:9`, promoted by A1), so id allocation is unblocked.

## Approach

### 0. New Go types (mirror the existing struct/Input pattern, store.go:26-70)

```go
type WorkspaceInput struct { Owner, Name string; CreatedAt time.Time }
type Workspace      struct { ID, Owner, Name string; CreatedAt time.Time }

type WorldInput struct { WorkspaceID, Name string; CreatedAt time.Time }
type World       struct {
    ID, WorkspaceID, Name string
    HeadRevisionID *int64    // NULL until first revision recorded
    SimTime        *float64  // NULL until head set
    CreatedAt      time.Time
}

type NodeInput struct { WorkspaceID, WorldID, NodeID, RunID, Status, CompatAddr, DropPath string; CreatedAt time.Time }
type Node      struct { ID, WorkspaceID string; WorldID string /* "" when NULL */; NodeID, RunID, Status, CompatAddr, DropPath string; CreatedAt time.Time }
```
- IDs are allocated **inside** `CreateWorkspace`/`CreateWorld`/`CreateNode` with
  `uuid.NewString()` (the workspace_plan locks ids as opaque UUIDs; A1's schema test already
  inserts uuid-allocated ids, store_test.go:353/359). Return the stored row (re-`Get` by id, like
  `RecordScriptRun` returns via `ScriptRunByID`, store.go:167).
- `WorldID string` on `Node`: map SQL `NULL` ↔ Go `""` via `sql.NullString` on scan and a helper
  that passes `sql.NullString{Valid:false}` when the field is empty on insert. Same for
  `nodes.world_id` in `BindNode`.

### 1. Extend `RevisionInput` / `Revision` and persist `world_id`

- Add `WorldID string` to `RevisionInput` (store.go:51) and `WorldID string` +
  `Tier string`, `BlobPresent bool`, `Refcount int64`, `MirrorSchemaVersion *int64` to `Revision`
  (store.go:60). **Keep `ParentID *int64`** — it already exists and is plumbed through
  `normalizeRevision` (store.go:355) and scanned (store.go:461); A2 just stops leaving it nil at
  the *workspace* callsite (done in `RecordRevisionAdvancingHead`, not here).
- In `RecordRevision` (store.go:188) add `world_id` to the INSERT column list and bind it via
  `sql.NullString` (empty `WorldID` ⇒ NULL; a non-empty one must reference an existing `worlds.id`
  or the FK `save_revisions.world_id → worlds(id)` rejects it — that's the desired guard). Do
  **not** insert `tier`/`blob_present`/`refcount`/`mirror_schema_version` explicitly — let the
  schema defaults apply (`'full'`/`1`/`0`/`NULL`), exactly as A1's schema test verifies
  (store_test.go:375-394).
- Extend the **two SELECT statements** that read `save_revisions` (`RevisionByID` store.go:246-250
  and `RevisionsBySHA256` store.go:267-272) to also project
  `world_id, tier, blob_present, refcount, mirror_schema_version`, and extend `scanRevisionRows`
  (store.go:442) to scan them (`world_id` via `sql.NullString`→`""`, `mirror_schema_version` via
  `sql.NullInt64`→`*int64`, `blob_present` via int→bool). `RevisionsForWorld` reuses the same
  scan. The existing sha256/size cross-check (store.go:469-474) stays.
- `normalizeRevision` (store.go:328): carry `WorldID` onto the returned `Revision`. No new
  validation needed beyond the existing `BlobRef.Validate`/`ScriptRunID>0`/`ParentID>0` checks.

### 2. `RecordRevisionAdvancingHead` — the atomicity centerpiece (plan lines 244-247)

Signature:
```go
func (s *Store) RecordRevisionAdvancingHead(ctx context.Context, worldID string, simTime *float64, input RevisionInput) (Revision, error)
```
- `input.WorldID` is forced to `worldID` (so caller can't desync). `input.ParentID` is whatever
  the caller passes (the world's current `head_revision_id`, read by the workspace before calling,
  or nil for the very first revision — C1).
- **One transaction** via `s.db.BeginTx(ctx, nil)` (there is no existing tx helper in the repo —
  this is the first; mirror the manual `Exec`/`LastInsertId` pattern of `RecordRevision` but on
  the `*sql.Tx`). Inside the tx: (a) INSERT the revision (same columns as `RecordRevision`),
  (b) `UPDATE worlds SET head_revision_id = ?, sim_time = ? WHERE id = ?` with the new revision
  id and `simTime`. `Commit()` on success; `defer tx.Rollback()` (the post-commit rollback is a
  no-op). On any error return it (rollback fires). This satisfies workspace_plan.md line 519
  "Head/parent lineage atomicity — a crash must not leave head pointing at a missing revision."
- After commit, re-read via `RevisionByID` (a fresh query, outside the tx) and return — matches
  the existing "insert then re-select" convention (store.go:216). `sim_time` denormalization is
  the head's `scene.simulated_time`; A2 stores whatever `*float64` the caller passes (the workspace
  derives it — A2 does not parse saves).
- **Reuse, don't duplicate, the column list.** Extract a small `insertRevisionTx(ctx, q sqlExec,
  rev Revision, blobRefJSON string, inlineBlob []byte) (int64, error)` where `sqlExec` is a tiny
  interface `{ ExecContext(...) (sql.Result, error) }` satisfied by both `*sql.DB` and `*sql.Tx`,
  and have both `RecordRevision` and `RecordRevisionAdvancingHead` call it. This keeps the INSERT
  in one place (the dual-key/parent risk is real — one INSERT site is safer).

### 3. World head / lineage queries

- `SetWorldHead(ctx, worldID string, revisionID int64, simTime *float64)` — standalone
  `UPDATE worlds SET head_revision_id = ?, sim_time = ? WHERE id = ?`; error if 0 rows affected
  (unknown world). Used by callers that advance head outside a record (and shared by
  `RecordRevisionAdvancingHead`'s update body if convenient, but inside the tx use the tx handle).
- `RevisionsForWorld(ctx, worldID string) ([]Revision, error)` —
  `SELECT … FROM save_revisions WHERE world_id = ? ORDER BY id` (uses A1's
  `save_revisions_world_id_idx`), scanning via `scanRevisionRows`. This is the lineage/history
  enumerator C4 builds on.

### 4. Refcount + tier methods (folded-in G1, catalog-only)

- `IncBlobRef(ctx, sha256 string) (int64, error)` / `DecBlobRef(...)` — refcount is **per content
  hash**, shared across revisions (workspace_plan.md lines 218/467-471). Implement as
  `UPDATE save_revisions SET refcount = refcount + 1 WHERE sha256 = ?` (Dec uses
  `MAX(refcount-1, 0)` or a `CHECK (refcount >= 0)`-safe `refcount - 1` guarded by a
  `WHERE refcount > 0`; the A1 `CHECK (refcount >= 0)` (schema.sql:53) must never be violated).
  Validate the digest with `validateSHA256` first. Return rows affected so the caller can detect
  an unknown hash. (Per-hash means every revision row sharing that sha256 carries the same count;
  that is the model the plan picks — keep it simple and consistent, increment/decrement all rows
  for the hash in one statement.)
- `EvictRevisionBlob(ctx, revisionID int64) error` (plan lines 235-237) — in one tx:
  1. Load the revision (id, sha256, world_id).
  2. **Refuse** if it is **any** world's head: `SELECT 1 FROM worlds WHERE head_revision_id = ?`
     → if found, return a typed error (e.g. `ErrRevisionIsHead`). (workspace_plan.md line 67/515:
     "A world's head is never evictable.")
  3. **Refuse** if other live refs to the bytes remain after accounting this revision's own ref —
     i.e. if `refcount > 1` (or `> 0` depending on the ref-accounting convention; document the
     chosen accounting). Return a typed error (e.g. `ErrBlobStillReferenced`).
  4. `UPDATE save_revisions SET tier='mirror_only', blob_present=0 WHERE id = ?`. **Do not** delete
     blob bytes (G2) and **do not** delete the row (history is retained — line 483). Commit.
- `UnreferencedBlobs(ctx) ([]string, error)` (plan line 238) — distinct `sha256` with
  `refcount=0` **and** no `full`/`blob_present=1` revision still needing them:
  `SELECT DISTINCT sha256 FROM save_revisions WHERE refcount = 0
   AND sha256 NOT IN (SELECT sha256 FROM save_revisions WHERE blob_present = 1)`. This is the GC
  candidate list G3 consumes; A2 only *reports*, never deletes.
- `PromoteRevision(ctx, revisionID int64) error` (plan line 239) — the `mirror_only → full`
  re-promotion when a blob reappears by hash:
  `UPDATE save_revisions SET tier='full', blob_present=1 WHERE id = ? AND tier='mirror_only'`.
  Idempotent; 0 rows affected on an already-`full` revision is fine (not an error).

### 5. Typed errors

Add package-level sentinel errors so callers (C2/D3/F1/G-track) can branch cleanly and the
Starlark surface (E) maps them to clean errors: `ErrRevisionIsHead`, `ErrBlobStillReferenced`
(both for `EvictRevisionBlob`). These complement the existing `IsNotFound` (store.go:556). Do NOT
add `ErrNotRematerializable` here — that is G4's guardrail at the *workspace/load* layer, not the
store catalog layer.

## Risks / seams

- **Do not regress the standalone commit path.** `script/thebibites/loadedsave.go:577`
  (`recordRevision`) calls `RecordRevision` with **no** `WorldID` and **no** `ParentID`. Adding the
  `WorldID` field must default to `""` ⇒ SQL `NULL` (not an FK violation), and `RecordRevision`'s
  behavior for the no-world case must be byte-identical to today. A2 does **not** change that
  callsite (B1/C3 will, to call `RecordRevisionAdvancingHead`); A2 only makes the recorder
  *available* and keeps the old one working.
- **First-ever transaction in the repo.** There is no existing `BeginTx`/`*sql.Tx` usage
  (confirmed by grep). `Open` sets `db.SetMaxOpenConns(1)` (store.go:85) and
  `?_pragma=foreign_keys(1)` per-connection (store.go:81) — with a single connection a tx holds
  that connection, so any concurrent store call serializes (acceptable; the workspace adds its own
  mutex per workspace_plan.md line 521). Use `defer tx.Rollback()` and check the `Commit` error;
  never leave a tx dangling.
- **FK target resolution / insert ordering.** `save_revisions.world_id → worlds(id)` and
  `worlds.head_revision_id → save_revisions(id)` are a mutual reference (both NULL-able). The
  correct order is: `CreateWorld` (head NULL) → insert first revision with `world_id` set → set
  head. `RecordRevisionAdvancingHead` enforces exactly this order within its tx. A `world_id`
  pointing at a missing world is rejected at insert (desired guard) — tests assert it.
- **Refcount semantics are per-content-hash, not per-revision-row.** Increment/decrement **every**
  row sharing that sha256 in one `UPDATE … WHERE sha256 = ?` so the value stays consistent across
  clones/dedup (workspace_plan.md lines 218/467). Document the eviction ref-accounting convention
  (whether the revision's own ref counts) in a comment so G2 reads the same rule. Never let
  `refcount` go below 0 (A1 `CHECK (refcount >= 0)`, schema.sql:53).
- **A2 is catalog-only for tiering.** It flips `tier`/`blob_present` and reports
  `UnreferencedBlobs`; it does **not** delete blobstore bytes (the `Store` interface has no
  `Delete` — blobstore/blobstore.go:20) and does **not** touch DuckDB. The crash-safe
  "catalog-fsync-before-byte-delete" ordering and `ReconcileBlobs` belong to G2 (workspace_plan.md
  lines 478-482); A2 supplies the catalog half they sequence around.
- **No `mirror_saves` physical table.** It is a projection over `save_revisions` (+ `worlds.sim_time`)
  materialized/ATTACH-ed by the workspace later (workspace_plan.md lines 53-56/223-226). Do not
  create it here. A2's job is the SQLite registry methods only.
- **`sim_time` is a `*float64` the caller supplies.** A2 stores/denormalizes it onto `worlds`;
  it does NOT parse a save to derive `scene.simulated_time` (that's the workspace's job in C1/C3).
  Keep it nullable end-to-end.
