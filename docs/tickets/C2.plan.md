Source: `docs/workspace_plan.md#c2` (Track C — C2, lines 311-316), read against the B1
`LoadInto` seam (lines 254-266), the dual-key mirror decision (lines 43-52), and the C1
AddWorld that already seeded both partitions (the `docs/tickets/C1.plan.md` it builds on).
Depends on C1 (`AddWorld` establishes the world row, head revision, blob, and the
`save_id=worldID` working partition) and B1 (`LoadInto(path, worldID, db)` —
`script/thebibites/loadedsave.go:146`).

Tier: medium — exec=sonnet, review=opus. Signals: manages the in-memory working-set
lifecycle that C3's `CommitWorld` builds directly on (1 direct dependent), and adds real
concurrency surface (map mutation under `w.mu` + a lazy-Load path that must do blob I/O
*outside* the lock, then re-check the map). But it is additive (new methods on an existing
struct, no schema/contract/migration change), imports **no** data into DuckDB (the working
partition already exists from C1; `LoadInto`'s injected handle short-circuits the import),
and every primitive it calls (`GetWorld`, `RevisionByID`, `blobs().Get`, `LoadInto`,
temp-file materialization) already exists and is tested. A cheap executor following this
plan is safe; a strong reviewer catches lock/lazy-Load mistakes.

## Goal

"Done" = the `workspace` package grows three methods on `*Workspace` that manage the
in-memory working set (`w.worlds map[string]*thebibites.LoadedSave`, allocated empty by B2
at `workspace.go:48`):

- `Load(ctx, worldID) (*thebibites.LoadedSave, error)` — materializes the world's **head**
  revision blob to a temp file, builds a `LoadedSave` via `LoadInto(tmp, worldID, w.duck())`
  (so its `saveID == worldID` and it reuses the shared per-workspace DuckDB handle — the
  working partition C1 already seeded under `save_id=worldID`), stashes it in `w.worlds`,
  and returns it. Idempotent-ish: re-loading replaces the stashed handle.
- `Unload(worldID) error` — drops the in-memory `LoadedSave` from `w.worlds`. The registry
  row, DuckDB partitions, and blobs all persist untouched. Unloading an absent world is a
  no-op (no error).
- `OpenWorld(ctx, worldID) (*thebibites.LoadedSave, error)` — returns the working-set
  handle for reads/mutations, lazily calling `Load` if the world is not already loaded.

All three serialize map access on the existing `w.mu` (the single per-workspace lock node.go
already shares). C2 implements **only** the working-set lifecycle — no query (C4), no commit
(C3), no node ops (D*). It does not import anything into DuckDB and does not write any
revision.

## Approach

### Imports for `working_set.go`
`context`, `fmt`, `os`, plus
`"github.com/asemones/bibicontrol/script/thebibites"` (the `LoadedSave` + `LoadInto` package
— note this is **`script/thebibites`**, the scripting wrapper, NOT `saveparser/thebibites`;
`workspace.go:21` already imports it as `thebibites`). No new third-party deps.

### `Load(ctx context.Context, worldID string) (*thebibites.LoadedSave, error)`
1. Nil/empty guards: `w == nil` → error; `worldID == ""` → error
   (`fmt.Errorf("workspace: worldID is required")`).
2. **Acquire `w.mu` only for the bookkeeping; do the blob I/O without holding it the whole
   time** is the safe shape, but C2 may also take the simpler correct path: take `w.mu` for
   the entire body (matching `importWorldFromArchive` at `world.go:81`, which holds the lock
   across DuckDB I/O). **Prefer the simple whole-body lock** for C2 — Load does a temp-file
   write + a `ParseFile` (inside `LoadInto`) but **no DuckDB write** (the injected handle
   short-circuits the import, `loadedsave.go:420-441`), so it does not contend with the
   single-writer rule and holding the lock is correct and race-free. Document the choice in
   a comment. (If a reviewer prefers releasing the lock around I/O, the OpenWorld re-check
   below already shows the pattern; either is acceptable, simple-whole-body preferred.)
3. Resolve the head revision:
   - `world, err := w.store().GetWorld(ctx, worldID)` (`store.go:596`). Wrap errors; map
     `revisionstore.IsNotFound(err)` (`store.go:1344`) to a clear
     `"workspace: world %q not found"`.
   - **Head guard:** `if world.HeadRevisionID == nil { return nil, fmt.Errorf("workspace:
     world %q has no head revision", worldID) }` (`World.HeadRevisionID` is `*int64`, NULL
     until the first revision — `store.go:115-122`). A world a caller can Load always has a
     head (C1 sets it), but a head-less world row is possible if a C1 import partially failed
     (C1 plan "Failure handling"), so fail loudly rather than panicking on the nil deref.
   - `rev, err := w.store().RevisionByID(ctx, *world.HeadRevisionID)` (`store.go:405`).
4. **Blob-present guard (forward-compat with G4).** `if !rev.BlobPresent { return nil,
   fmt.Errorf("workspace: world %q head revision %d is mirror_only (blob evicted): cannot
   load", worldID, rev.ID) }`. C2 ships a plain error here; G4 (which depends on C2,
   workspace_plan.md line 151) replaces it with the typed `ErrNotRematerializable`. Leave a
   `// G4: replace with ErrNotRematerializable` comment. A world's head is never evictable
   (workspace_plan.md line 67), so for any well-formed workspace this is a defensive check,
   but it is the exact blob-dependent path G4 enumerates (workspace_plan.md line 494).
5. **Fetch the head bytes.** `data, err := w.blobs().Get(ctx, rev.BlobRef)`
   (`blobstore/blobstore.go:22`; `rev.BlobRef` is the `blobstore.Ref` populated by
   `scanRevision`, `store.go:1263`). Wrap errors.
6. **Materialize to a temp file** (LoadInto is path-only — there is no in-memory load path;
   mirror the temp-file shape C1 uses in `AddWorldBytes`, `world.go:51-63`, and the verify
   pattern at `loadedsave.go:643-655`): `os.CreateTemp("", "bibiload-*.zip")`, write `data`,
   close, `defer os.Remove(tmpPath)`. Handle write/close errors (close before remove).
7. **Build the LoadedSave.** `ls, err := thebibites.LoadInto(tmpPath, worldID, w.duck())`
   (`loadedsave.go:146`). This (a) re-parses the bytes from the temp file (the one parse on
   the load path — acceptable, this is not the churn-bounded commit path), (b) sets
   `ls.saveID = worldID` so every mirror locator/`whereClause`/`fromClause` scopes to the
   `save_id=worldID` working partition C1 seeded (the invariant C4 reads and C3 re-seeds),
   and (c) injects `w.duck()` so the lazy `openDB` returns the shared handle without
   importing (`loadedsave.go:427-428`). `LoadInto` does **not** import any rows — the
   working partition is already present from C1's `AddWorld` (`world.go:147-150`). The temp
   file is only needed during `LoadInto`'s parse; `defer os.Remove` is safe because the
   `LoadedSave` keeps the parsed archive in memory (`newLoadedSave`, `loadedsave.go:167-178`)
   and never re-reads `ls.path` for reads/mutations (only `WriteSave`/commit does, which is
   C3 — see Risks).
8. **Stash and return.** `w.worlds[worldID] = ls; return ls, nil`. If a handle was already
   loaded under `worldID`, overwrite it (the new handle reflects the current head; drop the
   old one — there is no `Close` to call, `LoadedSave` owns no closeable resource of its own,
   `ls.db` is the caller-owned shared handle that `LoadInto`/B1 documents it never closes,
   `loadedsave.go:145`).

> **Temp-file lifetime caveat (load path only):** because `LoadInto` records `ls.path =
> tmpPath` and C3's `prepareCommit`/`WriteSave` (`loadedsave.go:561,592`) can re-read or
> overwrite `ls.path`, a deferred `os.Remove(tmpPath)` would delete the file C3 later writes
> to *if* C3 wrote to `ls.path`. **C3 does NOT write to `ls.path`** — `WriteSave(path)` takes
> an explicit path and `prepareCommit` serializes from the in-memory session, not from the
> on-disk temp file — so removing the temp file after Load is safe. Confirm this against
> `loadedsave.go:561-623` while implementing; if (and only if) some path re-reads `ls.path`,
> do **not** defer-remove and instead leave the temp file for the workspace to clean on
> Unload/Close. **Default: defer-remove the temp file in Load** (the verified-correct case).

### `Unload(worldID string) error`
1. Nil guard on `w`.
2. `w.mu.Lock(); defer w.mu.Unlock()`.
3. `delete(w.worlds, worldID)` — unconditional; deleting an absent key is a no-op in Go.
   No registry/DuckDB/blob mutation (workspace_plan.md line 315: "registry/DuckDB/blobs
   persist"). Return nil. (Signature returns `error` for symmetry/forward-compat even though
   it cannot currently fail; document that.)

### `OpenWorld(ctx context.Context, worldID string) (*thebibites.LoadedSave, error)`
1. Nil/empty guards.
2. Fast path under the lock: `w.mu.Lock(); if ls, ok := w.worlds[worldID]; ok { w.mu.Unlock();
   return ls, nil }; w.mu.Unlock()`.
3. Lazy-load: `return w.Load(ctx, worldID)`. **Do not hold `w.mu` across the `Load` call** —
   `Load` takes `w.mu` itself (non-reentrant `sync.Mutex` → self-deadlock otherwise). This
   is the same drop-lock-around-work-then-let-the-callee-relock shape `StartNode` uses
   (`node.go:94-121`). A benign race where two concurrent `OpenWorld`s both miss the fast
   path and both `Load` is acceptable: `Load`'s final `w.worlds[worldID] = ls` is
   last-writer-wins under the lock, both callers get a valid head-loaded handle, and the
   loser's handle is GC'd (no leaked resource — `LoadedSave` owns nothing closeable;
   `ls.db` is the shared workspace handle). Note this benign-race tradeoff in a comment so a
   reviewer does not mistake it for a bug.

### Locking summary (the seam a reviewer must check)
- `w.mu` is the **single** per-workspace mutex (declared `workspace.go:45`, shared with
  node.go's active set). C2 guards `w.worlds` with it.
- `Load` and `Unload` each take it; `OpenWorld` takes it only for the map peek and then
  delegates to `Load` (which re-takes it) — never nest `w.mu` (it is non-reentrant; see
  the identical caution in `world.go:78-82` and `node.go`).

## Risks / seams

- **`ls.saveID == worldID` is the load-bearing invariant** (workspace_plan.md lines 49-52,
  264-266). `LoadInto` enforces it by construction (`loadedsave.go:146-157` →
  `newLoadedSave(path, worldID, …)`), so C2 must pass `worldID` (not the revision sha256, not
  the file hash) as the second arg. Passing the wrong key would make every working-copy
  read/mutation (C3/C4) miss the C1-seeded partition. The `TestLoad…` partition-count
  assertion pins this.
- **No DuckDB import on Load — by design.** `LoadInto` injects `w.duck()`, so the lazy
  `openDB` short-circuits and imports nothing (`loadedsave.go:420-441`). C1 already seeded
  `save_id=worldID`. **Do not** call `ImportExtractedSave`/`ReplaceExtractedSave` in C2 — that
  would double-import and (worse) is a DuckDB write that would need the single-writer
  discipline. Load is a read path against the registry+blobstore plus an in-memory parse.
- **Non-reentrant `w.mu`.** `OpenWorld` must release the lock before delegating to `Load`
  (which relocks). Nesting self-deadlocks. This is the single subtlest line in the ticket;
  the `node.go` StartNode pattern (`node.go:94-121`) is the precedent to mirror.
- **Temp-file lifetime.** Load writes head bytes to a temp file only so `LoadInto` can parse
  it; the parsed archive lives in memory afterward, and C3's commit does **not** re-read
  `ls.path`, so `defer os.Remove(tmpPath)` in Load is safe. **Verify** `loadedsave.go:561-623`
  (`WriteSave`/`prepareCommit`) does not read `ls.path` before defer-removing; the plan's
  default (defer-remove) assumes the verified-safe case.
- **Head-never-NULL guard.** `World.HeadRevisionID` is `*int64`; dereference only after the
  nil check. A head-less world (partial C1 import) must yield a clean error, not a panic.
- **Blob-present guard is C2's G4 stub.** C2 returns a plain error on `!rev.BlobPresent`;
  G4 upgrades it to `ErrNotRematerializable` (workspace_plan.md lines 151, 493-500). Leave
  the marked comment so G4 finds the seam. Do not invent the typed error in C2 (out of scope;
  G4 owns the error type and the full blob-dependent-path sweep).
- **File-level contention (herder coordination — C4 is being planned in parallel; D2 is in
  flight).** C2 adds **only** `working_set.go` + `working_set_test.go` and edits **no**
  existing file. C1's `world.go`, B2's `workspace.go`, and D1/D2's `node.go`/`node_test.go`
  are untouched. C4 (Query) should add a new `query.go` (`world.query` / `world.history_query`
  / `workspace.query`) — there is **no** file overlap between C2 and C4 if both stay in their
  own new files, and the only shared surface is the read-only `Workspace` struct fields
  (`w.worlds`, `w.mu`) already declared in `workspace.go` by B2. C2 reads `w.worlds` (writes
  on Load/Unload); C4 reads `w.worlds` (via `OpenWorld`) for the working-copy query and never
  writes it. As long as neither C2 nor C4 edits `workspace.go`'s struct definition, they
  merge cleanly. The herder can run C2 and C4 concurrently; the only serialization point
  would be if C4's plan also adds methods to `world.go` or `workspace.go` (it should not).
