Source: `docs/missing.md` §7. Base: `7fe7671` (includes M1's READ-path edits to
`sql.go`: identityTable / fromClause / spanningFromExpr). M7 owns the WRITE-path
region of the same file and the per-surface write code.

Tier: **high** — every chosen bullet is a mutation/write-guard/mirror-consistency
change (persistence + shared write contract + a silent-failure invariant a passing
test can miss). exec=opus, review=opus.
Perf: no flag — all changes are per-call binding-layer work, no hot path / N+1 /
unbounded scan introduced (the existing batched bulk paths are reused unchanged).

---

## Goal

Close the four **correctness / silent-failure** write gaps §7 names, leaving the
pure-ergonomics bulk-predicate-on-every-surface gap explicitly OUT of scope (see
"Scope decisions" — flagged for the herder). "Done" means:

1. **Write NULL** — a script can clear an optional scalar (`b.field = None`) and
   round-trip `b.field = b.field` when the field currently reads `None`, instead of
   `fromStarlark` rejecting `None`. Mirror + stage + commit all carry a JSON null.
2. **`b.delete()` returns a count** — a single-entity delete returns `1` (int), not a
   silent `None`, matching the bulk `where(...).delete()` count contract.
3. **`species_id` set is referentially checked** — setting `species_id` (scalar or
   bulk) to an id that no species row carries is rejected loudly, not just
   non-negativity.
4. **Structural-edit in-run mirror visibility is made HONEST** — the read-after-write
   asymmetry (scalar sets are mirrored, structural append/delete are not) is no longer
   a silent trap: a same-run read after a structural edit fails loudly with a
   diagnostic, instead of silently returning the pre-edit set. (See "Approach §4" for
   why we make it loud rather than mirror structural ops — the latter is a much larger,
   higher-risk change deferred with a flag.)

## Files to change

All paths under `script/thebibites/` unless noted.

1. **`convert.go`** — `fromStarlark` (NULL write, bullet 1).
   - Add a sentinel for `starlark.NoneType`: return a distinct `nil`-bearing result so
     callers that OPT IN can stage a JSON null, while callers that do not opt in still
     reject None. **Do NOT** blanket-accept None (9 callers; most have no nil branch).
     Concrete shape: keep `fromStarlark` rejecting None (unchanged for existing
     callers) and add a SECOND helper `fromStarlarkNullable(v) (any, isNull bool, err)`
     that returns `(nil, true, nil)` for None and delegates to `fromStarlark` otherwise.
   - `setRowField`: add a `goVal == nil` branch that sets the in-memory field to its
     zero value (the read path already maps absent 1:1 rows / null cells to `None`, so a
     null write must read back as `None`; see Risks for the read-back subtlety).
2. **`guards.go`** — NULL validation + species referential guard (bullets 1, 3).
   - `validateValue` / a new `validateNullable`: a null write is allowed ONLY for a
     column that is genuinely optional (reads as None when absent). Today there is no
     "nullable" bit in `attrSpec`/`Rule`; the conservative, low-risk rule is: a null
     write is accepted for any writable scalar column whose current cell can be absent.
     Implement by gating null at the SetField call sites (entity scalar path), not by a
     new metadata bit — see Approach §1 for the exact gate.
   - `species_id` referential check: add a guard helper
     `ls.validateSpeciesID(goVal)` that scans `ls.tables.Species` for a row with
     `HasSpeciesID && SpeciesID == n`. Keyed off the generated source column
     `species_id` (the same key `nonNegativeColumns` uses), so it applies to both the
     scalar `SetField` path and the `bulkSet`/`bulkSetExpr` paths.
3. **`entity.go`** — `deleteBuiltin` (bullet 2) + species guard wiring + null write.
   - `deleteBuiltin`: return `starlark.MakeInt(1)` on a successful stage instead of
     `starlark.None`.
   - `SetField`: thread the nullable path (call `fromStarlarkNullable`) and the
     species-id referential guard before staging.
4. **`sql.go`** (WRITE-path region only — `bulkSet`, `bulkSetExpr`, `writeThroughAndStage`)
   — species referential guard in the bulk path (bullet 3). The READ-path builders
   (`identityTable`/`fromClause`/`spanningFromExpr`, owned by M1) are NOT touched.
   - `bulkSet`: after `validateSet`, if `spec.sourceColumn == "species_id"`, run
     `ls.validateSpeciesID(goVal)` once (constant value → one check, before the query).
   - `bulkSetExpr`: per-row, after `coerceExprResult` + `validateSet`, run the same
     check on the computed value.
5. **`mirror.go`** — structural-edit stale-read loudness (bullet 4). See Approach §4 for
   the exact mechanism (a "structural ops staged" flag consulted by `flushMirror` /
   the query head). The flag itself most likely lives on `LoadedSave` (loadedsave.go);
   `mirror.go`/`sql.go query()` consult it.
6. **`loadedsave.go`** — add the `structuralStaged bool` flag (or reuse an existing
   counter) set by every structural stage path; read by the query head. Confirm against
   the actual file before editing (the stage paths are `stageEntityDelete`,
   `bulkDelete`, `Zone.deleteBuiltin`, `PendingZone.appendBuiltin`, `Pellet.deleteBuiltin`,
   `PendingPellet.appendBuiltin`, and the subcollection append/delete).

NO change to `api/web/app.js` is required by M7 (see "Contention" — the chosen bullets
add no new top-level method NAMES; `set`/`set_expr`/`delete` already exist in
`AC_TYPE_MEMBERS`, and `b.field = None` is attribute assignment, not a method).

## Approach (steps, reusing existing utilities)

### §1 — Write NULL (`b.field = None`, round-trip `b.field = b.field`)

Read path today: `Entity.Attr` returns `starlark.None` when the 1:1 sub-table row is
absent (`entity.go:51-55`), and gene/setting cells read None for `ScalarNull`. So a
round-trip `b.field = b.field` feeds `None` back into `SetField`, which dies at
`fromStarlark` (`convert.go:95`). The mutator ALREADY accepts a nil value end-to-end:
`setJSONPath` → `normalizeJSONValue(nil)` returns nil → JSON null (`path.go:172,187`,
`json.go:89`). So the entire gap is the binding layer.

Steps:
1. Add `fromStarlarkNullable(v) (any, bool, error)` to `convert.go`: `(nil, true, nil)`
   for `starlark.None`; otherwise delegate to `fromStarlark` and return
   `(val, false, err)`. Keep `fromStarlark` itself unchanged so the 8 other callers
   that must NOT accept None are unaffected.
2. `setRowField` (`convert.go:128`): add a leading `if goVal == nil` branch that zeroes
   the field via `field.Set(reflect.Zero(field.Type()))` and returns `nil` as the
   staged value (so the mirror + stage carry nil → JSON null). The in-memory zero value
   then reads back through `toStarlark` as 0/""/false — NOT None — which is the
   subtlety: see Risks. The mirror writes a SQL NULL via the existing CAST-of-? path
   (a nil arg becomes SQL NULL).
3. `Entity.SetField` (`entity.go:87`): switch to `fromStarlarkNullable`; when `isNull`,
   SKIP `validateSet` (type/range/enum do not apply to a clear) but KEEP the
   species-id guard's "None is not a valid species" path if the column is species_id
   (a null species_id is the absence case — decide per Risks; default: allow null
   clear only for genuinely optional columns, reject null on species_id with a clear
   message since species_id is not nullable in practice).
4. Stage path: the existing `stageScalarSet` already records the mirror with the staged
   value (nil) and `flushMirrorColumn` already CASTs `?` to the column type, so a nil
   arg flushes as SQL NULL with no mirror.go change for the value itself.

Scope of NULL write in M7: the **entity scalar** surface (`b.field = None`). Gene and
setting null-clear are NOT added (their cells are typed scalars with no "absent"
semantic; out of scope, consistent with the ergonomics cut). State this in the doc
comment.

### §2 — `b.delete()` returns a count

`entity.go:159 deleteBuiltin` stages then `return starlark.None, nil`. Change the
success return to `starlark.MakeInt(1), nil`. The bulk path
(`collection.go:296 deleteBuiltin` → `bulkDelete`) already returns a count; this aligns
the single-entity path. The referential guard still fires at commit (unchanged) — the
count is "staged", not "committed", exactly as the bulk path documents.

### §3 — `species_id` referential guard

Today `species_id` only has the non-negative shape rule (`guards.go:100-110`); §8 notes
the same class of gap on gene-SET. Add:
1. `func (ls *LoadedSave) validateSpeciesID(n int64) error` in `guards.go` (or a small
   new helper file): scan `ls.tables.Species` for `HasSpeciesID && SpeciesID == n`;
   error `species_id %d does not match any species in this save` if none. The species
   set is small (one row per species), so a linear scan is fine — no perf flag.
2. Call it from `Entity.SetField` (after `validateSet`, before staging) and from
   `bulkSet` (once, before the query — the value is constant) and `bulkSetExpr`
   (per-row, after coerce+validate). Gate on `spec.sourceColumn == "species_id"` so it
   only runs for that column. Reuse the existing `int64` coercion (`asInt64`) the value
   already passed through.

This is a binding-layer ADD (does not touch the mutator's `resolveGeneColumn` gap from
§8 — that is a separate ticket). It composes with the existing non-negative rule
(both must pass).

### §4 — Structural-edit in-run mirror visibility (make the stale read LOUD)

Today: scalar sets mirror into DuckDB (`mirror.go` + `flushMirror` at the query head),
but structural ops (`stageEntityDelete`/`bulkDelete`/zone+pellet append/delete/subcoll)
are staged-but-not-mirrored (`mirror.go:17-18` contract, `entity.go:153`, `zones.go`,
`pellets.go`). So `zone.clone(i).append(); save.zones[...]` or
`coll.where(...).delete(); s.sql("...")` silently reads the PRE-edit set in the same run
— a read-after-write correctness trap (§7 bullet, §4 open question "Mirror consistency").

M7 makes this HONEST without the (large, risky) full structural-mirroring change:
1. Add `structuralStaged bool` to `LoadedSave` (loadedsave.go). Set it `true` in every
   structural stage path: `stageEntityDelete`, `bulkDelete` (after the loop or per
   stage), `Zone.deleteBuiltin`, `PendingZone.appendBuiltin`, `Pellet.deleteBuiltin`,
   `PendingPellet.appendBuiltin`, and the subcollection append/delete in
   `subcollection.go`. (`stagedOps++` already marks all of these — consider deriving
   the flag from a dedicated counter rather than overloading `stagedOps`, which scalar
   sets also bump; a dedicated `structuralStaged` is clearest.)
2. In the query head — `ls.query` (`sql.go:55`) right after/around `flushMirror` — if
   `structuralStaged` is set, RETURN A LOUD ERROR for any analytics read
   (`Query`/`sql`/`where`/aggregate) explaining that a structural edit was staged this
   run and in-run reads cannot observe it; the read must run before the structural edit,
   or after commit. This converts a silent wrong answer into a clean diagnostic.
   - Verify the exact seam: `flushMirror` lives in `loadedsave.go` and is called at the
     head of `ls.query`. The new guard belongs at the same head so EVERY push-down read
     (Query, sql, where-resolution via `matchingEntryNames`, scalarAgg, groupedAgg)
     hits it — they all route through `ls.query`.
   - This must NOT fire for a pure scalar-set run (those ARE mirrored and must stay
     readable) — hence a dedicated structural flag, not `stagedOps`.
   - Decision to FLAG: making the read a hard error (vs a warning) is the loud-failure
     choice consistent with the codebase's "loud but localized" philosophy. Confirm the
     herder is OK with the read becoming an error after a structural stage (the
     alternative — silently stale — is the bug §7 reports). Tests that today do
     `structural edit → in-run read` will need reordering; check the existing suite.

## Risks / seams

- **NULL read-back semantics (the subtle one).** A null write zeros the in-memory field
  (`setRowField` nil branch), so a later plain attr read returns 0/""/false via
  `toStarlark`, NOT `None`. But an ABSENT 1:1 row reads as `None` (`entity.go:51-55`).
  So `b.f = None` followed by `b.f` does NOT round-trip to None for a present row — it
  reads 0. This is acceptable (the JSON null commits correctly; the in-run read is a
  zeroed cell) but MUST be documented and the test must assert the ACTUAL semantics, not
  an assumed None round-trip. Do not try to make the in-memory row "absent" — that is a
  far larger change. The committed JSON null is the contract; the in-run read-back is
  best-effort. Call this out in the `setRowField` doc comment.
- **species_id nullability.** species_id is in `nonNegativeColumns` and is not optional
  in practice. Decide: `b.species_id = None` should be REJECTED (not a valid clear), so
  the null path must not bypass the species guard for species_id. Implement by checking
  species_id BEFORE the null short-circuit, or by simply not treating species_id as
  nullable. Default: reject null on species_id with a clear message.
- **Structural flag must not fire on scalar sets.** Use a DEDICATED `structuralStaged`
  flag, never `stagedOps` (scalar sets bump that too). A false positive would break
  every mirrored scalar-set read-after-write — the feature M7 must preserve.
- **`fromStarlark` is shared by 9 call sites.** Do NOT change its existing None-rejecting
  behavior; only ADD the nullable variant and route the entity scalar SetField through
  it. Changing `fromStarlark` itself silently opens None on gene/setting/zone/pellet/
  subcollection/bulk writes that have no nil handling — a latent corruption seam.
- **M1 owns the READ-path builders in sql.go.** Stay in the write-path region
  (`bulkSet`/`bulkSetExpr`/`writeThroughAndStage`/`bulkDelete`/`matchingEntryNames` and
  the new query-head guard). Do not touch `identityTable`/`fromClause`/
  `spanningFromExpr`/`rewritePredicate` (M1's region / shared read builders).
- **Read-only spanning scope.** The species guard and null write run only on the
  working (mutable) path; spanning collections already reject mutation
  (`collection.go rejectMutation`/`readOnly`). No change needed, but confirm the new
  code is not reached on a read-only collection (it isn't — mutation is gated before
  `bulkSet`).
