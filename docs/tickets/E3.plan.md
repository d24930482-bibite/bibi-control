# E3 — Spanning entity collections (world.bibites / workspace.bibites) over the object DSL

Tier: high — shared-contract DSL surface over generated metadata + a new cross-world
scoping seam injected into the existing push-down builders; correctness is a
cross-world-isolation invariant a passing test can easily miss (the E2 leak class).
Both exec and review = opus.

Perf-review: the spanning aggregate path runs `scalarAgg`/`groupedAgg` over the
RETAINED history partitions joined to `mirror_saves` (every world's every revision),
so its query shape grows O(R) with retained history. The catalog must be refreshed
ONCE per spanning call (reuse `refreshMirrorCatalog`, which is fingerprint-gated —
workspace/query.go:86) and the per-call work must stay O(1) round-trips: one
`refreshMirrorCatalog` (cache-hit when unchanged) + one aggregate SELECT. The
spanning collection must NOT re-resolve a per-revision query or rebuild the catalog
per `.where()`/`.count()`. Flag set so the reviewer runs the perf lens on this path.

## Goal

Bind the SAME `EntityCollection` / `GroupedCollection` object DSL (script/thebibites/
collection.go) to a scope broader than one open world, exposed as `world.bibites` /
`world.eggs` / `world.pellets` (one world's whole retained history) and
`workspace.bibites` / `workspace.eggs` / `workspace.pellets` (all worlds). The user
writes the identical surface they already know — `.where('energy < 10').count()`,
`.group_by('sim_time').mean('energy')`, `.group_by('world_id').count()` — with NO raw
SQL and NO JOINs. `world_id` and `sim_time` become friendly columns usable in
`.where()` / `.group_by()` like any entity column. The scope is READ/AGGREGATE ONLY:
the spanning surface exposes only count/sum/mean/median/min/max/quantile/group_by and
must REJECT every mutation (`.set`/`.set_expr`/`.delete`, and `SetField` on any
yielded entity) loudly. The save-scoping (which `save_id`s are in scope) is DERIVED
from the `mirror_saves` catalog and injected BY CONSTRUCTION, never by the caller
adding a JOIN/WHERE. "Done" = these spanning collections exist on the world/workspace
automation values, push down to DuckDB through the existing builders via a new
`SaveScope`, and the test matrix below is green. `workspace.query(sql)` /
`world.history_query(sql)` survive unchanged as the documented power-user escape hatch
(demote in doc comments only — no behavior change).

## Design: the `SaveScope` seam (reuse, do not fork the builder)

Today every push-down builder is a `*LoadedSave` method and `whereClause`
(sql.go:335) hardcodes the working-partition scope: `WHERE <identity>.save_id = ?`
with `args = [ls.saveID]`. That single hardcoded clause is the ONLY thing tying the
analytics path to one working partition. The plan introduces a `SaveScope` interface
that supplies that clause + args, plus writability + extra friendly columns, and
routes the existing builders through it. The single-save path keeps its current
behavior by construction (it gets a working-partition scope equal to today's clause).

`SaveScope` (new, script/thebibites — exported so the workspace package constructs
spanning scopes):

```go
// SaveScope supplies the save-partition filter the analytics push-down injects so a
// collection's reads stay scoped BY CONSTRUCTION (the caller never writes save_id /
// JOIN). It also declares writability (spanning scopes are read-only) and any extra
// friendly columns the scope contributes (world_id / sim_time from mirror_saves).
type SaveScope interface {
    // scopeClause returns the SQL fragment AND-ed into the WHERE that restricts
    // every push-down query to this scope's save_id set, plus its bound args. For
    // the single-save scope: "<identity>.save_id = ?" / [saveID]. For a spanning
    // scope: "<identity>.save_id IN (SELECT save_id FROM mirror_saves WHERE
    // world_id = ?)" / [worldID]  (one-world) or "<identity>.save_id IN (SELECT
    // save_id FROM mirror_saves)" / [] (all-worlds). identity is passed so the
    // clause qualifies correctly against the joined FROM.
    scopeClause(identity string) (clause string, args []any)
    // writable reports whether mutation is permitted (only the single-save working
    // scope is writable). Spanning scopes return false.
    writable() bool
    // catalogJoin returns an optional extra FROM/JOIN fragment that brings
    // mirror_saves into scope so world_id/sim_time resolve as columns. "" for the
    // single-save scope (no catalog needed). For spanning scopes it LEFT JOINs
    // mirror_saves ON <identity>.save_id = mirror_saves.save_id.
    catalogJoin(identity string) string
}
```

Keep the `*LoadedSave` receiver on the builders but thread an explicit `SaveScope`
through them instead of reaching for `ls.saveID`:
- `scalarAgg`, `groupedAgg`, `matchingEntryNames`, `bulkSet*`, `bulkDelete` and their
  query builders (`bulkSetQuery`, `bulkSetExprQuery`) currently call `ls.whereClause`.
  Change `whereClause` to take a `SaveScope` and emit `scope.scopeClause(identity)`
  instead of the hardcoded `save_id = ?`. The single-save callers pass a
  `workingScope{saveID: ls.saveID}`; the spanning aggregate entry points pass the
  spanning scope.
- `world_id` / `sim_time` resolution: extend `rewritePredicate` and `resolveColumn`
  so that, when the active scope contributes catalog columns, a bare `world_id` /
  `sim_time` token rewrites to `mirror_saves.world_id` / `mirror_saves.sim_time` and
  pulls the catalog join into the FROM `needed` set. Do this WITHOUT a hand-kept
  allowlist beyond the two catalog column names the catalog table actually defines
  (workspace/query.go:48-55 — `mirror_saves(save_id, world_id, tier, blob_present,
  sim_time)`); the entity columns continue to derive from generated metadata
  (`attrRegistry()`), satisfying the sqlref-generation-philosophy memory (no parallel
  entity allowlist).

This keeps ALL friendly-column rewriting, JOIN cardinality guarding
(`oneToManyTables`, sql.go:289), aggregate compilation (`aggExpr`), and grouped-dict
shaping in ONE place — the spanning path is the same code with a different scope and
a catalog join, not a forked query builder.

## Files to change

- **script/thebibites/scope.go** (NEW): `SaveScope` interface; `workingScope`
  (single working partition — `save_id = ?`, writable=true, no catalog join);
  `spanningScope` (carries `saveIDsClause` built from a `world_id` filter or none,
  writable=false, catalog LEFT JOIN). Exported constructors the workspace calls:
  `NewWorldHistoryScope(worldID string) SaveScope` and `NewWorkspaceScope() SaveScope`.
  These do NOT hold a `*sql.DB`; they only describe the WHERE/JOIN, because the
  spanning aggregate runs on a `*LoadedSave` whose `ls.db` is the SHARED workspace
  handle (see Approach step 4).
- **script/thebibites/sql.go**: `whereClause(kind, predicate, scope)` →
  `scope.scopeClause(identity)`. `fromClause` gains an optional catalog-join append
  driven by a `needsCatalog bool`. `scalarAgg`/`groupedAgg`/`matchingEntryNames` gain
  a `scope SaveScope` parameter (default callers pass the working scope). New
  scope-aware entry points used by the spanning collection:
  `scalarAggScoped(kind, where, scope, agg)`, `groupedAggScoped(kind, where, scope,
  group, agg)`. `resolveColumn`/`rewritePredicate` learn the two catalog columns when
  the scope exposes them (pass a small `catalogCols map[string]string` or a flag).
- **script/thebibites/collection.go**: add a `scope SaveScope` field to
  `EntityCollection` and `GroupedCollection` (nil ⇒ default working scope, preserving
  every existing single-save test). When `scope != nil && !scope.writable()`:
  - `Attr` must NOT return `set`/`set_expr`/`delete`/`group_by`-then-mutate; return
    only the aggregate + `where` + `group_by` builtins. `AttrNames` shrinks
    accordingly for a read-only scope.
  - `setBuiltin`/`setExprBuiltin`/`deleteBuiltin` must reject loudly even if reached
    ("cannot mutate a spanning <kind> collection; mutation is per-save: open a world
    and use s.bibites.set / .delete"). Belt-and-suspenders with the Attr gate.
  - `runAgg`/`groupByBuiltin`/`whereBuiltin` route through the scoped entry points.
  - Iteration: a read-only spanning collection must NOT yield mutable `Entity`s. The
    safest move per the non-negotiables is to make the spanning collection
    NON-iterable for v1 (aggregate/group_by only) — `Len`/`Iterate` return a loud
    "spanning <kind> collections are aggregate-only; iterate a single open world".
    (Yielding an `Entity` whose `SetField` is gated is the fallback if iteration is
    later wanted, but v1 stays aggregate-only to keep the read-only guarantee total
    and avoid materializing cross-world rows — matches the ticket's READ/AGGREGATE
    ONLY scope.)
- **script/thebibites/save_value.go**: no change to `Save` (single-save object keeps
  `bibites`/`eggs` writable). The spanning collections attach to world/workspace
  values, not to `Save`.
- **script/thebibites** pellets/eggs spanning: pellets today are an in-memory
  indexable surface (pellets.go), NOT a push-down `EntityCollection`. For spanning
  aggregate parity (ticket non-negotiable #3), expose `world.pellets` /
  `workspace.pellets` as an `EntityCollection`-style aggregate-only collection over
  the generated `pellets` table via the SAME scoped push-down. Register a `pellet`
  entry in `entityTables` for the analytics/push-down path ONLY (identity table
  `pellets`; aggregate-only — bulkSet/delete/iteration already reject non-bibite/egg
  kinds in several places, so audit `locatorSelect`, `EntryNames`, and
  `stageEntityDelete` to confirm a `pellet` kind cannot reach a mutation path; the
  read-only scope already blocks mutation, so the analytics-only registration is the
  minimal addition). `pelletRegistry()` already derives pellet columns from generated
  metadata, so reuse it for `attrRegistry["pellet"]` rather than a parallel list.
  NOTE: this is the one place to be careful — keep the change additive so the existing
  `save.pellets` in-memory surface is untouched.
- **workspace/spanning.go** (NEW): the binding glue. A scope-bound LoadedSave for the
  spanning read: build a `*thebibites.LoadedSave` whose `ls.db` is `w.duck()` (the
  shared workspace handle) but whose reads run under a spanning `SaveScope`. Reuse the
  existing `thebibites` accessor that injects the shared db (`LoadInto` requires a
  path/parse; instead add a tiny exported `thebibites.NewSpanningReader(db *sql.DB,
  scope SaveScope) *SpanningReader` in script/thebibites that holds the db+scope and
  produces scoped `EntityCollection`s WITHOUT parsing any save — a spanning read needs
  no in-memory archive, only DuckDB). The workspace refreshes `mirror_saves`
  (`refreshMirrorCatalog`, under `w.mu`) before handing out the reader so world_id/
  sim_time/save_id scoping is current.
- **workspace/automation.go**:
  - `worldValue.Attr`/`AttrNames`: add `bibites`, `eggs`, `pellets` → spanning
    collections via `NewWorldHistoryScope(v.world.ID)`.
  - `workspaceValue.Attr`/`AttrNames`: add `bibites`, `eggs`, `pellets` → spanning
    collections via `NewWorkspaceScope()`.
  - Reposition the `query`/`history_query` doc comments to call them the power-user
    escape hatch (doc only).

## Approach (steps, citing existing utilities)

1. **Introduce `SaveScope`** (scope.go). `workingScope.scopeClause(identity)` returns
   exactly today's clause (`quoteIdent(identity)+".save_id = ?"`, `[saveID]`) so the
   single-save path is byte-identical. `spanningScope` builds
   `"<identity>.save_id IN (SELECT save_id FROM mirror_saves" + optional " WHERE
   world_id = ?" + ")"`. The IN-subquery references `mirror_saves`, which is a native
   DuckDB table in the same workspace handle (workspace/query.go:48), so no ATTACH is
   needed — the spanning read runs on `w.duck()`.
2. **Thread the scope through the builders** (sql.go). Change `whereClause` to accept
   a `SaveScope`; update every internal caller. Add `scalarAggScoped`/
   `groupedAggScoped` that mirror `scalarAgg`/`groupedAgg` (sql.go:363/409) but take a
   scope and set `needsCatalog` when the predicate/group/agg references a catalog
   column, appending `scope.catalogJoin(identity)` to the FROM. Keep the
   `oneToManyTables` guard intact (sql.go:319).
3. **Catalog columns as friendly columns.** In `rewritePredicate` (sql.go:1004) and
   `resolveColumn` (sql.go:235), when the active scope exposes catalog columns, treat
   `world_id`/`sim_time` as resolving to `mirror_saves.world_id`/`mirror_saves.sim_time`
   and flag the catalog join needed. These two names come from the catalog DDL, not a
   hand-kept entity allowlist. A `world_id` predicate on the all-worlds scope is the
   friendly cross-world filter; on the one-world scope it is redundant but harmless.
4. **Spanning reader without a parsed archive** (script/thebibites). Add
   `NewSpanningReader(db *sql.DB, scope SaveScope)` returning a value with
   `.bibites/.eggs/.pellets` builtins that yield `&EntityCollection{ls: <reader-backed
   ls>, kind: …, scope: scope}`. The reader-backed `ls` is a `*LoadedSave` with only
   `db` and a no-op session set (it never mutates; reuse `newLoadedSave`-style
   construction but with a nil archive/empty tables, OR a thinner struct that
   `EntityCollection` can target). Simplest: give `EntityCollection` an optional `db`
   override path so a scoped collection can run `ls.query` against the shared handle.
   IMPLEMENTATION CHOICE (executor): the cleanest minimal change is a `*LoadedSave`
   constructed with `db=w.duck()`, `saveID=""` (unused by a spanning scope),
   `archive=nil`, no `buildAccess()` — its only used method is `ls.query` (sql.go:46),
   which needs `db` + `flushMirror` (a no-op when `mirrorDirty` is false). Verify
   `flushMirror`/`openDB` tolerate a nil archive on a read-only spanning ls (openDB
   early-returns since db!=nil, loadedsave.go:427); add a guard if `flushMirror`
   dereferences the archive.
5. **Wire the automation surface** (workspace/automation.go + workspace/spanning.go).
   `worldValue.bibites` etc. call a helper `w.spanningBibites(ctx, scope)` that (a)
   locks `w.mu`, runs `refreshMirrorCatalog(ctx)` (workspace/query.go:86 — fingerprint
   gated, O(1) when unchanged), unlocks, then (b) returns the scoped
   `EntityCollection`. The collection's later `.count()` runs ONE aggregate SELECT on
   `w.duck()` via the scoped builder. Refresh-once-per-call, not per-`.where()`.
6. **Read-only enforcement.** The spanning `EntityCollection` carries a non-writable
   scope, so `Attr` exposes only aggregate/where/group_by and the mutation builtins
   reject loudly. There is no `Entity` materialization (aggregate-only), so there is
   no `SetField` reachable on a spanning row — but ALSO add a guard test asserting a
   spanning collection has no `set`/`delete` attribute and that a spanning entity (if
   ever materialized) rejects SetField, to lock the invariant.

## Tests (and the command)

Add to **script/thebibites** (white-box, scope + read-only gates) and **workspace**
(integration, two real worlds). Reuse the workspace harness: `newWorkspace(t, ctx)`,
`fixturePath(t, fixtureA/fixtureB)`, `AddWorld` (world_test.go), and
`RunAutomation`/`AutomationGlobals` for the Starlark-level assertions.

1. **Cross-world isolation (the E2 leak guard).** `workspace/spanning_test.go`:
   AddWorld×2 with fixtureA and fixtureB (different bibite counts — assert they differ
   first). `world_a.bibites.count()` must equal world A's own bibite count and EXCLUDE
   world B's; same for B. This is the load-bearing test — it proves scoping is by
   construction (no caller JOIN). Run it via the spanning collection AND via a Starlark
   program through `RunAutomation` so the user-facing surface is covered.
2. **group_by('world_id') across the workspace.** `workspace.bibites.group_by(
   'world_id').count()` returns a dict with exactly two keys (the two world ids) whose
   counts sum to the total and match each world's individual count. No raw SQL/JOIN in
   the script.
3. **Friendly-column predicate over world_id / sim_time.**
   `workspace.bibites.where("world_id = '<A>'").count()` == world A count;
   `world_a.bibites.group_by('sim_time').count()` returns per-sim_time buckets (with a
   single revision, one bucket). Commit world A a 2nd time (CommitWorldLoaded) and
   assert `world_a.bibites.group_by('sim_time')` now spans BOTH revisions (history
   retention proven through the DSL, not history_query).
4. **Mutation rejected on a spanning scope.** Assert `world.bibites` has no `set`/
   `set_expr`/`delete` attribute (AttrNames + Attr returns nil/None) and that calling
   any of them through Starlark raises a clean error naming the per-save path. Assert
   the same for `workspace.bibites`. If a spanning entity is materialized in any path,
   its `SetField` returns the read-only error.
5. **No user-written JOIN anywhere.** A Starlark program using only
   `world.bibites.group_by('sim_time').mean('energy')` and
   `workspace.bibites.group_by('world_id').count()` runs green — documents that the
   cross-world read needs zero SQL. (Negative: `workspace.bibites.where("save_id IN
   (...)")` is unnecessary; the scope already filters.)
6. **eggs + pellets parity.** Repeat isolation + group_by('world_id') for `eggs` and
   `pellets` (pellets via the new analytics-only `pellet` kind). Skip a kind only if
   the fixtures contain zero of it (assert presence first).
7. **Single-save regression.** The whole existing script/thebibites suite stays green
   (the working scope is byte-identical), proving the `SaveScope` refactor is
   behavior-preserving for `s.bibites`/`s.eggs` set/delete/aggregate.
8. **Perf shape.** Reuse the catalog seams (workspace.go:65-75 `rebuildCount`,
   `scenesReadCount`, `insertExecCount`): N repeated spanning `.count()` calls on an
   unchanged workspace trigger at most ONE catalog rebuild total and ONE aggregate
   SELECT per call (no per-revision loop). Assert `rebuildCount` does not grow across
   repeated spanning reads when nothing changed.

Command: `go test ./script/thebibites/... ./workspace/...` (and `go test ./...` before
handing off).

## Risks / seams

- **Cross-world leak (E2 class, the headline risk).** The scope clause is the ONLY
  thing isolating a world's reads; an aggregate that forgets to AND the scope clause
  (or a catalog join that widens instead of filtering) silently reads all worlds. Test
  1 with two DIFFERENT-count worlds is the assertion that catches it. The scope MUST
  be injected by construction in the builder, never left to the script.
- **`mirror_saves` carries only sha256 history keys, never the world-id working key**
  (workspace/query.go:262-264). So the spanning scope reads HISTORY partitions, and
  the open world's mutable working partition (save_id=worldID) is correctly EXCLUDED —
  a spanning read sees committed history, not uncommitted staged edits. Document this:
  spanning = committed history; working-copy reads stay on `s.sql`/`world.query`.
- **Catalog freshness.** Refresh `mirror_saves` under `w.mu` BEFORE handing out the
  spanning collection (reuse the fingerprint-gated `refreshMirrorCatalog`); do not run
  the aggregate while holding `w.mu` (mirror the HistoryQuery lock-then-release shape,
  query.go:283-295).
- **Shared workspace seam (H3 contention).** automation.go is touched by E2/F2 and now
  E3; H3 runs in parallel on workspace/world.go + commit.go + duckdb (different files).
  Keep E3's automation.go edits method-scoped (add `bibites`/`eggs`/`pellets` Attr
  cases + AttrNames entries only) so the merge with H3 stays clean. Do NOT refactor
  shared helpers in automation.go.
- **`pellet` analytics kind.** Adding `pellet` to `entityTables` for the push-down
  must not open a mutation path: audit `locatorSelect` (sql.go:974 — only bibite/egg),
  `EntityCollection.EntryNames` (collection.go:158 — already restricts to bibite/egg),
  `stageEntityDelete`. The read-only spanning scope blocks mutation regardless, but the
  kind registration must stay aggregate-only by construction.
- **nil-archive spanning ls.** The spanning read uses a `*LoadedSave` with no parsed
  archive; confirm `ls.query`/`flushMirror`/`openDB` don't dereference `archive` on the
  read path (openDB early-returns on db!=nil). Add a nil guard in `flushMirror` if
  needed rather than parsing a throwaway save.
- **Generated-metadata fidelity (sqlref-generation-philosophy).** Entity columns stay
  derived from `attrRegistry()`; only `world_id`/`sim_time` are added, and they come
  from the catalog DDL, not a hand-kept entity allowlist. Do not duplicate column
  lists.
- **DSL-not-raw-SQL (memory).** The whole point: the spanning surface is the object
  DSL; `query`/`history_query` stay as the documented escape hatch. Do not add any new
  raw-SQL-first cross-world read entry point.
