## #1 check — history byte-equality (silent-corruption risk): PASS

- `TestCopySavePartitionEqualsDoubleExtract` is a GENUINE equivalence test, not a
  golden generated from new code. It parses the real 3.1MB/1027-bibite fixture, then
  builds the H partition TWO independent ways into TWO dbs: the OLD path
  (`OpenAndImport(W)` then `ReplaceExtractedSave(ExtractTables(H, archive))`) and the
  NEW path (`OpenAndImport(W)` then `CopySavePartition(W, H)`). It row-compares the full
  H row set across EVERY `tb.NormalizedTables` entry (44 tables), ordered by all columns
  deterministically; additionally compares H (new) vs W-rewritten-to-H; and asserts
  idempotency (second CopySavePartition yields identical row counts). It asserts the
  comparison is non-vacuous (a populated table exists). Verified by running it: PASS in
  7.97s against the real fixture — so wide tables (bibites, brain synapses, settings,
  json_scalars) are exercised with real data.
- `copySaveSelectList` was scrutinized independently. It emits a NAMED-column positional
  INSERT: `INSERT INTO "t" (c0,c1,save_id,c3,…) SELECT c0,c1,?,c3,… FROM "t" WHERE
  save_id = ?`. cols[i] and exprs[i] are produced in the same loop iteration from the
  same `tb.NormalizedTables[i].Fields`, so each target column is fed its same-named
  source value with `save_id` substituted IN PLACE by the bound dstSaveID. Because the
  INSERT names its targets, byte-equality holds regardless of physical CREATE-TABLE
  column order — there is no off-by-one or column-shuffle surface. The same `.Fields`
  metadata drives `insertExtractedTable`'s `NewAppenderWithColumns(columns)`, so the
  write path and the copy path cannot drift. Verified all 44 normalized tables carry a
  `save_id` field and none has empty fields (so the `if insertCols == ""` skip branch is
  dead for real tables — no table is silently skipped). `bibite_mutation_refs` is a VIEW
  and correctly needs no copy.
- Bind order is correct: SELECT's `?` (dstSaveID) binds before WHERE's `?` (srcSaveID);
  callsites pass `(dstSaveID, srcSaveID)` and Exec args are `(dstSaveID, srcSaveID)`.

## -race finding (PRE-EXISTING, recorded, NOT blocking)

`go test -race -gcflags=all=-d=checkptr=0 ./workspace/...` showed ZERO `DATA RACE`
markers and zero checkptr panics, but two automation tests
(TestAutomation_OpenMutateCommitAdvancesHead, TestAutomation_WorldQueryScopedToOpenWorld)
FAILED with `context deadline exceeded` at ~32–37s. Root cause: `testCtxAuto`
(automation_test.go:21) gives each test a fixed 30s context budget; under -race the
1027-bibite import/mutate/commit workflow exceeds 30s on this slow pre-H2 base. The
failure point varied across runs (record script run / re-seed working partition / count
bibites), i.e. it is a timing/deadline artifact, not a seam.

Crucially, I proved this is NOT introduced by H3 by comparing apples-to-apples on the
same machine: the H3 PARENT commit `1b0c8f3` (F2, pre-H3, no CopySavePartition) under
identical -race conditions FAILS these same two tests identically (32.3s, 35.0s). H3
actually makes the import faster, so it reduces (not increases) the likelihood of hitting
the deadline. This is the known slow-workspace-under-race flake (memory: test-suite-cost;
herder note: H1/H3). Recorded; does not affect the verdict.
