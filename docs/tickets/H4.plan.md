# H4 — Drop the `json_scalars` EAV: stop building the 99.4%-discarded scalars + remove the unread table

> Branch base: this worktree is off **H2-merged main** (`67b6a22`); parse is already
> parallelized (`parse_parallel.go`) and decoding uses goccy. Plan H4 **on top of** that.
> Measure the win on the H2 base, not pre-H2.

## Goal

"Done" means the save-parse hot path no longer materializes the generic per-entity scalar
tree at all, and the `json_scalars` table no longer exists in the DuckDB schema — while every
**typed** table projection lands byte-for-byte identically to before, and format drift stays
**loud** (a new/unknown section still surfaces as a Diagnostic). Concretely:
**(A)** `collectScalars`/`collectScalarsInto`/`scalarCount` and every call site are deleted, the
per-entity `Scalars`/`*Scalars` fields are removed, so the ~869K `Scalar` structs (and the
~864K string/number `json.Marshal`/format calls on the discarded 99.4%) are never allocated;
**(B)** `JSONScalars` leaves `ExtractedSave`, the `json_scalars` table spec leaves
`NormalizedTables`, a new migration drops the table, and `normalizeScalars`/`isRedundantScalar`/
`scalarKeepSuffixes`/`typedEntityPrefixes`/`ScalarRow` are removed. The whole suite stays green
(`go test ./... -timeout 20m`).

**Diagnosis is settled (do not relitigate):** measured 869,468 `Scalar` structs built into
`a.Scalars`, 5,332 (0.6%) kept after `isRedundantScalar`, 864,136 (99.4%) discarded inside the
~2s parse. **No production reader of `json_scalars` exists** — verified again below.

## Tier

**Tier: high — `exec opus / review opus`.** Signals: **shared contract** (`ExtractedSave` is the
DuckDB import contract every table-driven path reads via reflection; `NormalizedTables` /
`allTables` are consumed by `duckdb/import.go` and, transitively, H3's `CopySavePartition`) +
**persistence/data-shape** change (a DuckDB migration that DROPs a table) + **subtle invariant**
(the churn-resilience guardrail: removing the EAV catch-all must not make format drift silent).
Wide mechanical surface (≈14 files) raises misread risk. Set by the herder; recorded here for audit.

**Perf-review: flag SET.** This ticket *is* a perf change; the reviewer must measure before/after.
See Risks → Perf-review.

### Churn-resilience guardrail (HARD constraint 1 — what replaces json_scalars' visibility)

Today an **unknown JSON section** (`EntryUnknownJSON`, e.g. a future renamed/added top-level
`*.bb8scene`) is parsed by `parseGenericJSON`, and its only data trace was `json_scalars` rows;
`parseGenericJSON` emits a Diagnostic ONLY when the payload is *not* a JSON object
(`json_not_object`) or fails to decode (`json_decode_failed`). A new unknown section that *is* a
valid object currently produces **no Diagnostic** — its sole signal was the (now-removed) EAV
rows. Losing data is accepted; losing **visibility** is not.

**Fix (additive, low-risk):** in `parseGenericJSON`, when called for the `unknown_json` ownerKind
(i.e. the `EntryUnknownJSON` branch), emit a loud Diagnostic — e.g.
`ctx.addDiagnostic(SeverityWarning, "unknown_json_section", entry.Name, "unrecognized JSON section
captured only as a diagnostic; add a typed table if its data is needed")`. Distinguish the
unknown-json caller from the `vars` caller (both go through `parseGenericJSON`): pass the
ownerKind through (it already is — `ownerKind == "unknown_json"`), or branch in
`parseEntryPayload`'s `EntryUnknownJSON` case. **`vars.bb8scene` must NOT trigger this** (it is a
known section with a typed `vars` row). This makes drift loud via the **Diagnostics** table —
the memory `save-format-churn-strategy` "loud/localized" contract — independent of json_scalars.

Note the existing structural backstop also survives: `save_entries` records **every** entry with
its `kind` (`ExtractTables` lines 22–33), so `unknown_json` entries remain visible there even
without the new Diagnostic; and `TestSaveSchemaFingerprint` (walks raw `entry.JSON`, not
json_scalars) still flags any new path-shape. The new Diagnostic is the explicit, queryable
loud signal the constraint demands.

## Risks / seams

- **Shared contract — `ExtractedSave`/`NormalizedTables` is the import API.** `duckdb/import.go`
  drives `deleteSaveRows`/`insertExtractedSave`/`allTables` purely from `tb.NormalizedTables` via
  reflection — so dropping `JSONScalars` there makes json_scalars **auto-leave** the import path
  with NO edit to `import.go`. **Do not touch `duckdb/import.go`** (it is H3's hot file; see
  Contention). The compose-point: H3's `CopySavePartition` iterates `allTables` and will simply
  see one fewer table.
- **Generated-file discipline.** `normalize_metadata.go`, `0001_extracted_save.sql`, and
  `sqlref_generated.go` are `DO NOT EDIT` outputs of `cmd/gen_thebibites_schema`. Only edit
  `normalize_types.go` (+ the one stale `ScalarRow` key in the gen tool) and re-run `go generate`.
  A schema-fingerprint / consistency test will fail if metadata and DDL drift.
- **Migration ordering / persistence (data-shape).** The DROP must be a NEW `0002_*.sql`, not a
  `0001` edit — `IF NOT EXISTS` create can't un-create a table already in an existing workspace
  DuckDB file. The DROP is idempotent (safe on fresh DBs). This is the only persistence-shape change.
- **Keep vs delete the value path.** `scalarParts`/`toFloat`/`rawJSON` (json_helpers.go) and the
  `ScalarType` constants are the **typed** value path (genes, settings values, changer targets) —
  they must survive. Only the generic tree walk (`collectScalars` & friends) and the per-entity
  `.Scalars` accumulation go. Don't conflate the two.
- **Empty-helper cleanup.** Several parse helpers (`parseBibiteGenes`, `parseBibiteClock`,
  `parseEggGenes`, `parseEggState`, the `species template.genes` block) existed ONLY to call
  `collectScalars`; after removal they are no-ops — delete the functions and their call sites
  rather than leaving empty husks. Double-check each does nothing else first (the plan lists what
  each also touched: dead/dying + stomach/children stay in `parseBibiteBody`).
- **Perf-review (flag SET): `refresh-not-applicable; this IS the perf change`.** The concern to
  re-verify: H2 already *parallelized* this build, so frame the H4 win honestly as
  **allocation/GC relief** — Go's GC is global, so the 864K discarded allocations + per-leaf
  `json.Marshal`/`strconv` formatting (`json_scalars.go:collectScalarsInto`) stall H2's parallel
  workers under GC pressure; removing them cuts allocs/op and GC pauses, which speeds the whole
  parallel parse, not just one goroutine. **Reviewer MUST:** (a) measure parse before/after on
  `testdata/saves/the-bibites/autosave_20260301021357.zip` (wall time + `-benchmem` allocs/op if a
  bench exists, else a timed `ParseFile`+`ExtractTables` harness) and confirm a material drop;
  (b) INDEPENDENTLY re-grep `script/ workspace/ savemutator/` and the sqlref/attrRegistry metadata
  for any `json_scalars`/`JSONScalars`/`scalars`-collection reader (confirmed none at plan time —
  only the 3 producers + 4 tests); (c) diff the non-scalar normalized tables for the fixture
  before/after to confirm typed projections are byte-identical; (d) confirm a synthetic unknown
  section still produces a Diagnostic; (e) `go test ./... -timeout 20m` green.
