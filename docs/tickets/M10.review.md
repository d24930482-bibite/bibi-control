## Scrutiny points (herder checklist)

1. **ReprojectWorking is working-partition-only.** Verified statically, not just via the
   test: `reproject.go` calls `ReplaceExtractedSave(ctx, duck, ExtractTables(worldID, archive))`
   and NEVER `CopySavePartition`. `ReplaceExtractedSave` delete+inserts ONLY `save_id =
   worldID`; it is structurally incapable of touching the head-sha-keyed history partition.
   Head is resolved exactly like `Load`: `GetWorld` → nil-head guard → `RevisionByID(*HeadRevisionID)`
   → `BlobPresent` guard (`notRematerializable`) → `reparseCommitted` (which does the
   `blobs().Get`). PASS.
   - Caveat (non-blocking, test-strength note): `TestReprojectWorkingDoesNotTouchHistory`
     empties then reprojects the WORKING partition, so working == head when the assertion
     runs. If reproject had erroneously done `CopySavePartition(worldID→headSHA)`, that copy
     would re-derive content-IDENTICAL history rows, so counts AND the energy value (333)
     would be unchanged — the test would still PASS. The test therefore pins history
     immutability against a *divergent* write but not against a content-equivalent copy. This
     does NOT affect the verdict because the implementation provably omits the history write
     entirely (read, not inferred). Recorded so a future refactor does not lean on this test
     to catch an accidental history copy.

2. **OpenAndReconcile leaks no handle on failure.** On reconcile error it does `_ = w.Close()`
   and returns `(nil, ReconcileResult{}, err)`. Open() is unchanged.
   `TestOpenAndReconcileHeadMissingFailsLoud` genuinely proves the Close: it asserts
   `errors.Is(err, ErrHeadBlobMissing)`, nil workspace, AND that a subsequent plain `Open`
   succeeds (a leaked DuckDB writer on the same file would block/corrupt the reopen) — plus
   the head is NOT silently demoted (tier still 'full'). PASS.

3. **VerifyStorageConsistency is read-only.** Pure `FullRevisions`/`OrphanedBlobs` SELECTs +
   `blobs().Has` + `IsRevisionHead`; issues no UPDATE/DELETE. `TestVerifyReportsFullButMissing`
   and `TestVerifyFlagsMissingHead` both assert the row is UNCHANGED (tier='full',
   blob_present=true) after the pass, and that a found inconsistency returns a nil error
   (finding lives in the report). PASS.

4. **Lock discipline / no re-entrant w.mu.** `ReprojectWorking` and `VerifyStorageConsistency`
   each take `w.mu` for the whole body and call ONLY lock-agnostic helpers: `reparseCommitted`
   (confirmed it does NOT take `w.mu`), `*Store` reads, free `duckdb.*` funcs, `*FSStore.Has`.
   None re-acquire `w.mu`. `OpenAndReconcile` calls `ReconcileBlobs` with NO lock held;
   `ReconcileBlobs` takes `w.mu` itself — correct, no double-acquire. No deadlock seam.
   `-race` SKIPPED by policy: the diff adds no goroutine, no `t.Parallel`, no new shared
   mutable state — concurrency is unchanged, the methods are sequential under the existing
   mutex, and the re-entrancy risk is statically resolved above. A scoped DuckDB `-race` run
   (checkptr=0 full rebuild) would only re-validate pre-existing driver code at high cost
   without exercising anything new.

5. **M9 constraint intact.** `Close()` and the supervisor-stop-before-drain ordering are
   UNTOUCHED — the only change to `workspace.go` is the added `OpenAndReconcile` func.
   Confirmed via diff.

6. **Deferred bullets not implemented.** No LRU/sweeper, no `DeleteWorkspace` own-handle
   eviction, no own-handle admin variants. `registry_ops.go` unchanged. Confirmed via grep
   and name-only diff.

## FindingBytesMissing reuse for the refcount floor

The refcount-floor anomaly tags its finding `Kind: FindingBytesMissing` (the "closest typed
bucket") while incrementing the separate `RefcountAnomalies` counter and writing an explicit
`Description`. This is NOT hiding a distinct real condition: the authoritative signal is the
`RefcountAnomalies` count (and `HasViolations()` checks it independently of `BytesMissing`),
and the byte-missing path increments `BytesMissing` separately. The shared `Kind` enum value
is a cosmetic labelling choice the plan explicitly sanctioned (count anomalies on the
already-scanned `FullRevisions` rows, add no new SQL). A consumer that scans `Findings` by
`Kind` alone could conflate the two, but the per-category counts disambiguate and no test or
caller relies on `Kind` to separate them. Acceptable.
