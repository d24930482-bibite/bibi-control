# G3 — GC sweep + re-promotion

## Tier
**high — exec opus / review opus.** Signals: *irreversible delete* (the sweep is the
only place outside G2 that destroys blob bytes), *crash-safety / atomicity* (must keep
G2's catalog-durable-before-byte-delete ordering and idempotency), and a *subtle
invariant a passing test can miss* — the orphan condition is "no `blob_present=1` row
needs the sha", NOT refcount-0 (G2's evict leaves refcount intact, so the existing
`UnreferencedBlobs` list silently misses the real orphan), plus the no-resurrection rule
(re-promotion must never bring back an intentionally-evicted revision). Fan-out is only
H1 (test parallelization), so low on that axis, but the delete/invariant signals alone
set high. When unsure here, round up: a wrong delete is unrecoverable.

**Perf flag: NONE.** `GCUnreferencedBlobs` is a manual/host-triggered admin op (spec line
495: "Manual/host-triggered, not automatic"), not a hot path. It scans the catalog once
per explicit invocation; the per-orphan `blobs().Has`/`Delete` round-trips are O(orphans),
bounded by how much was evicted, and only run when an operator asks. Not per-request /
per-query / per-event. No N+1 inside a live path. The store method is a single grouped SQL
query (one round-trip), not a query-per-row. So no perf-review lens is needed.

## Goal
"Done" means the workspace can reclaim the reclaimable-orphan bytes that G2's crash-safe
eviction can leave behind, and can re-promote a mirror_only revision back to `full` when
its bytes legitimately return — both without ever deleting a live blob or resurrecting an
intentionally-evicted revision, and both honoring G2's crash-safe ordering. Concretely:

1. `GCUnreferencedBlobs(ctx)` deletes on-disk blob objects whose sha256 is referenced by
   no `blob_present=1` (full) revision — the orphan G2 leaves when it crashes between the
   catalog commit and the byte delete (or after `ReconcileBlobs` demotes the last full
   row sharing a sha). Crash-safe and idempotent: it only ever deletes bytes that the
   committed catalog already says are not needed, so a re-run is a no-op.
2. `PromoteReappearedBlob(ctx, ref)` (re-promotion) flips every mirror_only revision of a
   sha back to `full`/`blob_present=1` when the bytes are present again, reusing
   `store().PromoteRevision` (tier/blob_present only — refcount untouched, so no double
   refcount).
3. The G2-review carryover is tidied: `evictRevisionBlobLocked`'s doc comment is corrected
   so it no longer claims the public `EvictRevisionBlob` wrapper returns the post-commit
   soft delete error (the wrapper passes `result=nil` and returns `false, nil`).

No schema change. No DuckDB writes. No Starlark/host wiring (the `workspace.gc()` surface
in workspace_plan.md:405 is later/E-track; G3 stays at the `*Workspace` API level, the
same level G2's `EvictWorldHistory` sits at today).

## Files to change

### `revisionstore/store.go` — new orphan-candidate query
- Add `OrphanedBlobs(ctx) ([]Revision, error)`. Returns ONE representative revision per
  distinct sha256 for which **no** row is `blob_present = 1` — i.e. every revision of that
  content is mirror_only, so the bytes are not needed by any full revision. This is the
  *true* orphan condition the G2 review note (recorded on this ticket) calls out:
  `UnreferencedBlobs` (store.go:1065) gates on `refcount = 0`, but G2's `EvictRevisionBlob`
  (store.go:1012) flips tier only and leaves `refcount = 1`, so the evicted orphan never
  appears in `UnreferencedBlobs`. Do NOT reuse or widen `UnreferencedBlobs`; add a distinct
  method so each keeps a single clear meaning.
  - SQL shape (group to one row per sha, exclude any sha that still has a full row):
    ```sql
    SELECT id, sha256, size, parent_id, source_path, blob_ref, inline_blob, script_run_id,
           created_at, world_id, tier, blob_present, refcount, mirror_schema_version
    FROM save_revisions r
    WHERE r.blob_present = 0
      AND r.sha256 NOT IN (SELECT sha256 FROM save_revisions WHERE blob_present = 1)
      AND r.id = (SELECT MIN(id) FROM save_revisions WHERE sha256 = r.sha256)
    ORDER BY r.sha256
    ```
    The `MIN(id)` correlated subquery collapses each sha to one representative row, so the
    caller gets exactly one `BlobRef` (sha + size) per orphan sha to stat/delete. Reuse the
    existing `scanRevisionRows` helper and the same column list as `FullRevisions` /
    `MirrorOnlyRevisions` (store.go:502/541) — copy that established pattern verbatim,
    including the nil-guard, `usableContext`, `rows.Close()`, and `rows.Err()` handling.

### `workspace/gc.go` — NEW file (the G3 surface)
- `func (w *Workspace) GCUnreferencedBlobs(ctx context.Context) (GCResult, error)`:
  whole-body `w.mu.Lock()` / `defer Unlock()` (same single-writer shape as
  `EvictWorldHistory`, eviction.go:116, and `ReconcileBlobs`, eviction.go:289). For each
  `Revision` from `w.store().OrphanedBlobs(ctx)`:
  - Skip `rev.BlobRef.IsInline()` (blobstore.Ref.IsInline, blobstore.go:42) — inline refs
    have no backing object; `Delete` is a no-op and they must NOT be counted as bytes
    reclaimed.
  - **Re-verify the orphan invariant under the lock immediately before deleting** via
    `w.shaHasFullReference(ctx, rev.SHA256)` (the existing helper, eviction.go:357). If it
    reports a full reference (a concurrent record/promote re-introduced one), SKIP the
    delete and tally it as `Skipped`. This is the load-bearing "never delete a live blob"
    guard, defense-in-depth over the SQL filter, mirroring how G2 double-gates head/shared.
  - `blobs().Delete(ctx, rev.BlobRef)` (blobstore Delete, fsstore.go:190 — idempotent,
    NotFound→nil). On error, append to `GCResult.DeleteErrors` and continue (soft, same
    discipline as `EvictResult.DeleteErrors`, eviction.go:72); never abort the sweep on a
    blobstore error. A real store/DB error from `OrphanedBlobs` or `shaHasFullReference`
    DOES abort and is returned.
  - On success increment `GCResult.BytesDeleted`.
- `GCResult` struct: `Candidates int`, `BytesDeleted int`, `Skipped int`,
  `DeleteErrors []error` — modeled on `EvictResult` (eviction.go:57) so tests assert
  without re-querying.
- `func (w *Workspace) PromoteReappearedBlob(ctx context.Context, ref blobstore.Ref) (int, error)`:
  the explicit re-promotion lever for "bytes came back by hash" (re-ingest / re-import /
  shared by another world; spec line 496-498). Under `w.mu`:
  - `has, err := w.blobs().Has(ctx, ref)` (fsstore.go:158). If `!has`, return `(0, nil)` —
    NEVER promote a revision whose bytes are not actually present (that would falsely claim
    rematerializability; the no-mirror→blob invariant). This is the gate that keeps
    re-promotion honest.
  - `revs, err := w.store().RevisionsBySHA256(ctx, ref.SHA256)` (store.go:423). For each
    `r` with `r.Tier == "mirror_only"`, call `w.store().PromoteRevision(ctx, r.ID)`
    (store.go:1101 — flips tier/blob_present ONLY, idempotent, refcount untouched → no
    double refcount). Count and return promotions.
  - Returning the count (not a richer struct) keeps this lever minimal; the
    reconcile-driven path (`ReconcileBlobs` Direction 2, eviction.go:321) already covers the
    startup-drift case, so this method is the runtime "bytes just arrived" companion. Cite
    the relationship in the doc comment to avoid the reader thinking they overlap.

### `workspace/eviction.go` — carryover doc tidy (recorded note)
- Fix the `evictRevisionBlobLocked` doc comment (eviction.go:233-236) that claims the

  The public `EvictRevisionBlob` wrapper (eviction.go:204-215) calls the core with

## Risks / seams (do not break)
- **Orphan condition is `blob_present`, not `refcount`.** The whole correctness of the
  sweep hinges on this. Using `UnreferencedBlobs` (refcount=0) would silently never reclaim
  G2's orphans (which sit at refcount=1). The new `OrphanedBlobs` query MUST gate on "no
  row of this sha is blob_present=1", and `TestStoreOrphanedBlobs` must prove the refcount-1
  orphan is caught. This is the exact trap the recorded G2 review note flags.
- **Never delete a live blob.** A sha shared by a full + mirror row (dedup/clone) must be
  excluded both in SQL (`NOT IN (... blob_present=1)`) and re-checked under the lock via
  `shaHasFullReference` right before delete. Both gates required; `TestGCSkipsLiveBlob`
  proves it.
- **Never resurrect an evicted revision.** `PromoteReappearedBlob` only promotes when bytes
  are *actually present* (`blobs().Has` gate first). A mirror_only row with absent bytes is
  intentionally evicted and stays mirror_only — same rule `ReconcileBlobs` Direction 2
  enforces (eviction.go:336-344, the "never resurrect a present-but-unreferenced orphan"
  note). G3's promote is the dual: present bytes + a mirror row → promote; absent → leave.
- **Crash-safe / idempotent.** GC only deletes bytes the committed catalog already says are
  unneeded; `blobstore.Delete` is idempotent (NotFound→nil), so a crash mid-sweep and a
  re-run are both safe. Do not add any catalog write to the GC path — it is byte-delete-only
  (catalog is already in the terminal mirror_only state when bytes are an orphan).
- **No double refcount on promote.** `PromoteRevision` flips tier/blob_present ONLY
  (store.go:1109-1111); G3 must NOT call `IncBlobRef` on promote (the evicted row kept its
  self-ref through eviction, mirroring the IngestAutosave note at node_reload.go:350-351).
- **Single-writer discipline.** Both new workspace methods take `w.mu` for the whole body
  and call only `Locked`-safe helpers (none re-take `w.mu`; `shaHasFullReference` is already
  lock-agnostic, eviction.go:357). Do not call `EvictWorldHistory`/`ReconcileBlobs` (which
  take the lock) from inside the GC path — that would deadlock the non-reentrant mutex.
- **Test footprint disjoint.** New code lives in `workspace/gc.go` + `workspace/gc_test.go`
  and an additive `revisionstore/store_test.go` block; the only edit to an existing file is
  the doc-comment fix in `eviction.go` (no behavior change). This keeps G3 from colliding
  with E1/C4b (touching workspace tests now) and lets H1 parallelize cleanly later.
