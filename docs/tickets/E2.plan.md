# E2 — `world.open()` returns a Save object; `s.commit()` advances head; `world.query()`

Source: `docs/workspace_plan.md#e2` (lines 434-449); E1 boundary note on the ticket.
Tier: **high — shared-contract scripting capstone over the head-advancing commit path; subtle
invariant (a commit that does not actually advance head / re-import, or a working-copy
`world.query` that reads stale projections, is silent corruption); 1 downstream dependent (F2
also edits automation.go)**. `--model-exec opus --model-review opus`.

## Goal
Complete the object-based automation surface (E1 deferred three names to E2):

1. `world.open()` returns a **Save object** (new `saveValue`) wrapping the world's already-loaded
   working copy (`*thebibites.LoadedSave` via `ws.OpenWorld`). The object exposes the existing
   sandboxed read/mutation surface (`s.bibites`, `s.eggs`, `s.settings`, `s.zones`, `s.pellets`,
   `s.sql`, `.where().set()/.delete()`) by delegating to the proven `thebibites.Save` value.
2. `s.commit()` commits the mutations already staged on that working copy and **advances the
   world head** (parent = current head, new revision, dual-key DuckDB re-import), reusing the C3
   commit machinery — NOT by re-running a program. Churn DoD preserved (one WriteArchive, one
   history import + one working re-seed, one mirror reparse).
3. `world.query(sql)` runs a **read-only** working-copy query against the open world's
   `LoadedSave`, reusing the existing read-only gate (`ensureReadOnly`/`scanForbidden`) and the
   dual-key working-partition scoping (`ls.saveID == worldID`).

"Done" = an automation script can do `s = world.open(); s.bibites.where(...).set(...);
s.commit()` and the world head moves to a new revision whose projection reflects the mutation;
`world.query("SELECT …")` returns working-copy rows; non-SELECT working-copy queries and all
commit/load errors surface as clean Starlark errors.

## Design decision (read first — this is the load-bearing call)
The existing head-advancing commit (`workspace.CommitWorld` → `thebibites.RunAndCommitWorld`) is
**program-based**: it runs a fresh Starlark program against `ls` via `script.Run(Globals(ls))`,
then commits. E2's model is **object-based**: mutations are staged by method calls on `s` in the
SAME automation thread; `s.commit()` must commit the *already-staged* `ls.session` without
re-running a program.

The staging fields are unexported in package `thebibites` (`ls.session`, `ls.stagedOps`,
`ls.willCommit`, `ls.prepareCommit`), so package `workspace` cannot drive the commit directly.
Therefore E2 **factors the post-run commit body out of `RunAndCommitWorld` into a new exported
`thebibites` function** that commits an already-staged loaded save with advancing-head semantics,
and reuses the existing workspace dual-key re-import. This reuses every invariant (single
WriteArchive, no double IncBlobRef, dual-key keys, reparse-for-fresh-projection) instead of
duplicating them. `RunAndCommitWorld` is refactored to call the same new function so there is one
commit code path, not two.

## Risks / seams
- **Silent stale-projection corruption (the subtle invariant):** `s.commit()` MUST re-import via
  `w.reparseCommitted` (NOT `wc.Applied`/`ExtractTables` on the in-memory archive — its typed
  projections are stale after `Session.Apply`, commit.go:99-117 / commit_world.go:21-31). The
  factored `applyWorldCommit` keeps this; the executor must not "optimize" it away. A
  value-asserting test (energy==50 after commit) catches a stale re-import.
- **Same-pointer staging:** `world.open()` returns the cached `ls`; `s.commit()` commits the same
  `w.worlds[worldID]` handle. If anything `Unload`s/`Load`s between, the staged session is lost.
  `OpenWorld` fast-path returns the cached handle, so within one hermetic automation cycle this
  holds — document it and have `saveValue` carry `worldID` (not a detached `ls` copy) so commit
  re-resolves the cached handle under `w.mu`.
- **No double IncBlobRef / head atomicity:** `RecordRevisionAdvancingHead` self-refs in-tx;
  `CommitLoadedWorld` must NOT call `IncBlobRef` (commit_world.go:60-62). Refactor must preserve
  the single SQLite tx that inserts-revision + advances-head.
- **Raw working-copy query scoping:** `ls.Query` runs raw SQL over the SHARED workspace DuckDB
  handle, whose tables hold every world's working/history partitions keyed by save_id. A naive
  `SELECT … FROM bibites` would read ALL save_ids. `ls.query` is the right primitive ONLY if it
  scopes to `ls.saveID`; the existing push-down (`whereClause`) does, but the RAW `s.sql` hatch
  does not auto-scope. Decide: working-copy `world.query` should constrain to the open world. The
  simplest correct rule reuses the dual-key contract — the working partition is keyed by
  `save_id = worldID`, so the binding/`ls.Query` must inject a `save_id = worldID` filter OR
  document that `world.query` is the same surface as `s.sql` (caller scopes). **Executor: scope
  `world.query` to the open world's `save_id` (working partition) — a cross-world read belongs to
  `workspace.query`/`world.history_query`, not the working-copy read.** Confirm against
  `sql.go:232` whereClause semantics; reuse, do not reinvent.
- **F2 contention (herder):** F2 also edits `workspace/automation.go` (the `workspace.transfer`
  global). E2 and F2 both touch this file → the herder must serialize them. E2's footprint here is
  additive (new `saveValue` + two `worldValue` cases), so a later F2 rebase is mechanical.
- **Read-only gate reuse:** do NOT duplicate `ensureReadOnly`/`scanForbidden`. Gate `world.query`
  by calling the existing `workspace`-package `ensureReadOnly` in the binding before `ls.Query`.
- **Perf-review: not flagged.** `s.commit()` is one-cycle-per-invocation (not a hot path);
  `world.query` is a single SELECT; the commit re-import is the established one-reparse +
  two-ReplaceExtractedSave C3 shape, already perf-reviewed in C3/C4. No new per-row / N+1 /
  unbounded-scan shape is introduced. No perf flag.
