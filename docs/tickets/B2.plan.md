# B2 — `workspace/` package skeleton: Open/Create + per-ws DuckDB file

Source: `docs/workspace_plan.md#b2` (the "Track B — B2 — `workspace/` package skeleton"
section, lines 284-295), plus the locked decisions at lines 68-75 (user-scope = owner namespace,
**one persistent DuckDB file per workspace**, shared SQLite registry + shared content-addressed
blobstore) and the contention note at lines 196-197 (`workspace/` is **new** — B2 defines the type;
later W/N/T/A tickets layer method-scoped diffs onto the same struct, so keep B2's surface minimal
and additive-friendly).

## Goal

"Done" = a new `workspace/` package exists with a `Workspace` type that ties the three already-built
stores together for **one owner-scoped workspace** and exposes only the lifecycle skeleton:
`Create(root, owner, name)` allocates a new `workspaces` registry row and a fresh per-workspace
DuckDB file; `Open(root, id)` re-attaches to an existing one; `Close()` releases every handle the
workspace owns. The struct carries the fields every later ticket needs (`*revisionstore.Store`,
`blobstore.Store`, the per-workspace `*sql.DB` DuckDB handle, the in-memory working set
`map[worldID]*thebibites.LoadedSave`, the active-node set `map[nodeID]*noderuntime.Runtime`, and a
`sync.Mutex` serializing mutating ops), but B2 implements **no** world/node/query operations — those
are C1/C2/C3/C4/D*/E*. Concretely B2 ships:

1. The `Workspace` struct + its accessors (`ID()`, `Owner()`, `Name()`, plus internal getters the
   later tickets reach through — `store()`, `blobs()`, `duck()`).
2. `Create(root, owner, name) (*Workspace, error)` — establishes the on-disk layout under `root`,
   creates the registry row via `revisionstore.CreateWorkspace`, opens + migrates the per-workspace
   DuckDB file, and opens the shared blobstore.
3. `Open(root, id) (*Workspace, error)` — same wiring but looks up an existing `workspaces` row
   (`GetWorkspace`) instead of creating one, and opens the **existing** DuckDB file (still calling
   the idempotent migrations).
4. `Close() error` — closes the DuckDB handle, the revisionstore handle, and the blobstore, joining
   any errors; safe on a nil/partially-constructed workspace.

B2 is **workspace-package-only**: do **not** edit `revisionstore/` (A2 gave us every method B2
calls — `CreateWorkspace`/`GetWorkspace`/`Open`/`Close`), do **not** edit `duckdb/` (use the
existing `Open`/`ApplyMigrations`), do **not** edit `blobstore/` (use `NewFSStore`), and do **not**
edit `script/thebibites/` (the working-set value type `*thebibites.LoadedSave` is only *referenced*
in a map field; B2 never constructs or loads one — that is C1/C2). No `AddWorld`, no `Load`, no
`Query`, no node start — those tickets layer methods onto this struct.

## Approach

### 0. Package + imports

`package workspace` importing:
- `github.com/asemones/bibicontrol/revisionstore`
- `github.com/asemones/bibicontrol/blobstore`
- `github.com/asemones/bibicontrol/duckdb`
- `github.com/asemones/bibicontrol/script/thebibites` (for the `*thebibites.LoadedSave` map value type only)
- `github.com/asemones/bibicontrol/noderuntime` (for the `*noderuntime.Runtime` map value type only)
- stdlib: `context`, `database/sql`, `errors`, `fmt`, `os`, `path/filepath`, `sync`

Note both `thebibites` and `noderuntime` are imported **solely** to name the working-set/active-node
map value types. They introduce no behavior in B2. (Confirm there is no import cycle: `thebibites`
imports `revisionstore`/`duckdb`/`blobstore` but NOT `workspace`; `noderuntime` imports only `ipc`.
`workspace` depends on all of them — a clean one-directional edge. A quick `go build ./...` proves it.)

### 1. On-disk layout under `root`

Lock the layout the plan dictates (workspace_plan.md lines 70/291-293):
```
<root>/
  metadata.sqlite                      # the SHARED revisionstore registry (one per root)
  blobs/                               # the SHARED content-addressed blobstore (one per root)
  workspaces/<workspaceID>/analytics.duckdb   # the PER-WORKSPACE DuckDB file
```
- The registry and blobstore are **root-scoped and shared across workspaces** (workspace_plan.md
  lines 68-75: "Shared SQLite registry … = operational truth for all workspaces"; "Shared
  content-addressed blobstore (dedups bytes across workspaces)"). The DuckDB file is **per-workspace**
  (lines 69-71/291-292: "One persistent DuckDB file per workspace … per-file = workspace isolation").
- Add private path helpers so `Create` and `Open` agree on every path (single source of truth):
  ```go
  func registryPath(root string) string  { return filepath.Join(root, "metadata.sqlite") }
  func blobsRoot(root string) string      { return filepath.Join(root, "blobs") }
  func workspaceDir(root, id string) string { return filepath.Join(root, "workspaces", id) }
  func duckPath(root, id string) string   { return filepath.Join(workspaceDir(root, id), "analytics.duckdb") }
  ```
- `root == ""` is an error (mirror `blobstore.NewFSStore`'s "root is required", fsstore.go:43, and
  `revisionstore.Open`'s "path is required", store.go:154).

### 2. The `Workspace` struct (workspace_plan.md lines 285-289)

```go
type Workspace struct {
    root  string
    id    string
    owner string
    name  string

    registry *revisionstore.Store
    blobs    blobstore.Store          // interface; concretely *blobstore.FSStore
    duck     *sql.DB                  // per-workspace DuckDB handle (analytics)

    mu      sync.Mutex                       // serializes mutating ops (single writer per DuckDB)
    worlds  map[string]*thebibites.LoadedSave // working set (worldID -> loaded save); empty in B2
    nodes   map[string]*noderuntime.Runtime   // active node set (nodeID -> runtime); empty in B2
}
```
- Hold a `blobstore.Store` (the **interface**, blobstore.go:20) typed field so later tickets/tests
  can inject a fake; assign the concrete `*blobstore.FSStore` from `NewFSStore` to it. But keep a
  separate way to close it: `NewFSStore` returns a `*FSStore` with a `Close()` (fsstore.go:72) that
  the interface does not expose, so store the concrete `*blobstore.FSStore` in an unexported
  `blobsCloser *blobstore.FSStore` field (or keep the concrete value and expose via the interface
  accessor) — **decide once**: simplest is to keep `blobs *blobstore.FSStore` as the field type and
  have the `blobs()` accessor return it (it satisfies `blobstore.Store`); only widen to the
  interface if a later ticket needs injection. Pick the concrete field to keep `Close()` reachable.
- `worlds`/`nodes` maps are allocated empty in both constructors (so later tickets never nil-check).
- Accessors: `func (w *Workspace) ID() string`, `Owner() string`, `Name() string`. Internal getters
  (`store()`/`blobs()`/`duck()`) returning the handles for the W/N/C tickets; keep them unexported
  (white-box callers are in-package). Add nil-receiver guards on accessors that return strings (return
  "") to match the defensive style throughout `revisionstore` (e.g. store.go:193).

### 3. `Create(root, owner, name string) (*Workspace, error)`

Order matters (registry row must exist before its DuckDB file is named by its id; partial failures
must clean up so a half-built workspace does not leak a handle):
1. Validate `root != ""`.
2. Open the shared registry: `registry, err := revisionstore.Open(registryPath(root))`. This
   `MkdirAll`s nothing — but `revisionstore.Open` opens a SQLite file; ensure the parent dir exists
   first with `os.MkdirAll(root, 0o755)`. (`revisionstore.Open` runs `ApplyMigrations`, so the
   registry schema is present/idempotent — store.go:170.)
3. Open the shared blobstore: `blobs, err := blobstore.NewFSStore(blobsRoot(root))` (it `CreateDir`s
   its bucket, fsstore.go:59-63 — no manual mkdir needed for `blobs/`).
4. Create the registry row: `ws, err := registry.CreateWorkspace(ctx, revisionstore.WorkspaceInput{Owner: owner, Name: name})`
   (store.go:474). `ws.ID` is the freshly allocated UUID.
5. `os.MkdirAll(workspaceDir(root, ws.ID), 0o755)` so the DuckDB file has a directory.
6. Open + migrate the per-workspace DuckDB: `duck, err := duckdb.Open(duckPath(root, ws.ID))`
   (import.go:25) then `duckdb.ApplyMigrations(ctx, duck)` (import.go:52). `ApplyMigrations` is
   idempotent `CREATE … IF NOT EXISTS` (workspace_plan.md line 70), so calling it on a freshly
   created file is correct and on Open it is a no-op.
7. Assemble the `Workspace{...}` with empty `worlds`/`nodes` maps and return it.

**Failure handling:** on any error after a handle is opened, close the handles opened so far before
returning (so a failed `Create` leaks nothing). Use a small `cleanup` slice of `func() error` (or
explicit `_ = registry.Close()` / `_ = blobs.Close()` / `_ = duck.Close()` in the error paths),
mirroring the `_ = db.Close()` rollback style in `revisionstore.Open` (store.go:167-172) and
`duckdb.Open` (import.go:31). Do **not** delete files/rows on failure (no destructive cleanup in v1).

`ctx` parameter: `Create`/`Open` should take a `context.Context` first arg (matches every store
method) — `func Create(ctx context.Context, root, owner, name string) (*Workspace, error)`. Pass it
to `CreateWorkspace`/`ApplyMigrations`.

### 4. `Open(ctx, root, id string) (*Workspace, error)`

Same wiring as `Create` but **reuses** the existing workspace:
1. Validate `root != ""` and `id != ""`.
2. Open shared registry (same as Create step 2).
3. `ws, err := registry.GetWorkspace(ctx, id)` (store.go:497). If `revisionstore.IsNotFound(err)`
   (store.go:1303), close the registry handle and return a clear error wrapping it (the workspace
   does not exist). 
4. Open shared blobstore (Create step 3).
5. Open the **existing** DuckDB file: `duckdb.Open(duckPath(root, ws.ID))` + idempotent
   `ApplyMigrations`. (Do not MkdirAll the workspace dir on Open — it should already exist; but a
   harmless `os.MkdirAll` is acceptable for robustness. Prefer not to create it, so an Open against a
   missing file surfaces as a DuckDB open of a fresh empty (migrated) file — acceptable and matches
   "Open re-attaches"; document that DuckDB creating an absent file is the chosen behavior.)
6. Assemble + return.

Same failure-cleanup discipline as Create.

### 5. `Close() error`

Close every owned handle, joining errors with `errors.Join` (used already in `duckdb` at import.go:251):
```go
func (w *Workspace) Close() error {
    if w == nil { return nil }
    var errs []error
    // (later tickets stop/unload worlds+nodes here; B2's maps are empty so nothing to do)
    if w.duck != nil     { errs = append(errs, w.duck.Close()) }
    if w.registry != nil { errs = append(errs, w.registry.Close()) }
    if w.blobs != nil    { errs = append(errs, w.blobs.Close()) }
    return errors.Join(errs...)
}
```
Leave a comment that C2/D1 will drain `worlds`/`nodes` here (Unload working set / Stop runtimes)
before closing the DB — B2 just closes the three store handles. `errors.Join(nil, nil, nil)` is `nil`,
so a clean close returns nil.

### 6. Single-writer note (no code in B2, but document the seam)

The `mu sync.Mutex` field exists in B2 but is **not locked** by any B2 method (Create/Open/Close are
not concurrent-mutation ops). Add a doc comment on the field stating that AddWorld/CommitWorld/
IngestAutosave/ReloadNode (C3/D3) acquire it to serialize the single DuckDB writer per workspace
(workspace_plan.md lines 198-199/521). This makes the seam explicit for the next tickets without B2
taking a lock prematurely.

## Risks / seams

- **No import cycle, but verify.** `workspace` is the top of the dependency edge: it imports
  `thebibites` (for the working-set value type) which imports `revisionstore`/`duckdb`/`blobstore`,
  and `noderuntime` (which imports only `ipc`). None of those import `workspace`. If a cycle ever
  appears it means a lower package reached up — `go build ./...` catches it. Keep `thebibites`
  imported **only** as a map value type; do not call into it from B2.
- **Shared vs per-workspace boundary is load-bearing.** `metadata.sqlite` and `blobs/` are **one per
  root, shared by all workspaces**; only `analytics.duckdb` is per-workspace (workspace_plan.md lines
  68-75). Getting this wrong (e.g. a per-workspace SQLite) would break cross-workspace blob dedup and
  the "registry = operational truth for all workspaces" decision. The path helpers encode this; the
  `TestSharedRegistryAndBlobsAcrossWorkspaces` test pins it.
- **Handle-leak on partial failure.** `Create`/`Open` open up to three OS handles in sequence; an
  error after one is opened must close what was opened (mirroring `revisionstore.Open`'s
  `_ = db.Close()` on a failed `Ping`/migrate, store.go:166-172). A leaked SQLite/DuckDB handle holds
  a file lock and breaks the next test's TempDir cleanup.
- **`blobstore.Store` is an interface with no `Close`.** `Close()` lives on the concrete
  `*blobstore.FSStore` (fsstore.go:72), not the `Store` interface (blobstore.go:20). Store the
  concrete `*blobstore.FSStore` (or keep a separate closer reference) so `Workspace.Close` can close
  the bucket — closing it matters because `fileblob` holds the bucket open.
- **`duckdb.Open("")` is in-memory; B2 must pass a real file path.** The old per-run model opened
  in-memory (`OpenAndImport(ctx, "", …)`, the very thing B2 replaces — workspace_plan.md lines 14-15/
  69-71). B2 must call `duckdb.Open(duckPath(...))` with a non-empty path so the mirror is persistent
  and per-workspace. An empty path here would silently reintroduce the ephemeral mirror.
- **`ApplyMigrations` idempotency is relied on for Open.** `duckdb.ApplyMigrations` is
  `CREATE … IF NOT EXISTS` (import.go: migrations), so calling it on both Create (fresh file) and Open
  (existing file) is correct and cheap. Do not gate it behind a "first time" check — always call it,
  matching `revisionstore.Open` which always applies its schema (store.go:170).
- **B2 implements no world/node ops on purpose.** The struct exposes the *fields* (working set,
  active nodes, mutex) the later W/N tickets fill in, but B2 ships only Create/Open/Close. Resist
  adding AddWorld/Load/Query here — C1/C2/C4 own them and will append method-scoped diffs to this
  same struct (workspace_plan.md lines 196-197). Keeping B2 minimal keeps those parallel tickets from
  colliding on `workspace.go`.
- **Single-writer mutex is declared, not yet used.** The `mu` field is present so C3/D3 can lock it,
  but B2's Create/Open/Close don't take it (they're not concurrent mutators). Document the intended
  lock holders in a field comment; do not lock prematurely (it would be dead code and could mask a
  later correctness review).
