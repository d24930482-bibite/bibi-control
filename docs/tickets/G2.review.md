## Core safety invariants (staked the merge on these)
1. CRASH-SAFE ORDERING — VERIFIED. `evictRevisionBlobLocked` (eviction.go:237-266):
   step 1 = `w.store().EvictRevisionBlob` (G1) which commits its SQLite tx and returns
   on any error before a byte is touched; only after that commit returns nil does step 3
   call `w.blobs().Delete`. SQLite opens with default `synchronous=FULL` + rollback
   journal (only `_pragma=foreign_keys(1)` is set; no WAL/synchronous override), so the
   commit is fsync-durable before control returns. A crash post-commit/pre-delete leaves
   a `mirror_only` row with possibly-present orphan bytes — reclaimable, never a dangling
   head/full pointing at deleted bytes.
2. NEVER DELETE A LIVE BLOB — VERIFIED. Head-refusal and `refcount > 1` refusal both
   live inside G1's committed tx (store.go:1043-1049) and return typed errors BEFORE the
   tier flip, so the workspace caller never reaches Delete. Refcount is per-sha256
   (insertRevisionTx seeds from MAX over the hash then IncBlobRef, so a fresh unique
   revision = refcount 1, evictable; any sharing — including cross-world same-sha256 —
   lifts all rows to >=2, refused). TestEvictRefusesSharedBlob proves Has==true and the
   row stays full after a refused shared eviction; the cross-world-shared case is the
   same per-hash mechanism the test simulates via IncBlobRef.
3. NEVER WRITES DUCKDB — VERIFIED. eviction.go calls only `w.store()` (SQLite) and
   `w.blobs()` (blobstore); no `w.duck()`. TestEvictWorldHistoryKeepsMirrorDeletesBlob
   asserts evicted revisions' `save_archives` history rows are unchanged after evict.
4. IDEMPOTENT byte delete — VERIFIED. FSStore.Delete treats `gcerrors.NotFound` as nil
   and inline refs as no-op nil (fsstore.go:190-212); TestFSStoreDeleteIdempotent proves
   double-delete + inline no-op + ref validation.
5. SINGLE-WRITER w.mu — VERIFIED. `w.mu` is a plain `sync.Mutex` (non-reentrant).
   EvictWorldHistory/EvictRevisionBlob/ReconcileBlobs take it; the locked core
   `evictRevisionBlobLocked` and `shaHasFullReference` call only store/blobs methods
   (which never take w.mu) — no re-entrancy.

## The two flagged judgment calls
(a) OlderThanSimTime scoped to per-revision created_at (Unix seconds) — ACCEPTABLE, not
    drift. No per-revision sim_time column exists (sim_time is on the world row); the
    plan explicitly authorized the created_at fallback with a loud doc note and made this
    constructor secondary, with KeepLastN the must-ship policy. The fallback is documented
    on the constructor (eviction.go:45-52); created_at is RFC3339Nano-stored, .Unix() is
    coherent; selection excludes head and uses strict `<`. KeepLastN is fully exercised.
(b) Reclaimable-orphan asserted via RevisionsBySHA256 rather than UnreferencedBlobs —
    CORRECT, and in fact the plan's literal wording ("reported by UnreferencedBlobs")
    would have been WRONG: G1's evict flips tier only and does NOT decrement refcount, so
    the evicted blob keeps refcount=1 and UnreferencedBlobs (WHERE refcount=0) would never
    list it. The executor's substitute — assert no revision of that sha256 remains
    blob_present=1 (present-on-disk, no full revision needs it) — is the true
    reclaimable-orphan condition and is a meaningful, non-weakened check. Reasoning is
    documented in the test (eviction_test.go:307-325).

## Minor notes (non-blocking, no current blast radius)
- The doc comment on `evictRevisionBlobLocked` (eviction.go:232-236) claims a post-commit
  soft delete error is "also returned so the single-revision caller sees it." It is not:
  on the public single-revision `EvictRevisionBlob` path `result` is nil, so the soft
  error is neither appended to DeleteErrors nor returned (`return false, nil`) — it is
  silently swallowed. This is SAFE (catalog is correctly mirror_only; bytes are a
  reclaimable orphan) and the must-ship op EvictWorldHistory surfaces such errors
  correctly via EvictResult.DeleteErrors. There are no production callers of the workspace
  EvictRevisionBlob wrapper yet (G3/G4 wire the ops), so zero blast radius. Worth a
  one-line doc/observability tidy in a later ticket; not a merge blocker.
