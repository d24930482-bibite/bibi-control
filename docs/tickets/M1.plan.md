## Goal

Lift genes and brain nodes/synapses into the existing spanning machinery as new
read-only analytics "kinds", so a cross-world or whole-world gene/brain question is one
object-DSL call (`workspace.genes.where("name == 'SizeRatio'").mean()`,
`workspace.synapses.where("enabled").count()`, `world.nodes.group_by("sim_time")...`)
with the catalog join + save-partition scoping injected BY CONSTRUCTION — never raw SQL
touching `mirror_saves` / `save_id` / `number_value`. "Done" = `world.genes`,
`world.nodes`, `world.synapses` and `workspace.genes`, `workspace.nodes`,
`workspace.synapses` exist as aggregate-only spanning collections supporting
`count/sum/mean/median/min/max/quantile/where/group_by`, scoped per-world and
all-worlds, with `group_by("world_id")` / `group_by("sim_time")` working and the
read-only/mutation gates inherited unchanged. Plus the two same-slice minors:
all-worlds `Query` gets an automatic `world_id`-attributing CTE wrapper, and
`ensureReadOnly` keeps its current err-safe blocklist (document, don't rewrite).

## Design decision (load-bearing): do NOT overload `entityTables`

`entityTables` (attr_registry.go:58) is consumed by BOTH `buildRegistry`
(attr_registry.go:139) AND `buildAccess` (loadedsave.go:193), which index those tables
into `ls.access` for the SINGLE-SAVE working path (in-memory entry-name order,
iteration, scalar reads). The gene/brain tables are 1:many (`entitySubCollections`) and
are deliberately excluded from `entityTables` today. Adding "gene"/"node"/"synapse" to
`entityTables` would change the working path (new `ls.access` entries, new
`attrRegistry` kinds, possible interaction with `identityAccess()`/`entryNames()`),
which spanning kinds never use (they are aggregate-only, never materialized).

Therefore register the spanning kinds in a SEPARATE, spanning-only metadata seam so the
working path is byte-identical. Concretely:

- Add a `spanningKinds` registry (kind → identity table list + a derived friendly-column
  attrRegistry entry) that `identityTable`, `attrRegistry`, `fromClause`, and
  `oneToManyTables` can resolve, WITHOUT those tables joining the working-path
  `entityTables`/`buildAccess` iteration. The cleanest minimal form: extend the
  functions that read `entityTables[kind]` (`identityTable` in sql.go, `entityTables[kind]`
  in fromClause/collection.identityAccess) to fall back to a spanning-kind table map, and
  extend `attrRegistry()` build to also register the spanning kinds' friendly columns —
  but keep `buildAccess`'s loop reading ONLY the original working `entityTables` (it
  already iterates that map directly, so leaving it untouched is the isolation guarantee).

  Implementation choice for the executor: introduce
  `var spanningEntityTables = map[string][]string{ "gene": {"bibite_genes"}, ... }`
  and a small `tablesForKind(kind) []string` helper that returns
  `entityTables[kind]` else `spanningEntityTables[kind]`. Route `identityTable`,
  `fromClause`, and `EntityCollection.identityAccess` through `tablesForKind`. Build the
  spanning kinds' friendly-column registry into `attrRegistry()` (so
  `resolveColumn`/`rewritePredicate` resolve `weight`/`enabled`/`node_in`/gene `value`
  etc.) by extending `buildRegistry` to ALSO walk `spanningEntityTables`. Leave
  `buildAccess` untouched. This keeps the single-save working path byte-identical
  (verified by re-running the full existing suite, see Tests) while making the spanning
  push-down generic over the new kinds.

### The "one kind, two tables" problem (bibite_genes + egg_genes)

A spanning `genes` collection should span genes of BOTH bibites and eggs (and same for
nodes/synapses). The existing push-down assumes ONE identity table per kind. Two clean
options — pick (A):

- **(A) PREFERRED: per-source kinds are NOT exposed; the spanning kind unions via a
  view-like identity.** Define each spanning kind's identity as a UNION ALL over its two
  source tables, projected to a common column set, exposed to the push-down as a derived
  "identity" the `FROM`/scope clause targets. Concretely, `tablesForKind("gene")` returns
  a synthesized identity name (e.g. `genes_all`) whose `fromClause` emits
  `FROM (SELECT save_id, gene_name, value_type, number_value, ... FROM bibite_genes
  UNION ALL SELECT ... FROM egg_genes) AS genes`. The scope clause's `<identity>.save_id`
  binds to the union alias; `save_id` is carried through the union so the
  `save_id IN (SELECT save_id FROM mirror_saves ...)` scoping and the catalog LEFT JOIN
  still work unchanged. This matches §1's intent ("genes" = all genes in scope,
  regardless of carrier) and keeps the column surface single.

  The executor builds the union FROM in a small `spanningFromExpr(kind)` consulted by
  `fromClause` when `kind` is a spanning kind; the columns projected are exactly the
  friendly registry columns for that kind plus `save_id` (for scope + catalog join).
  The catalog LEFT JOIN keys on `<alias>.save_id = mirror_saves.save_id` (1:1 by
  save_id, never inflates).

  NOTE: `fromClause`'s `oneToManyTables()` guard must NOT trip here — the union members
  are the identity itself, not scalar-joined sub-tables, so they never go through the
  `tables[1:]` LEFT-JOIN loop. Keep that loop empty for spanning kinds (they have no 1:1
  sub-tables).

- (B) Rejected for v1: expose `bibite_genes` and `egg_genes` as two separate kinds
  (`gene`/`egg_gene`). Cleaner code but a worse surface (`workspace.genes` vs
  `workspace.egg_genes`) and diverges from §1's single-`genes` target.

## Approach (steps, citing reuse)

1. **Register the spanning kinds (attr_registry.go).** Add `spanningEntityTables` and
   `tablesForKind`. Extend `buildRegistry` to walk the spanning kinds via the existing
   `tableScalarSpecs` (generated-metadata-derived; no hand allowlist) so
   `attrRegistry()["synapse"]["weight"]` etc. resolve. For genes, apply an overrides map
   (`name`→`gene_name`, `value`→`number_value`, `type`→`value_type`) using the existing
   `applyOverrides` helper, and exclude the raw `*_value`/locator columns from the direct
   surface. Reuse `nonScalarColumns`/`subLocatorColumns` semantics to drop locator noise.

2. **Generic FROM for spanning kinds (sql.go).** Add `spanningFromExpr(kind) (string,
   error)` emitting the `(SELECT save_id, <friendly source cols> FROM <tbl1> UNION ALL
   SELECT ... FROM <tbl2>) AS <alias>` derived identity, then have `fromClause` use it
   for spanning kinds and skip the 1:1 sub-table loop. Route `identityTable` through
   `tablesForKind` (returns the alias for spanning kinds). `whereClause`/`scopeClause`
   already qualify `<identity>.save_id` — point that at the union alias. The catalog
   LEFT JOIN (`scope.catalogJoin(identity)`) keys on `<alias>.save_id`, unchanged.

3. **Gene typed-value column.** Default `value` → `number_value` (the §1 default). A
   `where("type == 'string'")` filter still works (`value_type` is a registered friendly
   `type`). Document that `.mean()`/`.sum()` over `value` is numeric-only by default; a
   per-type virtual value is deferred. (Keep this localized to the gene overrides map.)

4. **SpanningReader + dispatch (scope.go).** Add the three constructors + the
   `Collection` switch arms. Each returns an aggregate-only `EntityCollection` exactly
   like `Bibites()` — the read-only/mutation/iteration gates in collection.go already key
   off `c.readOnly()`/kind, so genes/nodes/synapses inherit them with no new gate code.

5. **Binding (automation.go).** Wire the three new names into `workspaceValue.Attr` and
   `worldValue.Attr` spanning cases + AttrNames. They route through the existing
   `spanningCollection` helper (catalog-refresh-once shape unchanged).

6. **world_id wrapper (query.go).** Prepend the all-worlds `world_saves` CTE to `Query`.
   Add the `ensureReadOnly` clarifying comment.

7. **Autocomplete (app.js).** Add the three names to `AC_TYPE_MEMBERS` for `workspace`
   and `world`.

## Risks / seams (do not break)

- **Cross-world isolation is the headline invariant.** A spanning aggregate that drops
  the `save_id IN (SELECT … FROM mirror_saves …)` clause silently reads every world's
  rows and a passing `count()` would not notice. The UNION-ALL identity MUST carry
  `save_id` through both members so `scopeClause` and the catalog join still bind. Every
  new isolation test asserts the per-world count, never just "non-zero" (mirroring the
  existing suite's discipline).
- **Working-path byte-identity.** `buildAccess` must stay reading only the original
  `entityTables`; do NOT let the gene/brain tables leak into `ls.access` or the
  single-save attr registry's kind set (`bibite`/`egg`/`pellet` must be unchanged). The
  `TestSpanningKindRegistration` `ls.access["bibite_genes"] == nil` assertion is the
  tripwire. Re-run the full existing suite to confirm no single-save regression.
- **`oneToManyTables()` guard.** The gene/brain tables ARE in
  `entitySubCollections`, so `oneToManySet` contains them. For spanning kinds they are
  the identity (union members), never scalar-joined, so they must bypass the `tables[1:]`
  LEFT-JOIN loop in `fromClause`. If a spanning kind ever reached that loop the guard
  would (correctly) error — so route spanning kinds down the `spanningFromExpr` branch
  BEFORE the loop.
- **Gene value typing.** Genes are number/bool/string; defaulting `value` →
  `number_value` means `.mean()` silently ignores bool/string genes (they have NULL
  `number_value`). Document this; a `where("type == 'number'")` is the explicit guard.
  This is the §1 open question resolved as "default to number_value" — flag it.
- **`Query` wrapper is prepend-only and read-only-safe.** The wrapper is a leading `WITH`
  that `ensureReadOnly` already accepts; it MUST run AFTER `ensureReadOnly` (the existing
  order) so the gate sees the user's verbatim query, not the wrapped form. Do not let the
  wrapper introduce a name (`world_saves`) that collides with a user CTE — keep the same
  name `HistoryQuery` already uses (documented), and note in the comment that a user CTE
  named `world_saves` would shadow it (pre-existing behavior, not new).
- **Spanning kinds are aggregate-only — never iterable/transferable.** Inherited from
  `c.readOnly()` (Len/Iterate panic; EntryNames errors). Confirm the new kinds hit the
  non-transferable default case in `EntryNames()` (bibite/egg only) so a gene collection
  can never be handed to `transfer`.
