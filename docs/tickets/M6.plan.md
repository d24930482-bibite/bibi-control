# M6 — Transfer: move semantics + body.id remap + full graft surface

> Source: `docs/missing.md` §6 ("Transfer is copy-only, partial, and surface-incomplete")
> and `docs/surface-map/03-cross-save-transfer.md` ("Missing seams").
>
> **Merge serialization:** this ticket edits `savemutator/thebibites/transfer*.go`,
> which **M8** also touches (parser/mutator write-path hardening). The herder must
> serialize M6 and M8 at the merge boundary — do not merge both into a shared base
> concurrently.

## Goal

`workspace.transfer(...)` today is a copy-only whole-entity graft: it appends onto
the destination, never deletes from the source, hard-fails on a `body.id` collision,
exposes only whole-bibite/egg grafts (settings-copy and array-feed are engine-complete
but surface-dead), and returns a count the UI regex-scrapes from printed output.

"Done" for M6 means the transfer surface is **complete and honest**:

1. **Move semantics (opt-in).** `workspace.transfer(..., move=True)` deletes the
   grafted entries from the **source** world after the destination commit succeeds,
   so the entity ends up in exactly one world (no biomass double-count). The default
   stays copy (`move=False`). The two-world commit ordering is **dst-first, then
   source-delete-commit**, with the cross-save atomicity limitation documented loudly
   in code and surfaced in the structured result.
2. **`body.id` remap (opt-in).** `workspace.transfer(..., remap_ids=True)` mints a
   fresh non-colliding `body.id` for a grafted bibite instead of aborting the whole
   batch on the 1-in-2^32 collision. Default stays fail-loud (`remap_ids=False`),
   preserving today's guard.
3. **Structured UI result.** The `workspace.transfer` return dict gains explicit
   `moved`/`source_committed` (and keeps `transferred`/`committed`/`revision_id`/
   `sha256`), and `api/web/app.js` reads the dict from a structured `/run` result
   instead of regex-scraping `Output`, so a format change can never show
   `transferred ?`.

Settings-copy / array-feed bridge wiring and species-ancestry remap are **explicitly
scoped OUT** of M6 (see "Scope cuts" — they are listed in §6 but are separable, lower-
value, and would balloon this already high-risk ticket). M6 closes the two seams that
make transfer *wrong* (copy-not-move, fatal collision) plus the UI honesty gap.

Tier: high — irreversible source delete + cross-save two-commit ordering (no two-phase)
+ body.id allocator invariant + return-dict contract shared with app.js. Exec=opus,
Review=opus.

## Approach

### 1. body.id remap allocator (`transfer_identity.go`)

Mirror the **proven species allocator** exactly — do not invent a new shape:

- Add `dstBodyIDUsage() (max int64, ok bool)` modeled on `dstSpeciesIDUsage`
  (`transfer_identity.go:96`): scan every dst bibite entry's `body.id` (reuse
  `bibiteBodyID`, `session.go:517`) **and** fold in `body.id`s already staged for
  append this session — scan `t.dst.StagedOperations()` for `OperationAppendEntry`
  payloads and read `bibiteBodyID(op.EntryPayload.JSON)`. This staged-fold is the
  load-bearing multi-graft distinctness guard (exact analogue of the species
  staged-fold at `transfer_identity.go:138-145`): without it, two colliding source
  bibites in one loop would both remap to the same fresh id.
- Add `freshDstBodyID() int64` = `dstBodyIDUsage()+1` (or `0`/`1` when none observed).
  body.id has no `nextBodyID` counter, so the allocator rests entirely on observed-max
  (unlike species which also reads `nextSpeciesID`). **Note the difference in a
  comment** so a reviewer/M8 does not "fix" it by looking for a counter.
- Eggs: `body.id` lives on bibites only (the collision guard at
  `reconcileGraftIdentity` already gates on `kind == tb.EntryBibite`). Egg grafts have
  no body.id to remap — leave them untouched.

### 2. Plumb a remap decision into the graft (`transfer.go` + `transfer_identity.go`)

`reconcileGraftIdentity` (`transfer_identity.go:46`) currently **refuses** a body.id
collision. Change it so the body.id branch is conditional:

- Add an options carrier. Prefer a small `GraftOptions{ RemapIDs bool }` struct passed
  to `AppendEntry` (so the public seam stays readable and M8's parallel edits collide
  minimally). Update the `Workspace` interface in `savemutator/thebibites/workspace.go`
  if `AppendEntry`'s signature there changes — confirm whether the interface method
  is the one the bridge calls and keep the compile-time assertion
  (`transfer.go:41`) green.
- When `RemapIDs` is **false**: behavior is byte-for-byte today (loud collision +
  dangling-child guard). Do NOT regress the existing tests.
- When `RemapIDs` is **true** and `dstBibiteWithBodyID(id)` collides: allocate
  `freshDstBodyID()` and `setJSONPath(cloned, "body.id", fresh, SetOptions{})` on the
  clone **before** any `StageAppendBibite`, preserving the 0-staged-ops invariant
  (`transfer.go:204-262` ordering: all fail-able work before the first stage).
- The **dangling-child guard stays loud regardless** of `remap_ids` — remapping body.id
  does not give us a way to fix cross-world `body.eggLayer.children` references, so a
  dangling child is still a hard failure. Document this so a future reader does not
  assume remap_ids loosens it.
- Interaction with species remap: independent axes. body.id remap and species remap
  both rewrite the SAME `cloned` JSON before staging; order does not matter (different
  paths). Keep species remap exactly as-is.

### 3. Source-side delete for move (`transfer_bridge.go`)

`TransferEntries` currently stages only dst grafts. For `move`:

- After the per-entry `CollectEntry`→`AppendEntry` loop stages the dst grafts, stage
  the corresponding **source deletes** on `srcLS` by reusing `srcLS.stageEntityDelete`
  (`script/thebibites/entity.go:196`) — the SAME guarded delete path
  `Entity.delete()` / bulk `where(...).delete()` use, so move inherits the identical
  cascade/referential guard. Bump `srcLS.stagedOps++` per delete (matching the dst
  graft `stagedOps++` and `entity.go:186`).
- **kind:** `CollectEntry` classifies the entry (`bibites`/`eggs` table); map back to
  the `stageEntityDelete` kind string (`"bibite"`/`"egg"`). The entry name is the
  source entry name passed in.
- **Do NOT commit here.** The bridge stages onto both sessions; the WORKSPACE layer
  owns the two commits. Return enough for the caller to know deletes were staged
  (e.g. the count, or a bool). Keep the signature minimal — the count of grafts is the
  same as the count of deletes for `move`.
- **Atomicity within staging:** if graft k fails, return the error before staging the
  source delete for k and before the caller commits anything — the partial dst+src
  stages die in-memory exactly as the existing copy path documents
  (`transfer_bridge.go:30-40`). Preserve that contract verbatim.

### 4. Two-commit orchestration (`workspace/transfer.go`)

This is the highest-risk seam — read `workspace/commit.go:113-161` (`CommitWorldLoaded`)
and the lock discipline note at `workspace/transfer.go:22-29` before editing.

- Resolve the **source** working copy too. The selector already carries `srcLS`
  (passed in), but for a clean source commit you need the **same cached source
  `*LoadedSave`** that staged the deletes to be what `CommitWorldLoaded(srcWorldID)`
  re-resolves. Confirm `srcLS` is the OpenWorld-cached handle for the source world
  (it comes from `EntityCollection.SourceLoadedSave()` /
  `Entity.SourceLoadedSave()`, which is the open working copy) and pass the source
  **world id** through so the second commit can re-resolve it. You will need the
  source world id at this layer — thread it from `automation.go` (the selector's
  source world) rather than reverse-deriving it.
- **Ordering (the data-safety decision):** commit **dst first**. Only if the dst
  commit succeeds (`rev.ID != 0`) do you commit the source deletes. Rationale: a
  crash/error between the two commits then leaves the entity in **both** worlds (a
  recoverable duplicate, identical to today's copy outcome) — never **neither** world
  (unrecoverable loss). Document this ordering and its single-failure mode in a
  prominent comment; it is the whole reason the cut is acceptable without two-phase
  commit.
- If `move` and the dst commit succeeded but the source-delete commit fails, return an
  error that **names the half-applied state explicitly** ("graft committed to dst rev
  X; source delete failed, entity remains in source — re-run delete manually") so the
  user is never silently left with a duplicate they think was moved.
- Keep the existing lock discipline: each `CommitWorldLoaded` takes `w.mu` for its own
  body; Transfer must NOT hold `w.mu` across both (it is non-reentrant). The two
  commits are sequential, each self-contained — do not try to make them one critical
  section.
- Return a result struct (or extend the existing return) carrying: dst revision,
  whether moved, whether the source delete committed, source revision. Empty-selection
  stays a clean no-op (`transfer.go:50-54`).

### 5. Result dict + builtin kwargs (`workspace/automation.go`)

- `transferBuiltin` (`automation.go:285`): add `move?` and `remap_ids?` bool kwargs via

- Build the dict with the existing keys plus `moved` (bool) and `source_committed`
  (bool), and `source_revision_id` if a source commit happened. Keep
  `transferred`/`committed`/`revision_id`/`sha256`. Update the doc comment

### 6. UI: structured result + controls (`app.js`, `index.html`)

- `index.html` `#m-transfer`: add a `move` checkbox and a `remap ids` checkbox near the
  existing `tr-kind`/`tr-where` controls. Match the existing modal markup style.
- `app.js` `transferToWorld()` (`app.js:1068-1110`): pass `move=`/`remap_ids=` into the
  built program's `workspace.transfer(...)` call; **stop regex-scraping** — the program
  already `print(...)`s the dict, but instead of a regex on `res.Output`, change the
  program to a structured emit the handler can read deterministically. Two acceptable
  approaches; pick the one matching how `/run` results are consumed elsewhere:
  (a) keep `print(dict)` but parse it as the structured dict the run result exposes, or
  (b) if `/run` already returns a structured value channel, use it.
  **Investigate `runProgram`/`/run` first** (search `app.js` for `runProgram` and the
  `/run` handler in `api/`) — do not assume; the goal is "a format change can't show
  `transferred ?`", so the count must come from a typed field, not a regex.
- Toast "moved N" when move succeeded, "copied N" otherwise; on a half-applied move
  surface the error message from the result, do not swallow it.

## Risks / seams

- **Cross-save non-atomicity (the load-bearing risk).** Two separate world commits, no
  two-phase. The MITIGATION is the ordering: dst commit first, source delete second, so
  the only failure window leaves a recoverable DUPLICATE (today's copy outcome), never
  data LOSS. Any executor change that commits source-first, or that couples the two into
  one `w.mu` critical section, is WRONG — flag it in review. This overlaps §10
  (workspace lifecycle / commit ordering); do not invent new commit logic, reuse
  `CommitWorldLoaded` twice.
- **0-staged-ops-on-rejection invariant.** `AppendEntry` runs every fail-able check
  before the first stage (`transfer.go:204-262`). The body.id remap must rewrite the
  clone BEFORE `StageAppendBibite`, never after. A regression here re-introduces
  half-staged-on-reject corruption.
- **body.id allocator distinctness.** The staged-fold in `dstBodyIDUsage` is the only
  thing keeping two colliding grafts in one loop from remapping to the same id — it is
  the exact analogue of the species staged-fold and is easy to omit. The multi-graft
  test is non-negotiable.
- **`remap_ids` must NOT loosen the dangling-child guard.** Remapping body.id does not
  fix cross-world child references; keep that failure loud.
- **Return-dict contract.** `app.js` (and any other reader) consumes the transfer dict.
  Adding keys is safe; renaming/removing `transferred`/`committed`/`revision_id`/
  `sha256` would break the existing UI — only ADD.
- **M8 merge serialization.** Both M6 and M8 edit `savemutator/thebibites`
  (transfer_identity / parser write-path). Keep edits localized; the herder serializes
  the merge.
- **Perf:** the body.id allocator scans every dst bibite entry per graft — same shape
  as the existing species allocator, bounded by dst population and selection size, run
  once per transfer (not a hot path). No new N+1. **No perf flag.**
