## Load-bearing seam 1 — lock-under-IPC (traced independently, not via tests)
- `Workspace.mu` is a non-reentrant `sync.Mutex` (workspace.go:45), so holding it
  across a blocking IPC `Request` would serialize/deadlock the workspace.
- `node_control.go` product code never references `w.mu` directly (only comments).
- Lookup uses `w.Node(nodeID)` (node.go:188-196) which Lock/defer-Unlock and returns
  the runtime — lock fully released before any IPC.
- `rt.State()` (runtime.go:142-164) and `rt.Connected()` (runtime.go:176-180) take the
  runtime's own RLock briefly and release before returning.
- `Runtime.Request` (runtime.go:239-247) grabs the session pointer under a brief RLock,
  releases, THEN calls the blocking `sess.Request`. No runtime lock held across the wire
  either.
- Conclusion: PASS — no `w.mu` (nor runtime lock) is held across any
  `c.Info/Stop/Resume` round-trip.

## Load-bearing seam 2 — NodeState shape (consumed by D3 + E1)
- `type NodeState struct { Runtime noderuntime.State; Connected bool; Info *ipc.InfoResult }`
  — matches the plan verbatim.
- `Info` is a pointer: nil == disconnected/no-telemetry, distinct from a zero-valued
  reading (Paused:false/TPS:0). `Connected` hoisted from `Runtime.Connected`.
- Disconnected handling is correctly bifurcated: `NodeState` treats not-connected as
  SUCCESS (Info nil, Connected false, no error); `NodeInfo/NodeStop/NodeResume` surface
  `noderuntime.ErrNoSession` wrapped with `%w` (errors.Is-able). A connected-but-failed
  INFO in NodeState returns an error (does not emit nil Info for a connected node).
- `noderuntime.State` carries NodeID/RunID/PID/Process/CompatAddr/Connected — coherent
  field set for downstream consumers.
