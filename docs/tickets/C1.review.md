## Critical invariant — blob refcount == 1 (the redo's whole reason)
VERIFIED by independent trace, not by the green suite alone:
- The AddWorld path contains NO explicit `IncBlobRef`/`incBlobRefTx` call. `world.go` step 5
  records the revision via `RecordRevisionAdvancingHead` and goes straight to the mirror
  (step 6). The only `IncBlobRef` tokens in the workspace package are in comments
  (world.go:130, world_test.go:128).
- `RecordRevisionAdvancingHead` (revisionstore/store.go:285) -> `insertRevisionTx`
  (store.go:337) seeds the row refcount from `COALESCE(MAX(refcount) WHERE sha256=?, 0)`
  (= 0 for a brand-new content hash) and then calls `incBlobRefTx` (store.go:380) inside the
  same SQLite tx. Net: a freshly recorded first revision is at refcount EXACTLY 1, atomically
  (G1's auto-self-ref). An additional explicit IncBlobRef would have produced 2 — the merge
  bug that reddened main. It is correctly absent.
- The test pins this: `world_test.go:129` fails unless `rev.Refcount == 1`, and
  `AddWorldBytes` re-pins it at `world_test.go:272`. The assertion reads the real `refcount`
  column (scanRevisionRows, store.go:1236 binds `&revision.Refcount`), not a zero-value
  default, so it is a genuine guard against both double-count and under-count.

Omission of the plan's stale step ~6 (explicit IncBlobRef) is the intended fix, not drift.
