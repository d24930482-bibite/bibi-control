# H2 — Parallelize + speed the save-parse hot path

**Tier: high — exec opus / review opus (set by herder). Signals: concurrency/atomicity
(parsing 1027 independent entries across a worker pool with order-sensitive apply),
subtle invariant (output Archive must be byte-for-byte identical to the sequential
parse), novel logic (worker pool + ordered apply written from scratch), and this is the
user's EXTREME-HIGH-PRIORITY product perf fix. The parallelism is the correctness risk.**

**Perf-review: YES.** `ParseFile`'s entry-parse loop is the hot path — it runs on every
save load / AddWorld / CommitWorld / world-open, and it scans all N zip entries (N grows
with world population). The fix is itself a perf change; the reviewer MUST measure
before/after on the 3.1MB fixture and assert materially-lower wall-time AND prove
deterministic equality. Concern + locus: `saveparser/thebibites/parse_archive.go:80`
sequential `for i := range archive.Entries { archive.parseEntry(...) }` runs `parseEntryPayload`
(pure, CPU-bound JSON decode + map-walk) on a single core for all 1027 entries — the
measured 70% of import wall-time.

## Goal

Make the JSON entry-parse loop in `ParseFile` run across all available cores instead of
one, and (secondarily) swap the generic decode-to-`any` onto the faster `goccy/go-json`
decoder, WITHOUT changing the parser's output for any input. "Done" means: (1) the
per-entry `parseEntryPayload` work runs concurrently over a GOMAXPROCS-bounded worker
pool; (2) results are applied to the shared `Archive` strictly in original entry order so
the produced `Archive` (Bibites/Eggs/Pheromones/Scalars/Diagnostics slices and all derived
counts) is byte-for-byte identical to the current sequential parse for the same archive;
(3) `go test -race ./saveparser/...` is clean; (4) measured wall-time on
`autosave_20260301021357.zip` (3.1MB, 1027 bibites) is materially lower than before. The
volatile genome/brain/body parse stays on the generic `map[string]any` intermediate — NO
rigid typed structs (hard constraint; see memory save-format-churn-strategy).

## Established facts (do NOT relitigate — verified in code this planning pass)

- `parseEntryPayload(entry *Entry) parseResult` (parse_payload.go:31) is already PURE per
  entry: it creates a fresh `ctx := &parserContext{}` (line 32, so `parserContext` and its
  `diagnostics` are already per-entry — no shared diagnostics buffer), decodes, map-walks,
  and returns a value `parseResult`. Its ONLY side effects are writes to the per-entry
  `*Entry`: `entry.HasUTF8BOM = hasBOM` (line 37) and `entry.JSON = value` (line 43).
  Because each entry is a distinct `&archive.Entries[i]` slot, those writes are race-free
  as long as each goroutine owns a distinct index. Keep passing `&archive.Entries[i]`.
- `applyParseResult(result parseResult)` (parse_payload.go:96) is the ONLY mutator of
  shared `Archive` state and is order-sensitive: it appends to `a.Bibites`, `a.Eggs`,
  `a.Pheromones`, `a.Scalars`, `a.Diagnostics` and assigns `a.Scene/Vars/Settings/Species/
  PelletData`. Apply order == entry order is the determinism contract. This MUST run
  single-threaded in entry order.
- No package-level mutable state in the parse path: package vars are read-only
  (`utf8BOM`, the `settingsChangerZoneTargetRE` regexp [safe for concurrent Match],
  `settingsTopLevelCollections`/`settingsZoneStructuralKeys` read-only maps,
  `NormalizedTables`, `scalarKeepSuffixes`, `typedEntityPrefixes`). `collectScalars`
  (json_scalars.go:10) allocates a fresh local slice per call. No globals are written.
- `decodeJSONWithBOM` (json_helpers.go:13) uses `encoding/json` `Decoder` with
  `UseNumber()` and a strict trailing-token check (decodes a second value, requires
  `io.EOF`). This is the slow decode-to-`any` path. `entry.HasUTF8BOM` is later read by
  `normalize.go:31`, so BOM semantics must be preserved exactly.
- `github.com/goccy/go-json v0.10.5` is ALREADY in go.mod (indirect, pulled by
  arrow/duckdb) and is in go.sum — using it adds NO new dependency. It is API-drop-in for
  `encoding/json`: `NewDecoder`, `Decoder.UseNumber()`, and `json.Number` all match.
- `recomputeCounts()` (archive_counts.go) runs AFTER the loop, reads the assembled slices,
  and appends count-mismatch diagnostics; leave it sequential and unchanged.
- Fixture present: `testdata/saves/the-bibites/autosave_20260301021357.zip` (3.1MB, 1027
  bibites, totalFiles 1043). Existing tests call `tb.ParseFile(filepath.Join(fixtureDir,
  name), nil)` from external test package `tests`
  (`saveparser/thebibites/tests/parse_fixture_test.go`).

## Approach

### Step 1 — Parallelize the entry-parse loop (PRIMARY, the big win)

Replace lines 80-82 of `parse_archive.go` with a bounded worker pool that computes
`parseResult` for every entry concurrently into a pre-sized result slice, then applies
results sequentially in index order.

Concrete shape (worker pool, no errgroup needed since `parseEntryPayload` returns no
error — diagnostics are carried in `parseResult.Diagnostics`):

- `n := len(archive.Entries)`. If `n == 0`, skip. For tiny archives (`n <= 1`, or below a
  small threshold like `n < 2`), just run the existing sequential path to avoid goroutine
  overhead — keep it simple, a single `if n < parallelParseThreshold` sequential branch is
  acceptable (threshold a small const, e.g. 2).
- Pre-size `results := make([]parseResult, n)`.
- Worker count: `workers := runtime.GOMAXPROCS(0)`; clamp to `[1, n]` (`if workers > n {
  workers = n }`). This is the concurrency knob — bounded by GOMAXPROCS, never spawns one
  goroutine per entry. Document it as a named local/const.
- Dispatch indices via a `chan int` (buffered to `n` or fed by a single producer
  goroutine) OR an atomic counter (`sync/atomic` `Int64` index claimed with `Add`). Either
  is fine; the atomic-counter pattern avoids channel overhead. Each worker loops: claim
  next index `i`, compute `results[i] = parseEntryPayload(&archive.Entries[i])`. Each
  worker writes ONLY `results[i]` and mutates ONLY `&archive.Entries[i]` (distinct slot per
  index) — no shared writes, no mutex on the hot path.
- `sync.WaitGroup` to join all workers.
- After join, apply sequentially in order: `for i := range results {
  archive.applyParseResult(results[i]) }`. This preserves the exact append order of the old
  loop, so `a.Bibites`, `a.Eggs`, `a.Pheromones`, `a.Scalars`, `a.Diagnostics` come out
  identical.
- Then `archive.recomputeCounts()` exactly as today.

Determinism argument to encode in a comment: the only order-dependent output is produced by
`applyParseResult`, which now runs single-threaded over `results[0..n-1]` in the same order
the old sequential loop ran. The parallel section only fills independent `results[i]` slots
and per-entry `Entry` fields, so it cannot affect ordering or content.

Imports to add to `parse_archive.go`: `runtime`, `sync`, and `sync/atomic` (if using the
atomic counter).

### Step 4 — Faster generic decoder (SECONDARY — include now; it is a zero-dep drop-in)

`goccy/go-json` is already vendored in go.mod/go.sum, so this is free to include and
multiplies on top of Step 1. In `json_helpers.go`:

- Change the JSON import used by `decodeJSONWithBOM` to `github.com/goccy/go-json`. The
  function body stays the same: `gojson.NewDecoder(bytes.NewReader(raw))`, `.UseNumber()`,
  two `Decode` calls with the `io.EOF` trailing check. goccy preserves `UseNumber` ->
  `json.Number` semantics and the strict-EOF behavior.
- CONSTRAINT: `json.Number` and `json.Marshal` are also used elsewhere in this file
  (`toFloat`/`toInt` type-switch on `json.Number`; `rawJSON` calls `json.Marshal`;
  `scalarParts` uses `json.Number`). `goccy/go-json` re-exports `Number` as the SAME type
  alias as `encoding/json.Number` is NOT guaranteed — verify: goccy's `json.Number` is its
  own `type Number string`. The decoder with `UseNumber()` will emit goccy's `Number` type,
  so the type-switch cases `case json.Number:` in `toFloat`/`toInt`/`scalarParts` MUST
  reference the SAME package that the decoder produced. Simplest safe approach: import
  goccy as the `json` identifier for the WHOLE file (`json "github.com/goccy/go-json"`) so
  every `json.Number` / `json.Marshal` / `json.NewDecoder` in this file refers to goccy
  consistently. goccy's `Marshal` is a drop-in for `encoding/json.Marshal` and produces
  identical canonical output for the scalar types `rawJSON` handles (string/number/bool),
  preserving `Scalar.RawJSON` bytes. Executor MUST confirm this with the existing
  scalar/fixture tests (they assert exact RawJSON), not assume it.
- If, and ONLY if, the goccy `Marshal` output for any scalar differs from `encoding/json`
  (a fixture/scalar test fails on RawJSON), fall back to: import goccy ONLY for the decoder
  (`gojson "github.com/goccy/go-json"`), keep `encoding/json` for `json.Marshal` AND switch
  the `json.Number` type-switch cases to `gojson.Number` (since the decoder now emits that
  type). This is the seam to watch. The fixture tests are the arbiter — let them decide,
  do not guess.
- BOM handling is unchanged (the `bytes.HasPrefix(raw, utf8BOM)` strip stays).

## Risks / seams

- **Determinism is the whole game.** Any path that lets the parallel section affect output
  ordering or content breaks byte-identity. The design confines all order-sensitive
  mutation to the single-threaded `applyParseResult` loop over an index-ordered `results`
  slice. Do NOT apply results from inside workers, and do NOT use a `map[int]parseResult`
  or a channel-drain-as-completed apply (completion order != index order).
- **Per-entry `*Entry` writes.** Workers mutate `entry.JSON` and `entry.HasUTF8BOM` on
  their own `&archive.Entries[i]`. This is safe ONLY because each index is owned by exactly
  one worker. The index-claim mechanism (atomic counter or channel) must guarantee no index
  is processed twice. `-race` will catch a violation.
- **goccy `json.Number` / `Marshal` type identity (the decoder seam).** `Scalar.RawJSON`
  bytes and the `json.Number` type-switches must stay correct after the import swap. The
  fixture/scalar tests assert exact RawJSON — they are the arbiter. If goccy's `Marshal`
  diverges on any scalar, use the Step-4 fallback (goccy decoder only, `encoding/json`
  Marshal, `gojson.Number` in the type-switches). If the decoder swap shows ANY test
  divergence that can't be reconciled quickly, SPLIT it out and ship Step 1 alone — Step 1
  is the structural ~70% win and must not be blocked by the optional decoder.
- **No new dependency churn.** goccy is already in go.mod/go.sum; do not add anything else
  (no sonic — it has cgo/asm constraints and is not vendored).
- **HARD constraint reminder:** map-based generic parse for genome/brain/body stays. No
  typed structs. Capabilities/columns keep deriving from generated metadata.
- **GOMAXPROCS-bounded workers.** Never spawn one goroutine per entry; the worker count is
  clamped to `[1, min(GOMAXPROCS, n)]`. On a 1-core/GOMAXPROCS=1 box it degrades to
  sequential (and the small-n threshold also takes the sequential branch), so behavior is
  safe everywhere.
- **Leave `recomputeCounts` and `readZipEntry` sequential.** Only the parse loop is
  parallelized; zip reading already streams entries into `archive.Entries` and the count
  pass reads the assembled archive.
