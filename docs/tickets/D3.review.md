## Load-bearing invariants verified against the code

1. CRASH-SAFE DROP FILE — PASS. `writeDropFileAtomic` (node_reload.go:139-179)
   does CreateTemp in `filepath.Dir(dropPath)` → Write → `tmp.Sync()` (fsync file)
   → Close → `os.Rename(tmp, dropPath)`. Sibling temp guarantees the rename is
   atomic on the same filesystem; a crash mid-write leaves the prior drop file
   intact (cleanup-on-error removes the temp). RELOAD is sent only after
   writeDropFileAtomic returns (ReloadNode:85→91). The crash-safe path is
   exercised by TestReloadNode_ShipsHeadAndSendsReload (drop bytes == head blob,
   RELOAD fired). Note: parent-directory fsync is NOT done — the herder flagged it
   as "ideally". This is consistent with the repo's other atomic-write helpers
   (savemutator/install.go, saveparser/write_archive.go) which do not even fsync
   the file; the plan (lines 69-73) only specified `f.Sync()` + atomic rename,
   which the code does. Not a blocking defect.

2. NO DOUBLE REFCOUNT — PASS. appendIngestedRevision calls
   RecordRevisionAdvancingHead and never IncBlobRef. Confirmed
   insertRevisionTx self-refs in-tx (store.go:380 incBlobRefTx), so refcount==1.
   TestIngestAutosave_AppendsAndAdvancesHead asserts rev2.Refcount == 1. grep
   confirms no IncBlobRef anywhere in the new file. (C1-redo lesson honored.)

3. NO REPARSE — PASS. ExtractTables is fed the fresh `archive` from ParseFile
   directly (node_reload.go:371,378); no reparseCommitted-style call (grep
   confirms). Comment documents why ingest differs from CommitWorld.

4. DEDUP NO-OP — PASS. Step 6 (node_reload.go:254-269) compares
   `head.SHA256 == archive.SHA256` (content hash) and returns
   (zero Revision, false, nil) BEFORE any blob Put or mirror write. The early
   return precedes ReadFile/appendIngestedRevision, so no mirror write occurs.
   TestIngestAutosave_DedupSkipsUnchanged proves ingested==false, head unchanged,
   RevisionsForWorld stays at 1.

5. TOCTOU / lock discipline — PASS. The authoritative parent is re-read INSIDE
   w.mu in appendIngestedRevision (GetWorld → parentID, lines 343-347) immediately
   before RecordRevisionAdvancingHead, whose insert+SetWorldHead are one SQLite tx.
   w.Node, nodeRowByLogicalID (ListNodes), Info IPC, stabilize, parse, ReadFile,
   and the dedup read all run OUTSIDE the lock; only record+dual-key-import+cache
   delete hold it. Non-reentrant w.mu never nested (verified by grep of lock
   acquisition sites). headBlobBytes takes its own lock and releases before RELOAD.

6. DUAL-KEY IMPORT + new-file-only — PASS. History keyed by `ref.SHA256` via
